Тема 3.2: Компоненти ZTA та Identity Governance

Логічна архітектура ZTA (NIST)

  • Policy Engine (PE): приймає рішення
  • Policy Administrator (PA): виконує рішення
  • Policy Enforcement Point (PEP): застосовує політики
  • Взаємодіють для кожного запиту на доступ

Policy Engine (PE)

  • "Мозок" системи Zero Trust
  • Аналізує: суб'єкт, ресурс, контекст
  • Консультується: threat intelligence, SIEM, CDM
  • Рішення: дозволити / відмовити / умовно

Policy Administrator (PA)

  • Виконує рішення Policy Engine
  • Створює сесійні токени/облікові дані
  • Налаштовує PEP для пропуску/блокування
  • Логує рішення для аудиту

Policy Enforcement Point (PEP)

  • "Двері" до кожного ресурсу
  • Перехоплює всі запити
  • Застосовує рішення PA
  • Реалізація: API Gateway, Service Mesh, ZTNA

Допоміжні компоненти

  • CDM: стан безпеки пристроїв
  • Threat Intelligence: інформація про загрози
  • SIEM: збір та аналіз логів
  • Data Access Policy: класифікація даних

Identity — новий периметр

  • Замість "звідки?" → "хто?"
  • Автентифікація: підтвердження ідентичності
  • Авторизація: перевірка прав
  • IdP: Identity Provider як центр

Multi-Factor Authentication (MFA)

  • Something you know: пароль, PIN
  • Something you have: телефон, YubiKey
  • Something you are: біометрія
  • Тренд: Passwordless (FIDO2/WebAuthn)

Conditional Access

  • Різні вимоги залежно від контексту
  • Корп. пристрій + офіс: SSO достатньо
  • Невідомий пристрій: вимагати MFA
  • Чутливі дані: MFA + device compliance

Service-to-Service Identity

  • Сервіси теж автентифікуються!
  • Workload Identity: AWS IRSA, Azure Managed Identity
  • SPIFFE/SPIRE: стандарт ідентичності workloads
  • Без збереження секретів у коді
