Тема 1.6: Топ загроз CSA та реальні інциденти

CSA Top Threats to Cloud Computing

  • Cloud Security Alliance — провідна організація
  • Регулярно публікує звіт про загрози
  • На основі реальних інцидентів
  • Практичні рекомендації захисту

#1 Неправильна конфігурація

  • Публічні S3 bucket'и
  • Відкриті бази даних
  • Security groups 0.0.0.0/0
  • Logging вимкнено
  • Причина 90% хмарних breach'ів

#2 Проблеми IAM

  • Надмірні привілеї
  • Відсутність MFA
  • Shared credentials
  • Hardcoded ключі в коді
  • Застарілі акаунти

#3 Небезпечні API

  • Broken authentication
  • Excessive data exposure
  • No rate limiting
  • Injection вразливості
  • OWASP API Security Top 10

#4 Компрометація акаунтів

  • Phishing — найпоширеніший вектор
  • Credential stuffing
  • Витік ключів у GitHub
  • Malware / keyloggers
  • MFA — критичний захист

#5 Внутрішні загрози

  • Зловмисні — крадіжка даних, саботаж
  • Недбалі — помилки конфігурації
  • Compromised — викрадені credentials
  • Principle of Least Privilege
  • Моніторинг привілейованих дій

Кейс: Capital One (2019)

  • 106 мільйонів записів клієнтів
  • Імена, адреси, SSN, кредитна історія
  • Штраф $80 мільйонів
  • Атакуюча — колишня працівниця AWS

Capital One — як це сталося

  • WAF з надмірними IAM правами
  • SSRF вразливість у WAF
  • Доступ до EC2 metadata service
  • Отримання IAM credentials
  • Доступ до S3 з даними клієнтів

Capital One — чия відповідальність?

  • AWS? — інфраструктура працювала правильно
  • Capital One? — неправильна конфігурація
  • Надмірні IAM права
  • Відсутність захисту від SSRF
  • = Відповідальність клієнта

Кейс: Uber (2016)

  • 57 мільйонів записів
  • AWS credentials у GitHub репозиторії
  • Uber заплатила хакерам $100K
  • Приховували інцидент рік
  • Урок: ніколи не hardcode credentials

Кейс: Log4Shell (2021)

  • CVE-2021-44228 — критична вразливість
  • Remote Code Execution
  • Мільйони Java-додатків вразливі
  • Supply chain — залежності мають вразливості
  • SCA (Software Composition Analysis)

Як захищатися

  • CSPM — автоматичне сканування конфігурацій
  • IAM гігієна — least privilege, MFA, аудит
  • Secrets management — ніякого hardcoding
  • Infrastructure as Code — версіонування, review
  • Continuous monitoring — виявлення аномалій

Ключові висновки

  • Конфігурація — #1 причина інцидентів
  • Shared Responsibility — знай свою частину
  • Автоматизуй безпеку
  • Моніторинг та аудит — постійно
  • Вчись на чужих помилках
