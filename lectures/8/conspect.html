Лекція 8. Cloud-Native Application Protection Platform (CNAPP)

2 год

Вступ

У попередній лекції ми детально розглянули CSPM — інструменти для управління станом безпеки хмарних конфігурацій. Однак CSPM — лише один з компонентів комплексної хмарної безпеки. Сьогодні ми розглянемо CNAPP (Cloud-Native Application Protection Platform) — концепцію об'єднання всіх аспектів захисту cloud-native додатків в єдину платформу.

Gartner ввів термін CNAPP у 2021 році, визнавши, що організації стикаються з "tool sprawl" — надмірною кількістю ізольованих інструментів безпеки, які не взаємодіють між собою. CNAPP — це відповідь на цю проблему: єдина платформа, яка захищає весь lifecycle cloud-native додатків.

Чому виникла потреба в CNAPP?

Еволюція хмарної безпеки створила кілька окремих категорій інструментів:

  • CSPM: Configuration та compliance для IaaS
  • CWPP: Захист workloads (VMs, containers, serverless)
  • CIEM: Identity та entitlement management
  • Container Security: Image scanning, runtime protection
  • IaC Security: Сканування Terraform, CloudFormation

Проблема в тому, що ці інструменти працювали ізольовано. Security team отримувала тисячі alerts з різних систем без можливості зрозуміти реальний ризик. CNAPP об'єднує ці компоненти, забезпечуючи:

  • Correlated context: Зв'язок між misconfiguration, vulnerability та excessive permissions
  • Attack path analysis: Розуміння, як атакуючий може досягти критичних активів
  • Unified prioritization: Фокус на реальних ризиках, а не окремих findings
  • Single pane of glass: Одна консоль для всієї cloud security

Цілі лекції

Після цієї лекції ви зможете:

  • Пояснити концепцію CNAPP та її відмінність від point solutions
  • Описати компоненти CNAPP та їх взаємодію
  • Розуміти attack path analysis та risk correlation
  • Оцінювати CNAPP-рішення для організації

8.1 Від Point Solutions до CNAPP

Історія розвитку хмарної безпеки

Еволюція cloud security tools відображає еволюцію самих хмарних технологій:

Фаза 1: Lift-and-Shift (2010-2015)

  • Міграція традиційних додатків в хмару
  • Security tools: адаптовані on-premise рішення
  • Фокус на VM security, network firewalls

Фаза 2: Cloud-Native Emergence (2015-2019)

  • Kubernetes, Docker, serverless набирають популярність
  • Нові категорії: Container Security, CSPM, CWPP
  • Кожна проблема — окремий tool

Фаза 3: Tool Sprawl Crisis (2019-2021)

  • Середня enterprise організація: 10-15 cloud security tools
  • Тисячі alerts без контексту та prioritization
  • Integration challenges, alert fatigue

Фаза 4: Consolidation/CNAPP (2021+)

  • Gartner вводить термін CNAPP
  • Vendors об'єднують point solutions
  • Фокус на correlated risk та attack paths

Визначення CNAPP за Gartner

Gartner визначає CNAPP як "unified, tightly integrated set of security and compliance capabilities designed to protect cloud-native infrastructure and applications across development and production".

Ключові характеристики CNAPP:

  • Unified: Єдина платформа замість набору point solutions
  • Tightly integrated: Компоненти працюють разом, обмінюються контекстом
  • Dev + Production: Захист на всіх етапах lifecycle
  • Cloud-native focus: Специально для containers, serverless, IaC

Проблема ізольованих інструментів

Чому point solutions недостатньо? Розглянемо приклад:

Сценарій: Публічна S3 bucket з sensitive data

  • CSPM знаходить: "S3 bucket is public" — Medium severity
  • DSPM знаходить: "Bucket contains PII data" — High severity
  • CIEM знаходить: "Lambda has access to bucket" — Low severity
  • CWPP знаходить: "Lambda has vulnerability CVE-XXX" — Medium severity

Кожен tool бачить свою частину картини. Але реальний ризик — це attack path:

Internet → Vulnerable Lambda → IAM role → Public S3 → PII data breach

CNAPP об'єднує ці findings і показує: "Critical: Exploitable path to PII data".

8.2 Компоненти CNAPP

Основні компоненти

CNAPP: інтеграція компонентів хмарної безпеки
Рис. 8.1. CNAPP об'єднує розрізнені інструменти безпеки в єдину платформу з централізованим аналізом ризиків

1. CSPM (Cloud Security Posture Management)

Ми детально розглядали в попередній лекції. У контексті CNAPP:

  • Configuration assessment для cloud infrastructure
  • Compliance monitoring (CIS, SOC 2, PCI DSS)
  • Asset inventory та visibility
  • Інтеграція з іншими компонентами для context enrichment

2. CWPP (Cloud Workload Protection Platform)

Захист робочих навантажень — VMs, containers, serverless:

  • Vulnerability Management: Сканування images та workloads на CVEs
  • Runtime Protection: Моніторинг behavior, anomaly detection
  • Malware Detection: Антивірус для cloud workloads
  • File Integrity Monitoring: Виявлення змін у файловій системі
  • Network Security: Microsegmentation, traffic analysis

3. CIEM (Cloud Infrastructure Entitlement Management)

Управління ідентифікацією та правами доступу:

  • Permission Analysis: Хто має доступ до чого
  • Least Privilege: Виявлення excessive permissions
  • Unused Access: Права, які не використовуються
  • Cross-account Access: Mapping доступу між accounts
  • Service Account Security: Non-human identities

4. Container Security

Специфічний захист для контейнерів та Kubernetes:

  • Image Scanning: Vulnerabilities, malware, secrets у images
  • Registry Security: Захист container registries
  • Kubernetes Security: RBAC, Network Policies, Pod Security
  • Runtime Protection: Container behavior monitoring

5. IaC Security (Shift-Left)

Безпека Infrastructure as Code:

  • IaC Scanning: Terraform, CloudFormation, Pulumi
  • Policy as Code: OPA/Rego, Sentinel
  • CI/CD Integration: Pre-deployment checks
  • Drift Detection: Різниця між IaC та deployed resources

6. DSPM (Data Security Posture Management)

Захист даних у хмарі:

  • Data Discovery: Автоматичне виявлення sensitive data
  • Data Classification: PII, PHI, PCI, secrets
  • Data Flow Analysis: Куди переміщуються дані
  • Encryption Status: At-rest та in-transit

Інтеграція компонентів

Сила CNAPP — у взаємодії компонентів:

Компонент A + Компонент B = Результат
CSPM (public bucket) DSPM (contains PII) Critical data exposure
CWPP (vulnerable image) CIEM (internet-facing) Exploitable from internet
Container (CVE) CIEM (admin IAM role) Privilege escalation path
IaC (misconfiguration) CSPM (deployed) Drift alert

8.3 Attack Path Analysis

Що таке Attack Path?

Attack path — це послідовність кроків, які атакуючий може виконати для досягнення цілі (наприклад, доступу до sensitive data або компрометації критичної системи).

Приклад attack path:

  1. Initial Access: Publicly exposed Lambda function з вразливістю
  2. Execution: Експлуатація CVE для виконання коду
  3. Privilege Escalation: Lambda role має надмірні permissions
  4. Lateral Movement: Доступ до RDS через IAM role
  5. Data Exfiltration: Копіювання даних з бази

Graph-based Approach

Сучасні CNAPP використовують graph databases для моделювання cloud environments:

Nodes (вершини):

  • Resources: VMs, containers, databases, storage
  • Identities: Users, roles, service accounts
  • Network: VPCs, subnets, security groups
  • Data: Data stores, data flows

Edges (зв'язки):

  • Network connectivity
  • IAM permissions
  • Data access
  • Trust relationships

Алгоритми graph traversal знаходять шляхи від entry points (internet-facing resources) до crown jewels (critical data/systems).

Risk Scoring та Prioritization

CNAPP пріоритезує findings на основі:

Exploitability factors:

  • Чи є публічний доступ? (Internet exposure)
  • Чи є known exploit для вразливості?
  • Чи потрібна автентифікація?
  • Складність експлуатації (CVSS metrics)

Impact factors:

  • Які дані можуть бути скомпрометовані?
  • Яка business criticality системи?
  • Regulatory implications (PCI, HIPAA)?
  • Blast radius — що ще буде затронуто?

Combined Risk = Exploitability × Impact

Приклад Attack Path Analysis

Scenario: E-commerce application на AWS

CNAPP виявляє attack path:

Internet
    ↓
[ALB] Public Load Balancer
    ↓
[ECS Task] Container з CVE-2024-XXXX (Critical)
    ↓
[IAM Role] ecs-task-role (має s3:*, rds:*)
    ↓
[RDS] Customer database (PII)
    ↓
[S3 Bucket] Backups (contains credit cards)

CNAPP Risk Assessment:

  • Entry point: Internet (high exploitability)
  • Vulnerability: Critical CVE з public exploit
  • IAM: Excessive permissions (s3:*, rds:*)
  • Data: PII + payment card data (high impact)
  • Combined Score: Critical — requires immediate action

Порівняння з point solutions:

  • CWPP: "Container has critical CVE" — одне finding серед тисяч
  • CIEM: "Role has excessive permissions" — немає контексту чи використовується
  • CSPM: "Bucket allows cross-account access" — здається безпечним
  • CNAPP: "Critical attack path to customer data" — чіткий пріоритет

8.4 CNAPP: Ринок та інструменти

Gartner Magic Quadrant

Gartner оцінює CNAPP vendors за двома осями:

  • Ability to Execute: Product, sales, market responsiveness
  • Completeness of Vision: Strategy, innovation, market understanding

Leaders (2024): Wiz, Palo Alto (Prisma Cloud), Microsoft, CrowdStrike

Challengers: Orca, Lacework, Aqua, Sysdig

Visionaries: Emerging vendors з innovative approaches

Niche Players: Specialized solutions

Огляд ключових платформ

Wiz

  • Differentiation: Agentless, graph-based, attack path focus
  • Architecture: API-only, snapshot scanning
  • Strengths: Speed of deployment, visualization, risk prioritization
  • Coverage: AWS, Azure, GCP, OCI, Alibaba, Kubernetes
  • Notable: Fastest-growing security startup ($10B+ valuation)

Palo Alto Prisma Cloud

  • Differentiation: Most comprehensive, enterprise features
  • Architecture: Agent + agentless hybrid
  • Strengths: Breadth of capabilities, compliance, enterprise support
  • Coverage: All major clouds + on-prem Kubernetes
  • Notable: Built through acquisitions (Twistlock, Bridgecrew, etc.)

Microsoft Defender for Cloud

  • Differentiation: Native Azure integration, Microsoft ecosystem
  • Architecture: Agent-based + agentless options
  • Strengths: Azure depth, Microsoft 365 integration, pricing
  • Coverage: Azure-first, AWS/GCP support
  • Notable: Included in many Microsoft licensing agreements

Orca Security

  • Differentiation: SideScanning (snapshot-based), no agents
  • Architecture: 100% agentless
  • Strengths: Deep visibility without agents, unified platform
  • Coverage: AWS, Azure, GCP, Alibaba
  • Notable: Pioneer of agentless cloud security

CrowdStrike Falcon Cloud Security

  • Differentiation: EDR heritage, threat intelligence
  • Architecture: Agent-based (Falcon sensor)
  • Strengths: Runtime protection, threat detection, XDR integration
  • Coverage: All major clouds
  • Notable: Strong in workload protection

Критерії вибору CNAPP

1. Coverage

  • Які cloud providers використовуєте?
  • Kubernetes (managed vs self-hosted)?
  • Serverless workloads?
  • On-premises components?

2. Deployment Model

  • Agentless vs agent-based vs hybrid?
  • Impact на performance?
  • Regulatory requirements (data residency)?

3. Component Depth

  • CSPM: Наскільки глибокі configuration checks?
  • CWPP: Runtime protection capabilities?
  • CIEM: Identity analysis sophistication?
  • Shift-left: IaC та CI/CD integration?

4. Risk Prioritization

  • Attack path analysis quality?
  • Context awareness (data sensitivity, business criticality)?
  • False positive rates?

5. Integrations

  • SIEM/SOAR integration?
  • Ticketing systems (Jira, ServiceNow)?
  • CI/CD pipelines?
  • API for custom integrations?

6. Total Cost of Ownership

  • Pricing model (per workload, per asset, per user)?
  • Implementation та training costs?
  • Ongoing operational overhead?

8.5 Впровадження CNAPP

Стратегія впровадження

Фаза 1: Assessment та Planning (2-4 тижні)

  • Inventory поточних security tools
  • Gap analysis: що покриваємо, що ні
  • Stakeholder alignment (Security, DevOps, Compliance)
  • Requirements gathering
  • Vendor shortlist (2-3 candidates)

Фаза 2: Proof of Concept (3-4 тижні)

  • POC з 1-2 vendors
  • Підключення до subset of environment
  • Evaluation criteria scoring
  • Technical та business decision

Фаза 3: Initial Deployment (4-6 тижнів)

  • Production deployment
  • Поступове підключення cloud accounts
  • Baseline assessment
  • Initial remediation priorities

Фаза 4: Operationalization (ongoing)

  • Integration з existing workflows
  • Alerting та escalation configuration
  • Shift-left integration (CI/CD)
  • Metrics та reporting
  • Continuous improvement

Best Practices

1. Start with visibility, not enforcement

Спочатку зрозумійте поточний стан. Не блокуйте deployments одразу.

2. Focus on attack paths, not individual findings

10 critical attack paths важливіші за 1000 isolated findings.

3. Enable DevOps self-service

  • Developer dashboards
  • Clear remediation guidance
  • IDE integrations
  • Pre-commit hooks

4. Integrate early in SDLC

Shift-left: знаходьте проблеми до deployment, а не після.

5. Consolidate tools gradually

Не відключайте existing tools одразу. Parallel running, потім migration.

6. Measure what matters

  • Critical attack paths resolved
  • Mean Time to Detect (MTTD)
  • Mean Time to Remediate (MTTR)
  • Coverage (% of assets protected)

Типові помилки

  • Boiling the ocean: Намагання виправити все одразу
  • Alert fatigue: Надто багато alerts без prioritization
  • Siloed implementation: Security team працює окремо
  • Ignoring shift-left: Тільки runtime, без CI/CD
  • Underestimating change management: Tool — лише частина рішення

8.6 Майбутнє CNAPP

Тренди розвитку

1. AI/ML Integration

  • Intelligent prioritization
  • Automated remediation recommendations
  • Anomaly detection
  • Natural language queries

2. Runtime Context

  • eBPF-based monitoring
  • Real-time behavior analysis
  • Combining posture + runtime signals

3. Application Security Integration

  • SAST/DAST integration
  • API security
  • Software composition analysis (SCA)
  • Code-to-cloud traceability

4. Multi-cloud Native

  • True multi-cloud (не окремі modules)
  • Cross-cloud attack paths
  • Unified policies across clouds

5. Platform Engineering Integration

  • Internal Developer Platforms (IDP)
  • Security as a service for developers
  • Self-service security guardrails

Gartner Predictions

  • До 2025: 75% нових CSPM purchases будуть частиною CNAPP
  • До 2026: 60% enterprises матимуть consolidated CNAPP
  • До 2027: CNAPP стане default для cloud security

Висновки

CNAPP представляє фундаментальну зміну в підході до хмарної безпеки — від набору ізольованих інструментів до єдиної інтегрованої платформи. Ключові висновки:

  • Consolidation is key: Point solutions створюють tool sprawl та alert fatigue
  • Context matters: Correlated findings важливіші за isolated alerts
  • Attack paths — це пріоритет: Фокус на реальних ризиках, а не окремих findings
  • Dev + Sec + Ops: CNAPP має підтримувати всі три команди
  • Shift-left essential: Безпека на всіх етапах lifecycle
  • Market is maturing: CNAPP стає стандартом для enterprise

Практичні рекомендації

  1. Проведіть audit поточних cloud security tools
  2. Визначте gaps у покритті (CSPM, CWPP, CIEM, etc.)
  3. Оцініть readiness для consolidation
  4. Створіть shortlist CNAPP vendors
  5. Проведіть POC з фокусом на attack path analysis
  6. Плануйте поступову migration, не big bang

У наступній лекції ми розглянемо застосування AI та Machine Learning у хмарній безпеці — як ці технології трансформують detection та response.
