Тема 3.1: Принципи та еволюція Zero Trust

Традиційна модель: "Замок та рів"

  • Периметр: firewall, DMZ, VPN
  • Всередині мережі = довірена зона
  • Зовні = ворожа територія
  • Проблема: проникнув всередину — маєш доступ до всього

Чому периметр більше не працює?

  • Хмарні сервіси: дані не в дата-центрі
  • Віддалена робота: користувачі скрізь
  • BYOD: особисті пристрої
  • Фішинг: атакуючі всередині периметра

Zero Trust: Нова парадигма

  • "Ніколи не довіряй, завжди перевіряй"
  • Кожен запит перевіряється індивідуально
  • Незалежно від розташування в мережі
  • Припускайте злам (Assume breach): припускайте, що зловмисник вже всередині

Історія Zero Trust

  • 2010: John Kindervag (Forrester) вводить термін
  • 2013: Злом Target через підрядника
  • 2014: Google запускає BeyondCorp
  • 2020: NIST SP 800-207
  • 2021: Указ президента США

NIST SP 800-207: 7 принципів (1-3)

  • 1. Усі ресурси — джерела даних, сервіси, пристрої
  • 2. Усі комунікації захищені незалежно від мережі
  • 3. Доступ надається на основі кожної сесії окремо

NIST SP 800-207: 7 принципів (4-7)

  • 4. Доступ визначається динамічною політикою
  • 5. Безпека всіх активів моніториться
  • 6. Автентифікація/авторизація — динамічні
  • 7. Збір інформації для покращення політик

Три стовпи Zero Trust

Ідентичність

MFA, SSO, IAM

Пристрій

MDM, EDR, відповідність

Мережа

mTLS, сегментація

Периметр vs Zero Trust

Ключовий принцип: мінімальних привілеїв (Least Privilege)

  • Доступ тільки до необхідних ресурсів
  • Тільки на необхідний час
  • Тільки з необхідними правами
  • Just-in-Time (JIT) замість постійного доступу

Моделі розгортання ZTA (NIST)

Агент/Шлюз

Агент на пристрої + шлюз перед ресурсом. Глибокий контроль

Анклав

Шлюз на межі групи ресурсів. Для legacy-систем

Ресурсний портал

Веб-портал/proxy. Без агента, через браузер

Порівняння моделей розгортання

  • Агент/Шлюз: Zscaler ZPA, CrowdStrike — для керованих пристроїв з високою безпекою
  • Анклав: VMware NSX, Cisco ACI — для on-premise та перехідного етапу
  • Портал: Google IAP, Cloudflare Access — для SaaS, BYOD, підрядників
  • Практика: організації комбінують кілька моделей

Trust Algorithm: на основі критеріїв

  • Policy Engine перевіряє набір бінарних критеріїв
  • Кожен: пройдено / не пройдено
  • Критерії: автентифікація, MFA, стан пристрою, геолокація, робочі години, threat intelligence
  • + детерміновано, зрозуміло для аудиту
  • − не враховує нюанси, може бути занадто жорстким

Trust Algorithm: на основі оцінки

  • Кожен фактор отримує числову оцінку (score)
  • Автентифікація: FIDO2 = 25, MFA = 20, пароль = 10
  • Пристрій: керований + відповідний = 25, BYOD = 10
  • Рішення: score ≥ порогу → дозвіл, близько → додаткова перевірка, нижче → відмова
  • Практика: гібрид — критерії + оцінка (Azure Conditional Access)
