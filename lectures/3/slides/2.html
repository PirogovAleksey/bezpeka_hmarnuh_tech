Тема 3.2: Компоненти ZTA та Identity Governance

Логічна архітектура ZTA (NIST)

Policy Engine (PE)

  • "Мозок" системи Zero Trust
  • Аналізує: суб'єкт, ресурс, контекст
  • Консультується: threat intelligence, SIEM, CDM
  • Рішення: дозволити / відмовити / умовно

Policy Administrator (PA)

  • Виконує рішення Policy Engine
  • Створює сесійні токени/облікові дані
  • Налаштовує PEP для пропуску/блокування
  • Логує рішення для аудиту

Policy Enforcement Point (PEP)

  • "Двері" до кожного ресурсу
  • Перехоплює всі запити
  • Застосовує рішення PA
  • Реалізація: API Gateway, Service Mesh, ZTNA

Допоміжні компоненти

  • CDM: стан безпеки пристроїв
  • Розвідка загроз (Threat Intelligence): інформація про загрози
  • SIEM: збір та аналіз логів
  • Політика доступу до даних: класифікація даних

Ідентичність — новий периметр

  • Замість "звідки?" → "хто?"
  • Автентифікація: підтвердження ідентичності
  • Авторизація: перевірка прав
  • IdP: Identity Provider як центр

Multi-Factor Authentication (MFA)

Знання

Пароль, PIN

Володіння

Телефон, YubiKey

Властивість

Біометрія

Умовний доступ (Conditional Access)

Міжсервісна ідентичність

  • Сервіси теж автентифікуються!
  • Workload Identity: AWS IRSA, Azure Managed Identity
  • SPIFFE/SPIRE: стандарт ідентичності робочих навантажень
  • Без збереження секретів у коді

PAM: Just-in-Time та Just-Enough-Access

  • Проблема: постійний привілейований доступ (admin 24/7) порушує Zero Trust
  • JIT: привілеї активуються на обмежений час (1-4 год) із обґрунтуванням
  • JEA: тільки мінімально необхідні дозволи для конкретного завдання
  • Break-Glass: аварійний доступ через HSM/сейф з автоалертами для SOC

Безперервна автентифікація

  • Ідея: автентифікація — не одноразова подія, а безперервний процес
  • Поведінкова біометрія: швидкість друку, рухи миші, нахил пристрою
  • Контекст сесії: зміна IP, геолокації, характеристик пристрою
  • Посилення: при зростанні ризику — повторний MFA або блокування

Рівні ризику в реальному часі

Низький (0-30)

Норма — сесія продовжується

Середній (31-60)

Обмеження чутливих ресурсів

Високий (61-85)

Повторний MFA

Критичний (86-100)

Блокування + алерт SOC

UEBA — аналітика поведінки

  • Побудова профілю: 2-4 тижні збору нормальної поведінки
  • Виявлення аномалій: ML порівнює поточну поведінку з базовою
  • Сценарії: незвичний час входу, impossible travel, масове завантаження даних
  • Рішення: Microsoft Sentinel, Amazon GuardDuty, Google Chronicle
