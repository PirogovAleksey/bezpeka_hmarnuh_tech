Тема 1.4: Shared Responsibility Model

Модель розподілу відповідальності

  • Фундаментальна концепція хмарної безпеки
  • Визначає, хто за що відповідає
  • Залежить від моделі обслуговування

Security OF vs IN the Cloud

  • Security OF the Cloud — провайдер
  • Фізична безпека, гіпервізори, мережа
  • Security IN the Cloud — клієнт
  • Дані, додатки, конфігурації, доступ

Клієнт ЗАВЖДИ відповідає за:

  • Свої дані
  • Управління доступом
  • Конфігурацію безпеки
  • Compliance власних систем

Провайдер відповідає за:

  • Фізичну безпеку дата-центрів
  • Безпеку гіпервізорів
  • Базову мережеву інфраструктуру
  • Власні сертифікації (SOC 2, ISO 27001)

Моделі розгортання

  • Public Cloud — загальнодоступна
  • Private Cloud — приватна
  • Hybrid Cloud — гібридна
  • Community Cloud — громадська

Типові помилки

  • "Провайдер відповідає за все" — НІ!
  • "Дані автоматично зашифровані" — не завжди
  • "Backup забезпечить провайдер" — не гарантовано
  • "SOC 2 провайдера = мій compliance" — НІ!

Топ загроз (CSA)

  • Неправильна конфігурація (#1)
  • Відсутність стратегії безпеки
  • Проблеми IAM
  • Небезпечні API
  • Компрометація облікових записів

Capital One Breach (2019)

  • 100+ мільйонів записів клієнтів
  • Причина: неправильна конфігурація WAF
  • SSRF атака через metadata service
  • Приклад: конфігурація — відповідальність клієнта

Ключовий висновок

Незалежно від моделі обслуговування, клієнт ЗАВЖДИ несе відповідальність за захист своїх даних та правильну конфігурацію.
