Безпека хмарних технологій

Тема 5.3: Технології ізоляції

Проблема multi-tenancy

  • Тисячі функцій різних клієнтів
  • Спільна інфраструктура
  • Ізоляція — критична вимога
  • Контейнери недостатньо безпечні

Проблема контейнерів

  • Спільне ядро Linux
  • Container escape вразливості
  • CVE-2019-5736: runc escape
  • Потрібна додаткова ізоляція

AWS Firecracker

  • MicroVM — легковагова VM
  • Запуск за ~125ms
  • ~5MB пам'яті overhead
  • Використовується в Lambda, Fargate

Firecracker архітектура

  • KVM-based — hardware virtualization
  • Мінімальний device model
  • Написаний на Rust (memory safety)
  • Тисячі VM на одному хості

Порівняння: VM vs MicroVM

  • Traditional VM: ~1GB RAM, 30+ sec boot
  • MicroVM: ~5MB RAM, 125ms boot
  • Container: ~1MB overhead, instant
  • MicroVM = безпека VM + швидкість container

Google gVisor

  • User-space kernel
  • Перехоплює syscalls
  • Реалізує 200+ syscalls безпечно
  • Використовується в GCP Cloud Run

gVisor компоненти

  • Sentry: user-space kernel
  • Gofer: file system proxy
  • Написано на Go
  • OCI-сумісний runtime (runsc)

Kata Containers

  • Легкі VM для кожного контейнера
  • OCI-сумісний runtime
  • Hardware isolation
  • Інтеграція з Kubernetes

Порівняння технологій

  • Firecracker: найкраща ізоляція, AWS-native
  • gVisor: syscall filtering, менший overhead
  • Kata: VM per container, K8s-friendly
  • Вибір залежить від threat model

Seccomp та AppArmor

  • Seccomp: фільтрація syscalls
  • AppArmor: MAC profiles
  • Додатковий рівень defence-in-depth
  • Lambda використовує обидва

Defence in Depth

  • 1. MicroVM — hardware ізоляція
  • 2. Seccomp — syscall filtering
  • 3. cgroups — resource limits
  • 4. Network isolation — VPC
