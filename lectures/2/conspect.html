Лекція 2. Стандартизація та управління ризиками

2 год

Вступ

У попередній лекції ми розглянули технічні основи хмарних обчислень — архітектуру NIST, моделі обслуговування та розподіл відповідальності. Сьогодні ми перейдемо до не менш важливої теми — стандартизації та управління ризиками в хмарних середовищах.

Чому це важливо? Уявіть, що ви — CISO (Chief Information Security Officer) компанії, яка планує мігрувати критичні системи в хмару. Як ви переконаєтесь, що хмарний провайдер надійний? Як оціните ризики? Як забезпечите відповідність регуляторним вимогам? Відповіді на ці питання дають стандарти та фреймворки, які ми сьогодні розглянемо.

Стандарти безпеки виконують кілька критичних функцій:

  • Спільна мова: дозволяють клієнтам та провайдерам говорити однією мовою про безпеку
  • Базовий рівень: визначають мінімальні вимоги до безпеки
  • Довіра: сертифікації підтверджують відповідність стандартам
  • Compliance: допомагають виконувати регуляторні вимоги

Цілі лекції

Після цієї лекції ви зможете:

  • Орієнтуватися в ключових стандартах безпеки хмарних обчислень (ISO 27017/27018, NIST, CSA)
  • Застосовувати методології оцінки ризиків для хмарних середовищ
  • Аналізувати SLA та контрактні аспекти безпеки при виборі хмарного провайдера
  • Розуміти процеси аудиту та compliance в хмарі
  • Використовувати практичні інструменти для перевірки compliance

Структура лекції

Ми рухатимемось від теорії до практики: спочатку стандарти (що вимагається), потім ризики (від чого захищаємось), далі SLA та контракти (як це оформлюється юридично), аудит (як перевіряється), і нарешті — практичні інструменти (як автоматизувати). Кожен розділ спирається на попередній.

2.1 Стандарти безпеки хмарних обчислень

Ландшафт стандартів

Існує багато стандартів та фреймворків, що стосуються безпеки хмарних обчислень. Для новачка це може виглядати заплутано — десятки абревіатур, які частково перекривають один одного. Але важливо розуміти, що ці стандарти не конкурують, а доповнюють один одного, і правильний вибір залежить від контексту організації. Їх можна класифікувати за кількома критеріями:

  • За походженням: міжнародні (ISO — International Organization for Standardization), національні (NIST — від уряду США), галузеві (PCI DSS — від платіжних систем), консорціумні (CSA — від хмарної індустрії)
  • За типом: стандарти (формальні вимоги, що підлягають сертифікації — ISO 27001), фреймворки (рекомендації та best practices без обов'язкової сертифікації — NIST CSF), benchmarks (конкретні технічні налаштування — CIS Benchmarks для AWS/Azure/GCP)
  • За фокусом: загальна інформаційна безпека (ISO 27001), хмарна безпека (ISO 27017, CSA CCM), приватність та захист даних (ISO 27018, GDPR), специфічні галузі (PCI DSS для фінансів, HIPAA для медицини)
  • За обов'язковістю: mandatory (обов'язкові) — PCI DSS, HIPAA, GDPR (порушення тягне штрафи) та voluntary (добровільні) — ISO 27001, NIST CSF, CSA STAR (прийняття — рішення організації, але часто вимагаються клієнтами)

Рекомендації щодо вибору

Вибір фреймворку — це стратегічне рішення, яке залежить від розміру організації, галузі, географії клієнтів та регуляторних вимог. Нижче — типові комбінації для різних типів організацій з поясненням, чому саме ці фреймворки:

  • Startup / SMB (Small and Medium Business): почніть з CSA CCM + SOC 2. Чому? CSA CCM — це cloud-native фреймворк, спеціально створений для хмарних сервісів, і він безкоштовний. Він дає чітку структуру контролів без надмірної бюрократії ISO. SOC 2 — це те, що першим запитають ваші B2B-клієнти (особливо американські) перед підписанням контракту. SOC 2 Type I можна отримати за 2-3 місяці та $20-50K — це доступно навіть для стартапу. Разом CSA CCM + SOC 2 покривають основні потреби безпеки та дають конкурентну перевагу на ринку без витрат, характерних для ISO сертифікації
  • Enterprise (міжнародний бізнес): ISO 27001 + 27017/27018. Чому? ISO 27001 — це золотий стандарт, визнаний у всьому світі. Якщо ви працюєте з клієнтами в ЄС, Азії, Латинській Америці — ISO 27001 буде першим, що вони запитають. Додавання ISO 27017 демонструє зрілість хмарної безпеки (специфічні контролі для cloud), а ISO 27018 покриває вимоги щодо захисту персональних даних, що критично для відповідності GDPR. Ця комбінація — універсальна "мова довіри" для міжнародного бізнесу
  • US Government contractor (підрядник уряду США): NIST 800-53 + FedRAMP. Чому? Це не рекомендація, а обов'язкова вимога. Федеральний уряд США не купуватиме хмарний сервіс без авторизації FedRAMP, яка базується на контролях NIST SP 800-53. Рівень авторизації (Low, Moderate, High) визначається класифікацією даних, що оброблюються. Процес коштує $500K-$3M та займає 6-18 місяців, але відкриває доступ до ринку федеральних контрактів обсягом понад $100B на рік
  • Healthcare (охорона здоров'я, США): HIPAA + HITRUST. Чому? HIPAA — це федеральний закон, дотримання якого обов'язкове для будь-кої організації, що обробляє медичні дані (PHI) в США. Однак HIPAA не має формальної сертифікації — немає "сертифікату HIPAA". Тут на допомогу приходить HITRUST CSF (Health Information Trust Alliance Common Security Framework) — фреймворк, що об'єднує вимоги HIPAA, NIST, ISO та PCI DSS в єдину систему з формальною сертифікацією. HITRUST-сертифікат став де-факто стандартом для підтвердження HIPAA compliance та все частіше вимагається лікарнями та страховими компаніями від своїх хмарних постачальників
  • Фінанси (банки, платіжні сервіси, fintech): PCI DSS + SOC 2 Type II. Чому? PCI DSS — обов'язковий для будь-якої організації, що зберігає, обробляє або передає дані платіжних карток. Це не опція — без PCI DSS compliance ви не зможете працювати з платіжними системами Visa/Mastercard. SOC 2 Type II доповнює PCI DSS, покриваючи ширший спектр безпеки (не тільки карткові дані) та демонструючи стабільність контролів протягом 6-12 місяців. Для фінансових організацій саме Type II є критичним — партнери та регулятори хочуть бачити, що безпека працює стабільно, а не лише "на папері" в один конкретний день

ISO/IEC 27017:2015

ISO/IEC 27017 — це міжнародний стандарт, що надає спеціалізовані рекомендації з інформаційної безпеки для хмарних сервісів. Формальна назва — "Code of practice for information security controls based on ISO/IEC 27002 for cloud services". Стандарт розширює ISO/IEC 27002 (загальний кодекс практики з безпеки, який описує імплементацію контролів з Annex A стандарту 27001) додатковими контролями, специфічними для хмарних обчислень.

Головна цінність ISO 27017 полягає в тому, що він адресує унікальні аспекти хмарної безпеки, які не покриваються загальними стандартами: багатоорендність (multi-tenancy), віртуалізація, розподіл відповідальності між провайдером і клієнтом, особливості видалення даних у хмарі. Для кожного контролю стандарт чітко розмежовує, що має робити хмарний провайдер (CSP — Cloud Service Provider), а що — клієнт (CSC — Cloud Service Customer).

Ключові особливості ISO 27017

  • Базується на ISO/IEC 27002 — адаптує існуючі контролі для хмарного контексту та додає 7 абсолютно нових контролів (ідентифікатори CLD.*)
  • Розділяє відповідальність — для кожного контролю визначено, хто відповідає: CSP (Cloud Service Provider), CSC (Cloud Service Customer), або обидва
  • Універсальність — застосовується до будь-якої моделі обслуговування (IaaS, PaaS, SaaS) та розгортання (public, private, hybrid, community cloud)
  • Доповнює, а не замінює — ISO 27017 не є самостійним стандартом для сертифікації; організація спочатку має впровадити ISO 27001, а потім розширити його хмарними контролями з 27017

ISO/IEC 27018:2019

ISO/IEC 27018 — перший міжнародний стандарт, який спеціально фокусується на захисті персональних даних (PII — Personally Identifiable Information) у публічних хмарах. Формальна назва — "Code of practice for protection of personally identifiable information (PII) in public clouds acting as PII processors". PII — це будь-яка інформація, що дозволяє прямо чи опосередковано ідентифікувати фізичну особу: ім'я, email, IP-адреса, номер телефону, біометричні дані тощо.

Цей стандарт особливо важливий у контексті GDPR (General Data Protection Regulation — регламент ЄС про захист персональних даних), CCPA (California Consumer Privacy Act — закон Каліфорнії про конфіденційність споживачів) та інших регуляцій приватності. Він надає хмарним провайдерам конкретні вказівки щодо того, як обробляти PII відповідно до міжнародних стандартів приватності.

Стандарт розрізняє дві ролі: PII Controller (організація-клієнт, яка визначає цілі та засоби обробки персональних даних) та PII Processor (хмарний провайдер, який обробляє PII за інструкціями Controller'а). ISO 27018 містить вимоги саме до Processor'ів.

Ключові принципи ISO 27018

Стандарт базується на загальновизнаних принципах захисту приватності, що перегукуються з GDPR:

  • Consent and choice (Згода та вибір): обробка PII виключно за явною згодою суб'єкта даних (data subject — фізична особа, чиї дані обробляються). Провайдер не може обробляти дані для цілей, на які суб'єкт не давав згоди
  • Purpose legitimacy (Легітимність мети): обробка даних тільки для визначених, чітко сформульованих та законних цілей. Провайдер не може змінити мету обробки без повідомлення та згоди
  • Data minimization (Мінімізація даних): збір та обробка тільки тих даних, які мінімально необхідні для досягнення визначеної мети. Зайві дані не збираються та не зберігаються
  • Transparency (Прозорість): відкритість щодо практик обробки даних — суб'єкт має знати, які дані збираються, де зберігаються, кому передаються, протягом якого часу зберігаються
  • Accountability (Підзвітність): провайдер несе відповідальність за дотримання всіх принципів та повинен бути здатний продемонструвати відповідність (demonstrate compliance)

Вимоги до провайдера за ISO 27018

  • Заборона використання PII для маркетингу — провайдер не має права використовувати персональні дані клієнтів для реклами, профілювання чи аналітики без явної згоди. Це прямо стосується практик Big Tech компаній
  • Прозорість щодо субпідрядників (sub-processors) — провайдер зобов'язаний розкривати повний перелік третіх сторін, які мають доступ до PII, та повідомляти про будь-які зміни у цьому переліку заздалегідь
  • Сповіщення про запити правоохоронних органів — якщо провайдер отримує запит від поліції чи суду на розкриття даних клієнта, він повинен повідомити клієнта (якщо це не заборонено законом)
  • Право на видалення (erasure) — забезпечити можливість повного та безповоротного видалення PII на вимогу клієнта, включаючи резервні копії (backups)
  • Data residency disclosure — провайдер зобов'язаний документувати та розкривати клієнтам конкретні країни та регіони, де зберігаються та обробляються персональні дані
  • Breach notification — оперативне повідомлення клієнта про будь-які інциденти безпеки, що стосуються PII, із зазначенням масштабу та вжитих заходів

ISO/IEC 27001:2022

ISO/IEC 27001 — базовий міжнародний стандарт для створення, впровадження, підтримки та безперервного вдосконалення системи управління інформаційною безпекою (ISMS — Information Security Management System). ISMS — це не окремий продукт чи інструмент, а комплексний набір політик, процедур, технічних контролів та організаційних заходів, що разом забезпечують систематичний підхід до управління конфіденційною інформацією. Хоча ISO 27001 не специфічний для хмари, це фундамент, на якому будуються 27017 та 27018.

Стандарт побудований на основі циклу PDCA (Plan-Do-Check-Act), також відомого як цикл Демінга — методології безперервного вдосконалення:

  • Plan (Планування): визначення scope ISMS, оцінка ризиків, вибір контролів — на цьому етапі організація аналізує свої активи, загрози та вразливості
  • Do (Впровадження): імплементація обраних контролів, навчання персоналу, впровадження процедур
  • Check (Перевірка): моніторинг ефективності контролів, внутрішній аудит, management review — перевірка чи контролі працюють як очікувалося
  • Act (Покращення): коригувальні дії, оновлення політик на основі результатів моніторингу та аудитів

Структура ISO 27001 складається з основних розділів (clauses 4-10), кожен з яких відповідає за окремий аспект ISMS:

  • Clause 4 — Контекст організації: розуміння зовнішнього та внутрішнього середовища, визначення зацікавлених сторін (stakeholders) та їхніх очікувань щодо безпеки, визначення scope (меж) ISMS
  • Clause 5 — Лідерство: підтримка з боку топ-менеджменту, визначення політики інформаційної безпеки, розподіл ролей та відповідальностей. Без commitment керівництва ISMS приречена на провал
  • Clause 6 — Планування: оцінка ризиків (risk assessment), визначення risk treatment plan, формулювання цілей безпеки (security objectives) з вимірюваними показниками
  • Clause 7 — Підтримка: забезпечення ресурсами (бюджет, персонал, інструменти), розвиток компетенцій, awareness-програми, документація
  • Clause 8 — Операції: впровадження плану обробки ризиків, операційне планування та контроль, виконання risk assessments
  • Clause 9 — Оцінка продуктивності: моніторинг, вимірювання, аналіз та оцінка ISMS; внутрішній аудит; management review
  • Clause 10 — Покращення: обробка невідповідностей (nonconformities), коригувальні дії, безперервне вдосконалення ISMS

Annex A стандарту містить каталог з 93 контролів (у версії 2022), організованих у 4 категорії: організаційні (37 контролів), контролі для людей (8), фізичні (14) та технологічні (34). Раніше (у версії 2013) було 114 контролів у 14 доменах. Організація обирає з цього каталогу контролі, релевантні для її ризиків, та документує обґрунтування у Statement of Applicability (SoA) — ключовому документі ISMS, що пояснює, які контролі обрані, чому, і як вони впроваджені.

NIST Cybersecurity Framework (CSF)

NIST CSF (Cybersecurity Framework) — фреймворк, розроблений NIST (National Institute of Standards and Technology) — Національним інститутом стандартів і технологій США. NIST — це федеральна агенція, що входить до Міністерства торгівлі США і є одним з найавторитетніших джерел стандартів та рекомендацій з кібербезпеки у світі.

CSF був створений у 2014 році на виконання Executive Order Президента Обами для захисту критичної інфраструктури США, однак завдяки своїй універсальності та практичності він став де-факто стандартом для організацій усіх розмірів і галузей по всьому світу. У 2024 році вийшла версія CSF 2.0, яка додала шосту функцію — Govern (Управління) — та посилила фокус на governance та supply chain security.

Ключова перевага NIST CSF — це його гнучкість. На відміну від ISO 27001, який вимагає формальної сертифікації, CSF є добровільним та дозволяє організаціям адаптувати його під свої потреби. Він описує "що" потрібно робити, а не "як", залишаючи реалізацію на розсуд організації.

П'ять основних функцій NIST CSF

  • Identify (Ідентифікація): розуміння бізнес-контексту, активів, ризиків та governance. Включає asset management (інвентаризація всіх ресурсів), business environment (розуміння критичних процесів), governance (політики та процедури), risk assessment (оцінка ризиків) та risk management strategy
  • Protect (Захист): впровадження safeguards (захисних заходів) для забезпечення надання критичних сервісів. Включає access control (управління доступом), awareness training (навчання персоналу), data security (захист даних), information protection processes, maintenance та protective technology
  • Detect (Виявлення): своєчасне виявлення кіберінцидентів. Включає anomalies and events detection (виявлення аномалій), continuous security monitoring (безперервний моніторинг) та detection processes — використання інструментів, таких як SIEM (Security Information and Event Management — система кореляції безпекових подій), IDS (Intrusion Detection System — система виявлення вторгнень) та IPS (Intrusion Prevention System — система запобігання вторгненням)
  • Respond (Реагування): вжиття заходів при виявленні інциденту. Включає response planning (план реагування), communications (комунікації з зацікавленими сторонами), analysis (аналіз інциденту), mitigation (зменшення наслідків) та improvements (покращення на основі досвіду)
  • Recover (Відновлення): повернення до нормальної роботи після інциденту. Включає recovery planning (план відновлення), improvements (вдосконалення на основі lessons learned) та communications (інформування зацікавлених сторін про хід відновлення)

NIST SP 800-53 Rev. 5 — Детальний огляд

NIST SP 800-53 (Special Publication 800-53) — документ під назвою "Security and Privacy Controls for Information Systems and Organizations" — це найповніший та найдетальніший каталог security controls (заходів безпеки) у світі. Якщо NIST CSF — це високорівневий фреймворк, що описує "що робити", то SP 800-53 — це детальний довідник, що описує "чим саме захищатися". Версія Rev. 5 (2020) містить понад 1000 контролів у 20 сімействах (families). Кожен контроль має базовий рівень та додаткові enhancements (розширення), які поглиблюють захист.

SP 800-53 є обов'язковим для федеральних агенцій США та основою програми FedRAMP (Federal Risk and Authorization Management Program — програма авторизації хмарних сервісів для федерального уряду). Однак його використовують і комерційні організації як еталонний каталог best practices.

Приклад контролю: AC-6 Least Privilege

AC-6 Least Privilege (Принцип мінімальних привілеїв) — один з найважливіших контролів у каталозі. Суть принципу: кожен користувач, процес або система повинні мати лише ті права доступу, які мінімально необхідні для виконання їхніх функцій — і не більше. Базовий контроль має кілька enhancements (розширень), що деталізують вимоги:

  • Базовий контроль AC-6: організація застосовує принцип мінімальних привілеїв, надаючи доступ лише до тих ресурсів і операцій, які необхідні для виконання призначених завдань
  • AC-6(1): авторизація доступу тільки для конкретних функцій — користувач отримує доступ не "до сервера", а до конкретних операцій на сервері
  • AC-6(2): непривілейований доступ для несекюріті функцій — адміністратор, який перевіряє пошту, має робити це з непривілейованого акаунту, а не з root/admin
  • AC-6(3): мережевий доступ до привілейованих команд — обмеження того, з яких мережевих сегментів можна виконувати привілейовані дії (наприклад, SSH до production лише з management VPN)
  • AC-6(5): привілейовані акаунти обмежені конкретним персоналом — чіткий список осіб, що мають root/admin доступ, з регулярним переглядом
  • AC-6(9): логування привілейованих функцій — всі дії, виконані з підвищеними правами, мають записуватись (CloudTrail, Azure Activity Log)
  • AC-6(10): заборона виконання привілейованих функцій непривілейованими користувачами — технічне обмеження, а не лише організаційна заборона

CSA Cloud Controls Matrix (CCM)

Cloud Security Alliance (CSA) — провідна міжнародна галузева організація, заснована у 2008 році, що спеціалізується саме на безпеці хмарних обчислень. На відміну від ISO (який охоплює всі галузі) та NIST (який фокусується на федеральних потребах США), CSA створена "хмарною індустрією для хмарної індустрії" і об'єднує понад 80,000 членів — фахівців з безпеки, хмарних провайдерів та організацій-споживачів.

CCM (Cloud Controls Matrix) — флагманський фреймворк CSA, спеціально розроблений для оцінки безпеки хмарних сервісів та провайдерів. Головна перевага CCM — це його mapping (відображення) на інші стандарти: кожен контроль CCM має відповідність до ISO 27001, NIST SP 800-53, PCI DSS, HIPAA та інших фреймворків, що дозволяє організаціям одночасно відповідати вимогам кількох стандартів.

Структура CCM v4

CCM v4 містить 197 контролів, організованих у 17 доменах. Кожен домен охоплює окрему сферу хмарної безпеки:

  • A&A — Audit & Assurance: вимоги до аудиту та підтвердження відповідності — як провайдер демонструє ефективність контролів
  • AIS — Application & Interface Security: безпека додатків та інтерфейсів — захист API, secure SDLC (Software Development Lifecycle), input validation
  • BCR — Business Continuity & Operational Resilience: безперервність бізнесу та операційна стійкість — disaster recovery, backup, redundancy
  • CCC — Change Control & Configuration Management: контроль змін та управління конфігураціями — як відстежуються та затверджуються зміни в інфраструктурі
  • CEK — Cryptography, Encryption & Key Management: криптографія та управління ключами — алгоритми шифрування, зберігання ключів, ротація, HSM (Hardware Security Module)
  • DCS — Data Center Security: фізична безпека дата-центрів — доступ, відеоспостереження, захист від стихійних лих
  • DSP — Data Security & Privacy: безпека та конфіденційність даних — класифікація, шифрування, retention, disposal, захист PII
  • GRC — Governance, Risk & Compliance: управління, ризики та відповідність — політики, процедури, risk management, regulatory compliance
  • HRS — Human Resources Security: безпека персоналу — background checks, awareness training, offboarding procedures (відкликання доступу при звільненні)
  • IAM — Identity & Access Management: управління ідентифікацією та доступом — автентифікація, авторизація, RBAC, MFA, privileged access
  • IPY — Interoperability & Portability: сумісність та портативність — можливість міграції даних та workloads між провайдерами, стандартні формати, API
  • IVS — Infrastructure & Virtualization Security: безпека інфраструктури та віртуалізації — ізоляція tenants, hardening гіпервізорів, мережева безпека
  • LOG — Logging & Monitoring: логування та моніторинг — збір, зберігання та аналіз логів, alerting, SIEM інтеграція
  • SEF — Security Incident Management: управління інцидентами безпеки — incident response plan, notification procedures, forensics
  • STA — Supply Chain, Transparency & Accountability: ланцюг постачання та прозорість — оцінка субпідрядників, розкриття sub-processors, third-party risk management
  • TVM — Threat & Vulnerability Management: управління загрозами та вразливостями — vulnerability scanning, penetration testing, patch management, threat intelligence
  • UEM — Universal Endpoint Management: управління кінцевими пристроями — BYOD (Bring Your Own Device) політики, MDM (Mobile Device Management), endpoint protection

CSA STAR Program

STAR (Security, Trust, Assurance, and Risk) — програма сертифікації CSA, що дозволяє хмарним провайдерам публічно продемонструвати рівень своєї безпеки. STAR реєстр доступний публічно на сайті CSA — будь-хто може перевірити, чи зареєстрований провайдер і на якому рівні. Це найбільший у світі реєстр безпеки хмарних провайдерів.

  • Level 1 — Self-Assessment (Самооцінка): провайдер самостійно заповнює CAIQ (Consensus Assessments Initiative Questionnaire) — стандартизовану анкету з понад 260 питань, що покривають усі 17 доменів CCM. Це безкоштовний рівень, який є хорошим початком, але не передбачає незалежної перевірки
  • Level 2 — Third-Party Audit (Незалежний аудит): провайдер проходить аудит від незалежної акредитованої організації. Два варіанти: CSA STAR Certification (на основі ISO 27001) або CSA STAR Attestation (на основі SOC 2). Результати публікуються у STAR реєстрі. Це значно ціннішій рівень, оскільки підтверджує заявлені контролі незалежною стороною
  • Level 3 — Continuous Monitoring (Безперервний моніторинг): найвищий рівень, що передбачає автоматизований безперервний моніторинг безпекових контролів у реальному часі. Знаходиться в процесі розробки та пілотування

Галузеві стандарти

Окрім загальних стандартів безпеки, існують галузеві стандарти, що встановлюють специфічні вимоги для певних секторів економіки. Ці стандарти зазвичай є обов'язковими (mandatory), а не добровільними, і за їх порушення передбачені значні штрафи.

PCI DSS (Payment Card Industry Data Security Standard)

PCI DSS — стандарт безпеки даних платіжних карток, розроблений PCI SSC (Payment Card Industry Security Standards Council) — організацією, заснованою п'ятьма найбільшими платіжними системами: Visa, Mastercard, American Express, Discover та JCB. Стандарт є обов'язковим для будь-якої організації, що зберігає, обробляє або передає дані власників карток (cardholder data) — номери карток (PAN — Primary Account Number), дати закінчення дії, CVV-коди, імена власників.

  • 12 вимог у 6 категоріях: від побудови захищеної мережі (firewall, зміна стандартних паролів) до регулярного тестування систем безпеки (penetration testing, vulnerability scanning)
  • Рівні compliance: залежать від обсягу транзакцій — Level 1 (понад 6 млн транзакцій/рік) вимагає щорічного аудиту QSA (Qualified Security Assessor), Level 4 (менше 20 тис.) — self-assessment questionnaire (SAQ)
  • Cloud-specific вимоги: PCI DSS v4.0 (2024) містить спеціальні вказівки для хмарних середовищ, включаючи shared responsibility matrix — чіткий розподіл відповідальності між провайдером та клієнтом. Наприклад, у моделі IaaS провайдер відповідає за фізичну безпеку, а клієнт — за шифрування cardholder data та управління доступом
  • Scope reduction: для зменшення scope PCI DSS у хмарі рекомендується токенізація (заміна реальних номерів карток на токени) та використання сертифікованих payment gateway'ів

HIPAA (Health Insurance Portability and Accountability Act)

HIPAA — федеральний закон США (1996), який регулює захист медичної інформації. Закон застосовується до covered entities (медичні установи, страхові компанії, аптеки) та їхніх business associates (будь-які організації, що обробляють медичні дані від їхнього імені, включаючи хмарних провайдерів). Захищувана інформація називається PHI (Protected Health Information) — будь-які дані, що стосуються стану здоров'я, лікування або оплати медичних послуг і можуть ідентифікувати конкретну особу.

  • Privacy Rule — визначає, хто і за яких умов може отримати доступ до PHI, права пацієнтів на доступ до своїх даних та обмеження на розкриття інформації
  • Security Rule — встановлює технічні, адміністративні та фізичні safeguards (захисні заходи) для електронної PHI (ePHI): шифрування, access control, audit logging, integrity controls
  • Breach Notification Rule — вимагає повідомлення пацієнтів, HHS (Department of Health and Human Services) та ЗМІ (при breaches понад 500 осіб) про порушення безпеки PHI
  • BAA (Business Associate Agreement) — обов'язковий юридичний договір між covered entity та хмарним провайдером. Без підписаного BAA зберігання PHI у хмарі є порушенням HIPAA. Усі великі провайдери (AWS, Azure, GCP) пропонують BAA для своїх HIPAA-eligible сервісів
  • Штрафи: від $100 до $50,000 за порушення, максимум $1.5M на рік за однотипні порушення. У 2023 році Advocate Health Care отримав штраф $5.55M за незашифровані ноутбуки з ePHI

SOC 2 (Service Organization Control 2)

SOC 2 — аудиторський стандарт, розроблений AICPA (American Institute of Certified Public Accountants) — Американським інститутом сертифікованих бухгалтерів. SOC 2 став де-факто стандартом для SaaS та хмарних компаній, оскільки клієнти все частіше вимагають SOC 2 звіт як підтвердження належної безпеки перед початком співпраці. Це не сертифікат (як ISO 27001), а незалежний аудиторський звіт.

  • Trust Service Criteria (TSC) — п'ять критеріїв, за якими оцінюється організація:
    • Security (обов'язковий): захист від несанкціонованого доступу — firewalls, MFA, encryption, intrusion detection
    • Availability: доступність системи відповідно до SLA — redundancy, disaster recovery, incident monitoring
    • Processing Integrity: точність та повнота обробки даних — quality assurance, process monitoring
    • Confidentiality: захист конфіденційної інформації — encryption, access controls, data classification
    • Privacy: захист персональних даних відповідно до privacy notice — consent, data retention, disposal
  • Type I: оцінка дизайну контролів на конкретну дату — "чи правильно спроєктовані контролі?" (знімок стану на один день)
  • Type II: оцінка операційної ефективності контролів за період (зазвичай 6-12 місяців) — "чи контролі реально працюють протягом тривалого часу?" Type II значно цінніший для клієнтів, оскільки доводить стабільність безпеки
  • SOC 2+: деякі організації додають специфічні критерії, наприклад CSA CCM або HIPAA, створюючи комбінований звіт

2.2 Управління ризиками в хмарі

Що таке ризик?

У контексті інформаційної безпеки, ризик (risk) — це комбінація ймовірності того, що загроза експлуатує вразливість, та потенційного впливу (збитків) від її реалізації. Це не абстрактне поняття — ризик можна і потрібно вимірювати, оцінювати та управляти ним:

Ризик = Ймовірність (Likelihood) × Вплив (Impact)

Розуміння ризику починається з чіткого розмежування ключових термінів — їх часто плутають навіть досвідчені фахівці:

  • Загроза (Threat): потенційна причина небажаного інциденту. Загрози можуть бути зовнішніми (хакери, стихійні лиха, конкуренти) та внутрішніми (недбалі співробітники, зловмисні інсайдери). У хмарному контексті загрози включають: misconfiguration (помилки конфігурації), credential theft (крадіжка облікових даних), DDoS-атаки (Distributed Denial of Service — розподілена атака на відмову в обслуговуванні, коли тисячі машин одночасно перевантажують ціль запитами), insider threats з боку провайдера
  • Вразливість (Vulnerability): слабкість у системі, процесі або конфігурації, яку може експлуатувати загроза. Наприклад: незашифрований S3 bucket, відсутність MFA, непропатчена CVE, надмірні IAM permissions. Вразливість сама по собі не є ризиком — вона стає ризиком лише у поєднанні з загрозою
  • Актив (Asset): будь-що, що має цінність для організації і потребує захисту. У хмарі це: дані клієнтів, бази даних, API ключі, інтелектуальна власність, обчислювальні ресурси, репутація бренду. Класифікація активів (data classification) — перший крок в управлінні ризиками
  • Контроль (Control / Safeguard): технічний, організаційний або фізичний захід для зменшення ризику. Контролі бувають превентивні (запобігають інциденту — MFA, encryption), детективні (виявляють інцидент — моніторинг, IDS) та коригувальні (мінімізують наслідки — backup, incident response plan)
  • Вплив (Impact): наслідки реалізації загрози — фінансові збитки, репутаційний збиток, юридичні наслідки, втрата даних, простій бізнесу
  • Risk appetite (Апетит до ризику): рівень ризику, який організація готова прийняти. Визначається на рівні board of directors та є основою для прийняття рішень з безпеки

Специфіка ризиків у хмарі

Хмарні обчислення створюють унікальні ризики, яких немає в традиційних середовищах. Розуміння цих ризиків — перший крок до їх мітигації.

1. Втрата контролю

Це, мабуть, найфундаментальніший ризик хмари. Коли ви переносите дані та системи до хмарного провайдера, ви фізично не контролюєте інфраструктуру. Ви не можете підійти до сервера, не можете перевірити, хто має фізичний доступ до вашого обладнання, і не завжди знаєте, що саме провайдер робить "під капотом".

  • Дані поза вашим периметром — інфраструктура фізично розташована в дата-центрах провайдера, доступ до яких ви не маєте
  • Залежність від безпеки провайдера — якщо провайдер допустить помилку, ваші дані під загрозою, навіть якщо ви зробили все правильно
  • Обмежена видимість — ви не завжди знаєте, як саме провайдер оброблює ваші запити, де зберігаються тимчасові копії даних, хто з персоналу має доступ

Як мітигувати: шифрування на стороні клієнта (client-side encryption), customer-managed keys (CMK), детальне логування через CloudTrail/Azure Monitor/Cloud Audit Logs.

2. Multi-tenancy ризики

У хмарі ваші робочі навантаження працюють на тому ж фізичному обладнанні, що й інших клієнтів. Хоча провайдери вкладають величезні ресурси в ізоляцію, сам факт спільного використання створює вектори атак, які неможливі в ізольованому дата-центрі.

  • Side-channel attacks — атаки типу Spectre/Meltdown, що дозволяють "підслуховувати" дані сусіднього tenant'а через спільний процесор
  • Noisy neighbors — інший tenant може споживати непропорційно багато ресурсів, впливаючи на вашу продуктивність
  • Data leakage — помилки в ізоляції можуть призвести до того, що один клієнт отримає доступ до даних іншого

Як мітигувати: dedicated hosts/instances для критичних навантажень, шифрування всіх даних, моніторинг аномальної поведінки.

3. Vendor lock-in

Vendor lock-in — це не просто технічна незручність. Це стратегічний ризик. Чим глибше ви інтегруєтесь у специфічні сервіси одного провайдера, тим складніше та дорожче буде мігрувати, якщо виникне потреба. А потреба може виникнути з різних причин: зростання цін, зміна умов, санкції, банкрутство провайдера.

  • Пропрієтарні API — AWS Lambda, Azure Functions, Cloud Functions мають різні API, тому код FaaS не переноситься між провайдерами
  • Формати даних — деякі бази даних (DynamoDB, Cosmos DB, Spanner) мають унікальні моделі даних, що ускладнює міграцію
  • Зміна умов — провайдер може змінити ціни, припинити підтримку сервісу або змінити SLA

Як мітигувати: використовуйте portable технології (Kubernetes, Terraform), проєктуйте abstraction layers, регулярно тестуйте exit strategy.

4. Data sovereignty

Де фізично знаходяться ваші дані? Це питання, яке здається простим, але в хмарі може бути неочевидним. Ваші дані можуть реплікуватися між регіонами, тимчасові копії можуть створюватися в іншій юрисдикції, а backup може зберігатися на іншому континенті — все це без вашого явного відома.

  • Юрисдикція — дані, що зберігаються в США, підпадають під CLOUD Act (уряд США може вимагати доступ); дані в ЄС — під GDPR
  • Cross-border transfers — передача даних між країнами може порушувати локальне законодавство (після Schrems II — це особливо актуально для ЄС-США)
  • Локальні вимоги — деякі країни (Китай, Росія, Індія) вимагають зберігання даних громадян на території країни (data localization)

Як мітигувати: чітко обирайте регіони зберігання, використовуйте data residency policies, перевіряйте наявність регіональних endpoint'ів.

5. Supply chain ризики

Ваш хмарний провайдер — це не монолітна компанія. Він сам залежить від десятків субпідрядників: постачальників обладнання, розробників програмного забезпечення, CDN-провайдерів. Компрометація будь-якої ланки цього ланцюга може вплинути на вас.

  • Субпідрядники провайдера — хто має фізичний доступ до обладнання? Хто обслуговує мережу?
  • Third-party компоненти — open-source бібліотеки, от Log4j вразливість (2021) торкнулася мільйонів хмарних систем
  • Supply chain атаки — SolarWinds (2020) показав, як компрометація одного постачальника може торкнутися 18,000+ організацій, включаючи урядові агенції

Як мітигувати: перевіряйте sub-processor lists провайдера, використовуйте SCA (Software Composition Analysis) для своїх залежностей, моніторте CVE для критичних компонентів.

Стратегії обробки ризиків

Після ідентифікації ризику є чотири основні стратегії:

  1. Уникнення (Avoid) — не використовувати технологію або процес, що створює ризик. Наприклад, не зберігати PII в публічній хмарі, якщо регулятор це забороняє.
  2. Мітигація (Mitigate) — зменшити ймовірність або вплив. Шифрування, MFA, моніторинг — це все мітигація. Найпоширеніша стратегія.
  3. Передача (Transfer) — передати ризик іншій стороні. Кіберстрахування (cyber insurance) — класичний приклад. SLA з компенсаціями — інший. Але пам'ятайте: ви передаєте фінансовий ризик, не репутаційний.
  4. Прийняття (Accept) — свідомо прийняти ризик. Це валідна стратегія для низькоймовірних або низькоімпактних ризиків. Ключове слово — "свідомо". Це має бути задокументоване рішення, а не ігнорування.

Реальні кейси порушення Compliance

Capital One (2019) — $80M штраф

Що сталося:

  • Колишня співробітниця AWS експлуатувала SSRF (Server-Side Request Forgery) вразливість — тип атаки, при якій зловмисник змушує сервер робити запити до внутрішніх ресурсів, до яких він не повинен мати доступу
  • Отримала доступ до 106 млн записів клієнтів (імена, адреси, кредитні рейтинги, номери соціального страхування)
  • Використала неправильно сконфігурований WAF (Web Application Firewall) — міжмережевий екран для веб-додатків, який мав надмірно широку IAM role (роль з надлишковими правами доступу)

Порушені контролі:

  • AC-6 Least Privilege: WAF мав доступ до всіх S3 buckets замість лише тих, що необхідні для роботи
  • CM-2 Baseline Configuration: використовувався IMDS v1 (Instance Metadata Service) — сервіс метаданих EC2-інстансів, перша версія якого вразлива до SSRF (v2 вимагає токен і захищена від цієї атаки)
  • AU-6 Audit Review: аномальна активність (масове завантаження даних з S3) не була виявлена та розслідувана вчасно

Наслідки: $80M штраф від OCC (Office of the Comptroller of the Currency) — регулятора банківської системи США, $190M class action settlement (колективний позов), значні репутаційні втрати

Marriott (2018) — €20.4M GDPR штраф

Що сталося:

  • Breach (витік даних) системи бронювання Starwood, яку Marriott придбала у 2016 році в ході M&A (Mergers & Acquisitions — злиття та поглинання)
  • 383 млн записів гостей скомпрометовано — імена, адреси, паспортні дані, номери кредитних карток
  • Зловмисники мали доступ з 2014 року — 4 роки непомітно! Marriott виявила breach лише у 2018, через 2 роки після придбання Starwood

Порушені контролі:

  • RA-3 Risk Assessment: недостатній due diligence (належна перевірка) безпеки Starwood при M&A — Marriott не провела повний security assessment придбаної компанії перед інтеграцією систем
  • SI-4 System Monitoring: відсутність detection capabilities — системи моніторингу не виявили присутність зловмисників протягом років
  • IR-4 Incident Handling: 4 роки без виявлення свідчать про повну відсутність ефективного incident detection процесу

GDPR порушення: Art. 5(1)(f) — integrity and confidentiality, Art. 32 — security of processing

Equifax (2017) — $700M settlement

Що сталося: Equifax — одне з трьох найбільших кредитних бюро США, що зберігає фінансову інформацію практично на кожного дорослого американця.

  • Вразливість у Apache Struts (популярний Java web framework) — CVE-2017-5638 (ідентифікатор у базі Common Vulnerabilities and Exposures) — не була пропатчена протягом 2 місяців після випуску патча
  • 147 млн записів американців скомпрометовано — це майже половина дорослого населення США
  • Викрадені дані включали: SSN (Social Security Number) — номер соціального страхування (найцінніший ідентифікатор у США, що використовується для кредитних перевірок, податків, відкриття рахунків), дати народження, адреси, номери водійських посвідчень

Порушені контролі:

  • RA-5 Vulnerability Scanning: vulnerability була відома
  • SI-2 Flaw Remediation: патч не застосований вчасно
  • CM-3 Configuration Change Control: відсутній процес

British Airways (2018) — £20M GDPR штраф

Що сталося:

  • Magecart attack — назва групи кіберзлочинців (та типу атаки), що спеціалізується на крадіжці платіжних даних через впровадження зловмисного JavaScript-коду на сайти електронної комерції (digital skimming)
  • Понад 400,000 клієнтів постраждали — викрадені імена, адреси, номери кредитних карток, CVV-коди
  • Зловмисний скрипт (всього 22 рядки коду) був впроваджений у сторінку оплати та перенаправляв копію всіх введених платіжних даних на сервер атакувальників у реальному часі

Уроки для хмарної безпеки:

  • Supply chain security для JavaScript бібліотек — перевірка цілісності зовнішніх скриптів, що завантажуються на сторінку
  • CSP (Content Security Policy) headers — HTTP-заголовок, що обмежує, з яких джерел браузер може завантажувати скрипти. Правильно налаштований CSP заблокував би зловмисний скрипт
  • SRI (Subresource Integrity) — механізм, що дозволяє браузеру перевірити, чи зовнішній ресурс (JavaScript, CSS) не був модифікований, порівнюючи його hash із очікуваним значенням

Процес управління ризиками

Типовий процес управління ризиками включає наступні етапи:

Процес управління ризиками: п'ять етапів
Рис. 2.3. Циклічний процес управління ризиками в хмарних середовищах

1. Ідентифікація ризиків

Перший і найважливіший крок — зрозуміти, що саме потрібно захищати і від чого:

  • Інвентаризація активів (Asset Inventory): створення повного переліку всіх активів — дані (бази клієнтів, фінансові записи, інтелектуальна власність), системи (сервери, бази даних, API), сервіси (хмарні підписки, SaaS-додатки). У хмарі це ускладнюється динамічністю: ресурси з'являються і зникають автоматично через auto-scaling
  • Ідентифікація загроз (Threat Identification): визначення потенційних джерел загроз. Використовуються реєстри: CSA Top Threats (специфічно для хмари), ENISA (European Union Agency for Cybersecurity — Агентство кібербезпеки Європейського Союзу) Threat Landscape (європейська перспектива загроз), MITRE ATT&CK Cloud Matrix — структурована база знань від некомерційної організації MITRE, що каталогізує тактики (tactics), техніки (techniques) та процедури (procedures) атакуючих у хмарних середовищах. ATT&CK допомагає відповісти на питання "як саме атакують хмару?"
  • Оцінка вразливостей (Vulnerability Assessment): пошук слабких місць — vulnerability scanning (автоматизований пошук відомих CVE), configuration review (перевірка конфігурацій на відповідність CIS Benchmarks), code review (аналіз коду на вразливості)

2. Аналіз ризиків

Аналіз ризиків може бути якісним, кількісним або комбінованим:

  • Якісний аналіз (Qualitative): категоризація ризиків за шкалою (Low / Medium / High / Critical) на основі експертних оцінок. Простіший у використанні, але суб'єктивний. Зазвичай використовується на початковому етапі або коли бракує даних для кількісного аналізу
  • Кількісний аналіз (Quantitative): фінансова оцінка ризику у грошовому вираженні. Базується на формулі:
    • SLE (Single Loss Expectancy) — очікуваний збиток від одного інциденту. Наприклад, якщо data breach торкнеться 10,000 записів, а вартість одного запису — $150, то SLE = $1,500,000
    • ARO (Annualized Rate of Occurrence) — очікувана кількість інцидентів на рік. Наприклад, якщо подібний breach трапляється раз на 5 років, ARO = 0.2
    • ALE (Annualized Loss Expectancy) — річний очікуваний збиток: ALE = SLE × ARO. У нашому прикладі: ALE = $1,500,000 × 0.2 = $300,000/рік. Ця цифра допомагає обґрунтувати бюджет на безпеку — якщо контроль, що запобігає цьому ризику, коштує менше $300K/рік, він окупається

3. Оцінка ризиків (Risk Evaluation)

На цьому етапі результати аналізу порівнюються з бізнес-контекстом організації для прийняття рішень:

  • Порівняння з risk appetite: risk appetite (апетит до ризику) — це рівень ризику, який організація готова прийняти, визначений керівництвом. Наприклад, стартап може мати високий risk appetite (швидкість важливіша за безпеку), а банк — дуже низький
  • Пріоритезація ризиків: ранжування за комбінацією ймовірності та впливу. Risk matrix (матриця ризиків) — візуальний інструмент, де по осях розміщуються likelihood та impact, а кожен ризик наноситься як точка. Ризики у правому верхньому куті (high likelihood + high impact) потребують негайної уваги
  • Визначення прийнятного рівня: risk tolerance (толерантність до ризику) — конкретна межа, нижче якої ризик вважається прийнятним. Наприклад: "ALE менше $50K — прийнятний, $50K-$500K — потребує мітигації, понад $500K — неприйнятний"

4. Обробка ризиків (Risk Treatment)

Для кожного ризику, що перевищує прийнятний рівень, обирається стратегія обробки. Вибір стратегії залежить від співвідношення вартості контролю та потенційних збитків:

  • Avoid (Уникнення): повна відмова від діяльності, що генерує ризик. Приклад у хмарі: не зберігати критичні дані (military secrets, ядерні коди) у публічній хмарі взагалі
  • Mitigate (Зменшення): впровадити контролі для зниження ймовірності або впливу. Найпоширеніша стратегія. Приклад: увімкнути шифрування для захисту від data breach, впровадити MFA для захисту від credential theft
  • Transfer (Передача): передати фінансові наслідки ризику третій стороні. Приклад: cyber insurance (кіберстрахування) — страхова компанія покриє витрати на incident response та юридичні витрати. SLA з компенсаціями — інший варіант, хоча суми зазвичай мізерні
  • Accept (Прийняття): свідоме, задокументоване рішення прийняти ризик без додаткових заходів. Валідно для ризиків з ALE нижче вартості мітигації. Ключове: це має бути формальне рішення, підписане risk owner (відповідальною особою), а не просто ігнорування

5. Моніторинг та перегляд (Monitor & Review)

Управління ризиками — це не одноразова активність, а безперервний процес. Хмарне середовище змінюється постійно, і ризики змінюються разом з ним:

  • Регулярний перегляд ризиків: щоквартально або при значних змінах (нові сервіси, нові регуляції, зміна провайдера, інцидент безпеки). Risk register (реєстр ризиків) — живий документ, що постійно оновлюється
  • Оновлення при змінах середовища: додавання нового хмарного сервісу, міграція workload, зміна compliance-вимог, вихід на новий ринок — все це trigger для переоцінки ризиків
  • KRI (Key Risk Indicators) моніторинг: автоматизовані метрики, що сигналізують про зміну рівня ризику. Наприклад: кількість unpatched critical vulnerabilities, відсоток ресурсів без шифрування, кількість failed login attempts, кількість public-facing storage buckets. Перевищення порогу KRI — trigger для дій

Фреймворки оцінки ризиків

NIST RMF (Risk Management Framework)

NIST RMF, визначений у документі NIST SP 800-37 Rev. 2, — це структурований 7-кроковий процес інтеграції безпеки та управління ризиками у життєвий цикл інформаційних систем. RMF є обов'язковим для всіх федеральних інформаційних систем США та широко використовується у приватному секторі:

  1. Prepare (Підготовка): підготовка організації до управління ризиками — визначення ролей (risk executive, system owner, authorizing official), встановлення risk management strategy, ідентифікація критичних бізнес-процесів
  2. Categorize (Категоризація): категоризація інформаційної системи за рівнем впливу (Low, Moderate, High) згідно з FIPS 199 (Federal Information Processing Standard). Рівень визначається за трьома критеріями: confidentiality, integrity, availability. Цей крок визначає, скільки контролів знадобиться
  3. Select (Вибір): вибір базового набору контролів з каталогу NIST SP 800-53, відповідного до категорії системи. Базові набори (baselines) визначені у SP 800-53B. Організація може додати або виключити контролі на основі специфічних ризиків (tailoring)
  4. Implement (Впровадження): практична імплементація обраних контролів та документування того, як саме кожен контроль впроваджений у конкретній системі
  5. Assess (Оцінка): незалежна оцінка того, чи контролі впроваджені коректно, працюють як очікується та дають бажаний результат. Проводиться assessor'ом (оцінювачем)
  6. Authorize (Авторизація): AO (Authorizing Official) — уповноважена особа приймає рішення на основі assessment results: чи прийнятний залишковий ризик (residual risk) для початку або продовження експлуатації системи. Результат — ATO (Authorization to Operate) або відмова
  7. Monitor (Моніторинг): безперервний моніторинг контролів, середовища та ризиків. Включає ongoing assessments, configuration management, incident response та звітування AO про зміни в ризиковому профілі

FAIR (Factor Analysis of Information Risk)

FAIR — єдиний міжнародний стандарт (OpenFAIR, прийнятий The Open Group) для кількісної оцінки інформаційних ризиків у грошовому вираженні. На відміну від якісних методів (Low/Medium/High), FAIR дає конкретні цифри: "цей ризик коштуватиме нам від $200K до $1.5M на рік з 90% ймовірністю". Це дозволяє порівнювати ризики між собою та обґрунтовувати бюджет на безпеку перед керівництвом мовою фінансів.

  • Розкладає ризик на компоненти: TEF (Threat Event Frequency — частота загрозливих подій), Vulnerability (ймовірність того, що загроза переросте в збиток), Loss Magnitude (масштаб збитків — первинних та вторинних)
  • Monte Carlo симуляції: замість точкових оцінок використовує діапазони ("від X до Y") та проганяє тисячі сценаріїв, щоб отримати розподіл ймовірностей. Це дає більш реалістичну картину, ніж однозначна оцінка
  • Фінансова оцінка: кінцевий результат — ALE (Annualized Loss Expectancy) — річний очікуваний збиток з confidence interval (наприклад, "з 90% ймовірністю від $330K до $3.3M")
  • Інструменти: RiskLens (комерційна платформа на базі FAIR), FAIR-U (безкоштовний інструмент для навчання)

CSA Cloud Risk Assessment

CSA пропонує екосистему інструментів, спеціально розроблених для оцінки ризиків у хмарних середовищах. Ці інструменти взаємопов'язані та доповнюють один одного:

  • CAIQ (Consensus Assessments Initiative Questionnaire): стандартизована анкета з понад 260 питань типу "так/ні" для оцінки безпеки хмарного провайдера. Питання охоплюють усі 17 доменів CCM. Використовується при виборі провайдера: ви надсилаєте CAIQ потенційному провайдеру, він заповнює, ви порівнюєте відповіді кількох провайдерів. Також є частиною STAR Level 1 (Self-Assessment)
  • CCM (Cloud Controls Matrix): матриця контролів для mapping — дозволяє відобразити ваші compliance-вимоги (ISO, NIST, PCI DSS) на конкретні хмарні контролі та перевірити, чи провайдер їх впровадив
  • Top Threats: регулярно оновлюваний звіт про топ загроз хмарних обчислень (детально розглянутий нижче), що допомагає пріоритезувати ризики на основі актуальних даних від індустрії
  • Cloud Controls Matrix + CAIQ mapping: кожне питання CAIQ прив'язане до конкретного контролю CCM, що дозволяє системно оцінити покриття: якщо провайдер відповів "ні" на питання, пов'язане з критичним для вас контролем — це червоний прапорець

CSA Top Threats 2024

Cloud Security Alliance регулярно публікує звіт "Top Threats to Cloud Computing" — рейтинг найбільш критичних загроз безпеці хмарних обчислень, складений на основі опитування експертів з усього світу. Цей документ є важливим орієнтиром для організацій при плануванні стратегії хмарної безпеки:

  1. Misconfiguration and Inadequate Change Control (Помилки конфігурації та недостатній контроль змін) — найпоширеніша загроза. Відкриті S3 buckets, надмірні IAM permissions, відсутність шифрування — все це наслідки помилок конфігурації. За даними Gartner, до 2025 року 99% збоїв безпеки хмари будуть спричинені помилками клієнтів
  2. IAM and Access Control Failures (Збої управління ідентифікацією та доступом) — слабкі паролі, відсутність MFA, надмірні привілеї, невідкликані облікові записи звільнених працівників, спільні (shared) облікові записи
  3. Insecure Interfaces and APIs (Небезпечні інтерфейси та API) — API є основним способом взаємодії з хмарними сервісами, тому незахищені API (відсутність автентифікації, rate limiting, input validation) — це прямий вектор атаки
  4. Limited Cloud Security Architecture and Strategy (Обмежена архітектура та стратегія хмарної безпеки) — відсутність цілісної стратегії безпеки, "lift and shift" міграція без адаптації безпекових практик до хмарного середовища
  5. Insecure Software Development (Небезпечна розробка ПЗ) — вразливості у коді, відсутність SAST/DAST сканування, insecure CI/CD pipelines, hardcoded secrets у коді
  6. Unsecured Third-Party Resources (Незахищені ресурси третіх сторін) — вразливості у бібліотеках, плагінах, marketplace images, а також компрометація через supply chain attacks
  7. System Vulnerabilities (Системні вразливості) — непропатчені операційні системи, застаріле ПЗ, zero-day вразливості у хмарній інфраструктурі
  8. Accidental Data Disclosure (Випадкове розкриття даних) — ненавмисна публікація конфіденційних даних через misconfigured storage, помилки в access policies, logging чутливої інформації
  9. Serverless and Container Workload Risks (Ризики serverless та контейнерних навантажень) — вразливості у container images, misconfigured orchestration (Kubernetes), insecure serverless functions з надмірними правами
  10. Organized Crime and APT (Організована злочинність та APT) — APT (Advanced Persistent Threat — складна цілеспрямована загроза) — це тривалі, добре фінансовані атаки, зазвичай державного рівня, спрямовані на конкретні організації. Хмарна інфраструктура стає все привабливішою ціллю через концентрацію даних
  11. Cloud Storage Data Exfiltration (Ексфільтрація даних з хмарного сховища) — несанкціоноване вилучення (крадіжка) даних з хмарних сховищ, часто через скомпрометовані облікові записи або insecure configurations

2.3 SLA та контрактні аспекти безпеки

Що таке SLA?

SLA (Service Level Agreement) — угода про рівень обслуговування, юридично обов'язковий документ (або розділ контракту) між хмарним провайдером та клієнтом, що формалізує очікування щодо якості сервісу. SLA — це не просто маркетинговий документ; це юридичний інструмент, який визначає права та обов'язки обох сторін. У більш широкій ієрархії SLA пов'язаний з SLO (Service Level Objective) — конкретними цільовими показниками (наприклад, "99.99% availability"), та SLI (Service Level Indicator) — фактичними вимірюваними метриками (наприклад, "реальний uptime за останній місяць"). SLA визначає:

  • Метрики якості сервісу — що саме вимірюється (availability, latency, throughput, error rate)
  • Цільові показники (SLO — Service Level Objectives) — конкретні числові значення, яких провайдер зобов'язується дотримуватися
  • Компенсації (service credits) за невиконання — що отримає клієнт, якщо провайдер не дотримається зобов'язань. Зазвичай це кредити на рахунок (10-100% від місячної оплати), а не грошове відшкодування
  • Exclusions (виключення) — що НЕ покривається SLA: планове обслуговування (maintenance windows), force majeure, дії клієнта, проблеми з третіми сторонами
  • Процедури вимірювання та звітування — як вимірюється compliance, хто вимірює, як подавати claim (заявку на компенсацію)

Ключові метрики SLA

Availability (Доступність)

Найпоширеніша метрика SLA, зазвичай виражається як відсоток часу, протягом якого сервіс доступний та функціонує коректно. У індустрії прийнято вимірювати availability у "дев'ятках" (nines) — кількості цифр 9 у відсотку доступності. Кожна додаткова "дев'ятка" означає десятикратне зменшення допустимого downtime, але й значне зростання вартості інфраструктури (потрібна більша надлишковість — redundancy). Формула розрахунку: Availability = (Total time - Downtime) / Total time × 100%.

Інші важливі метрики

  • RTO (Recovery Time Objective) — цільовий час відновлення: максимально допустимий час, протягом якого система може бути недоступна після інциденту чи збою. Наприклад, RTO = 1 година означає, що після падіння система має бути відновлена протягом години. Чим менший RTO, тим дорожче рішення (потрібна active-active інфраструктура, hot standby)
  • RPO (Recovery Point Objective) — допустима точка відновлення: максимальний обсяг даних (виражений у часі), який організація готова втратити. RPO = 15 хвилин означає, що резервне копіювання має відбуватися мінімум кожні 15 хвилин. RPO = 0 означає zero data loss і вимагає синхронної реплікації
  • MTTR (Mean Time To Repair / Recovery) — середній час відновлення: середня тривалість від виявлення збою до повного відновлення роботи. Ключова метрика для оцінки операційної ефективності команди та провайдера
  • MTBF (Mean Time Between Failures) — середній час між збоями: середній час безвідмовної роботи системи. Чим вищий MTBF, тим надійніша система. Наприклад, MTBF = 8760 годин означає, що в середньому система працює рік без збоїв
  • Response Time (Latency) — час відповіді: час від отримання запиту до відправки відповіді. Зазвичай визначається як percentile: p50 (медіана), p95, p99. Наприклад, "p99 latency < 200ms" означає, що 99% запитів отримують відповідь за менше ніж 200 мілісекунд
  • Throughput — пропускна здатність: кількість операцій (requests, transactions), яку система обробляє за одиницю часу (зазвичай на секунду — RPS/TPS)
  • Error Rate — частота помилок: відсоток запитів, що завершуються помилкою (5xx статус-коди). SLA може визначати, наприклад, "error rate < 0.1%"

SLA великих провайдерів

Усі три hyperscalers (AWS, Azure, GCP) публікують свої SLA відкрито. Важливо розуміти: availability SLA зазвичай вимагає правильної архітектури з боку клієнта — наприклад, 99.99% для VM доступний лише при розгортанні у кількох AZ (Availability Zones — зони доступності) — фізично розділених дата-центрах у межах одного регіону. Одна VM в одній AZ матиме значно нижчий SLA. Також зверніть увагу на різницю між availability (доступність) та durability (довговічність) — для storage сервісів durability означає ймовірність того, що дані не будуть втрачені.

AWS (Amazon Web Services)

  • EC2 (Elastic Compute Cloud — віртуальні сервери): 99.99% для Multi-AZ deployment (розгортання у кількох зонах доступності), 99.5% для single instance
  • S3 (Simple Storage Service — об'єктне сховище): 99.9% availability (доступність), 99.999999999% (11 nines) durability (довговічність) — це означає, що з 10 мільйонів об'єктів ви можете втратити лише один за 10,000 років
  • RDS (Relational Database Service — керовані бази даних) Multi-AZ: 99.95% — нижчий за EC2, оскільки бази даних мають складніші механізми failover
  • Компенсація: service credits — 10% при availability нижче SLA, 25% при <99%, 100% при <95%. Клієнт повинен самостійно подати claim (заявку) протягом 30 днів

Azure (Microsoft)

  • Virtual Machines: 99.9% (single VM з Premium SSD), 99.95% (Availability Set — група VM у різних fault/update domains), 99.99% (Availability Zones — різні фізичні дата-центри)
  • Azure SQL Database: 99.99% — один з найвищих SLA серед керованих баз даних
  • Компенсація: service credits — 10% при availability 99-99.99%, 25% при <99%, до 100% при <95%

Google Cloud Platform (GCP)

  • Compute Engine (віртуальні машини): 99.99% для multi-zone instances, 99.5% для single instance
  • Cloud Storage: 99.95% (Standard — для часто доступних даних), 99.9% (Nearline — для даних з доступом раз на місяць), 99.0% (Coldline — для архівних даних)
  • Компенсація: financial credits аналогічно до AWS та Azure, з прогресивною шкалою залежно від рівня порушення SLA

Безпекові аспекти в контрактах

SLA — це лише частина контракту з хмарним провайдером. Не менш важливі безпекові положення, які визначають, як провайдер захищатиме ваші дані, як реагуватиме на інциденти, і які права ви маєте як клієнт.

Data Protection

Захист даних — це серце будь-якого хмарного контракту. Переконайтеся, що наступні аспекти чітко прописані:

  • Шифрування at rest: яким алгоритмом (AES-256 — стандарт), чи є опція customer-managed keys (CMK), де зберігаються ключі
  • Шифрування in transit: TLS 1.2 мінімум, бажано TLS 1.3. Зверніть увагу на внутрішній трафік провайдера — він теж має бути зашифрований
  • Data residency: конкретні регіони зберігання. Не "Європа", а "Frankfurt (eu-central-1)". Це важливо для GDPR та інших регуляцій
  • Data deletion: як провайдер видаляє дані при завершенні контракту? Чи є криптографічне стирання (crypto-shredding)? Чи видаляються backup'и? Протягом якого часу?

Access and Authentication

Хто і як може отримати доступ до ваших ресурсів — питання, яке має бути чітко регламентоване:

  • Identity management: підтримка SSO (SAML 2.0, OIDC), інтеграція з вашим корпоративним IdP (Active Directory, Okta)
  • MFA вимоги: чи підтримує провайдер hardware tokens (FIDO2/WebAuthn)? Чи є MFA обов'язковим для адміністративних операцій?
  • Audit logging: які дії логуються? Яка retention period? Чи можна експортувати логи у ваш SIEM?
  • Доступ персоналу провайдера: критичне питання — за яких умов інженери провайдера можуть отримати доступ до ваших даних? Чи логується це? Чи потрібна ваша згода?

Incident Response

Інциденти трапляються навіть у найкращих провайдерів. Контракт має чітко визначати, що відбувається, коли щось йде не так:

  • Notification timelines: протягом якого часу провайдер зобов'язаний повідомити вас про інцидент? GDPR вимагає 72 години для notification регулятору, але ви хочете знати раніше. Найкращі провайдери сповіщають протягом 24 годин.
  • Forensics support: чи надає провайдер дані для розслідування? Логи, snapshots, network captures — все це може бути критичним для incident response.
  • Root Cause Analysis (RCA): чи зобов'язаний провайдер надати детальний звіт про причини інциденту?

Compliance

Переконайтеся, що контракт підтримує ваші compliance-зобов'язання:

  • Сертифікації провайдера: які сертифікати має провайдер (SOC 2, ISO 27001, CSA STAR)? Чи зобов'язаний він підтримувати їх актуальними?
  • Audit rights: ваше право на аудит — прямий або через третю сторону. Деякі провайдери надають SOC 2 звіт замість прямого аудиту.
  • Compliance reports: доступ до звітів про відповідність, penetration test results, vulnerability scan reports

Data Processing Agreement (DPA)

Якщо ви обробляєте персональні дані громадян ЄС (а це майже завжди так), вам потрібен DPA — юридично обов'язковий додаток до контракту, що регулює обробку персональних даних.

Ключові елементи DPA:

  • Ролі: ви — Data Controller (визначаєте цілі обробки), провайдер — Data Processor (обробляє за вашими інструкціями). Це важливо, бо controller несе основну відповідальність перед суб'єктами даних.
  • Цілі обробки: для чого саме провайдер обробляє дані? Лише для надання сервісу чи також для своїх цілей (аналітика, покращення моделей AI)?
  • Sub-processors: хто ще обробляє ваші дані? Провайдер зобов'язаний надати список субпідрядників та повідомляти про зміни. Ви маєте право заперечити.
  • Cross-border transfers: після Schrems II потрібні Standard Contractual Clauses (SCCs) для передачі даних за межі ЄС
  • Data subject rights: як провайдер допомагає вам виконувати права суб'єктів (доступ, видалення, портативність)?

Важливі контрактні положення

Окрім безпекових аспектів, є кілька контрактних положень, які критично впливають на вашу позицію:

Termination clauses (Умови розірвання)

Що станеться, якщо ви вирішите піти від провайдера? Або якщо провайдер припинить роботу? Ці питання мають бути вирішені до підписання контракту, а не після.

  • Data portability: право отримати всі свої дані у стандартному форматі (JSON, CSV, SQL dump). Скільки часу це займе? Чи є додаткові витрати?
  • Transition period: після завершення контракту провайдер має надати перехідний період (зазвичай 30-90 днів) для міграції
  • Data deletion: підтвердження повного видалення ваших даних, включаючи backup'и, протягом визначеного терміну (зазвичай 30-60 днів)

Liability limitations (Обмеження відповідальності)

Це, мабуть, найбільш контроверсійна частина будь-якого хмарного контракту. Провайдери намагаються максимально обмежити свою відповідальність.

  • Caps on liability: типово — 12 місяців оплати. Тобто якщо ви платите $10K/місяць, максимальна компенсація — $120K, навіть якщо збитки від breach — мільйони
  • Exclusions: більшість провайдерів виключають непряму шкоду (втрачений прибуток, репутаційні втрати). Це означає, що реальна компенсація може бути мізерною
  • Indemnification: хто компенсує збитки третім сторонам? Наприклад, якщо витік даних призведе до позовів від ваших клієнтів

Change management (Управління змінами)

Хмарні провайдери постійно змінюють свої сервіси. Контракт має визначати правила гри:

  • Notification: за скільки днів провайдер зобов'язаний повідомити про зміни? 30 днів — мінімум, 90 — бажано
  • Deprecation policy: що трапляється зі старими API та сервісами? Чи є migration path?
  • Breaking changes: чи має клієнт право відмовитися від міграції без штрафів, якщо зміни несумісні з його системами?

2.4 Аудит та Compliance

Роль Cloud Auditor

Згадаймо еталонну архітектуру NIST з першої лекції — один з п'яти акторів хмарної екосистеми це Cloud Auditor. Аудитор — це незалежна третя сторона, яка об'єктивно оцінює безпеку, відповідність стандартам та ефективність контролів хмарного провайдера або організації-клієнта. Його роль:

  • Незалежна оцінка хмарних сервісів — аудитор не залежить ні від провайдера, ні від клієнта, що забезпечує об'єктивність
  • Перевірка відповідності (compliance verification) — чи відповідає провайдер/клієнт заявленим стандартам, політикам, регуляторним вимогам
  • Аудит трьох вимірів: безпека (security audit), приватність (privacy audit), продуктивність (performance audit)
  • Формування довіри — результати аудиту (сертифікати, звіти) допомагають клієнтам приймати обґрунтовані рішення про вибір провайдера

Типи аудитів

В залежності від того, хто проводить аудит та як часто, виділяють три основні типи:

Internal Audit (Внутрішній аудит)

Проводиться власною командою організації (або внутрішнім відділом аудиту). Це "перша лінія перевірки":

  • Регулярна перевірка ефективності впроваджених контролів — чи працюють вони як очікується
  • Виявлення gaps (прогалин) та невідповідностей до того, як їх знайдуть зовнішні аудитори
  • Підготовка до зовнішніх аудитів — internal audit допомагає "навести лад" заздалегідь
  • Обов'язковий елемент ISO 27001 (Clause 9.2) — без внутрішнього аудиту неможливо отримати або підтримувати сертифікацію

External Audit (Зовнішній аудит)

Проводиться незалежними сертифікованими аудиторськими фірмами. Результат — офіційний звіт або сертифікат:

  • Аудиторські фірми: Big Four (Deloitte, PwC, EY, KPMG), спеціалізовані фірми (Coalfire, Schellman, A-LIGN)
  • Типи зовнішніх аудитів: сертифікація ISO 27001 (акредитовані органи сертифікації), SOC 2 Type II (ліцензовані CPA фірми), PCI DSS QSA assessment
  • Регуляторні перевірки: перевірки з боку регуляторів (наприклад, DPA — Data Protection Authority для GDPR compliance)
  • Результат: формальний звіт з findings (знахідками), recommendations та opinion (висновком аудитора)

Continuous Audit/Monitoring (Безперервний аудит)

Традиційний підхід "аудит раз на рік" вже недостатній для динамічних хмарних середовищ, де конфігурації змінюються щохвилини. Безперервний аудит використовує автоматизацію для постійної перевірки:

  • Автоматизована перевірка контролів — скрипти та інструменти перевіряють конфігурації в реальному часі (наприклад, "чи всі S3 buckets зашифровані?", "чи увімкнений MFA для всіх IAM users?")
  • Real-time compliance monitoring — dashboards, що показують поточний стан compliance у відсотках, з alerts при відхиленнях
  • Drift detection — виявлення відхилень від baseline (базової) конфігурації — коли хтось змінює налаштування, що порушує compliance
  • Інструменти: CSPM (Cloud Security Posture Management — управління безпековою позою хмари), CNAPP (Cloud-Native Application Protection Platform — платформа захисту хмарних додатків), AWS Config, Azure Policy, GCP Security Command Center

GDPR — Детальний огляд

GDPR (General Data Protection Regulation) — Загальний регламент про захист даних, прийнятий Європейським Союзом у 2016 році та чинний з 25 травня 2018 року. GDPR є найвпливовішим регуляторним актом у сфері приватності у світі та має екстратериторіальну дію — він застосовується не тільки до організацій в ЄС, а й до будь-якої організації у світі, що обробляє персональні дані резидентів ЄС. Це означає, що українська компанія, яка має клієнтів з ЄС, також повинна відповідати вимогам GDPR.

Ключові ролі в GDPR: Data Controller (визначає цілі та засоби обробки — зазвичай організація-клієнт), Data Processor (обробляє дані за інструкціями Controller'а — зазвичай хмарний провайдер), Data Subject (фізична особа, чиї дані обробляються), DPO (Data Protection Officer) — відповідальна особа за захист даних, яку повинні призначити певні організації, та DPA (Data Protection Authority) — наглядовий орган країни-члена ЄС, що контролює дотримання GDPR.

Максимальні штрафи за порушення GDPR: до €20 мільйонів або 4% від глобального річного обороту (що більше) за найсерйозніші порушення (Art. 83(5)), та до €10 мільйонів або 2% обороту за менш серйозні (Art. 83(4)).

Data Residency та International Transfers

Data residency (резидентність даних) — це питання про фізичне місцезнаходження даних. У хмарі це критично важливо, оскільки GDPR (Art. 44-49) обмежує передачу персональних даних за межі EEA (European Economic Area — Європейський економічний простір), який включає всі країни ЄС плюс Ісландію, Ліхтенштейн та Норвегію.

Schrems II — рішення Суду ЄС від 16 липня 2020 року (справа C-311/18), яке визнало недійсним Privacy Shield — угоду між ЄС та США, що дозволяла вільну передачу даних. Рішення названо на честь австрійського активіста Макса Шремса (Max Schrems), який оскаржив передачу його даних Facebook до США. Суд вирішив, що американське законодавство про спостереження (зокрема FISA Section 702 та Executive Order 12333) не забезпечує адекватного рівня захисту персональних даних громадян ЄС. Це рішення вплинуло на тисячі організацій, що передавали дані до США.

Після Schrems II передача даних з ЄС до третіх країн потребує одного з наступних механізмів:

  • Adequacy Decision (Рішення про адекватність): Європейська комісія визнає, що третя країна забезпечує адекватний рівень захисту даних. Станом на 2024 рік такі рішення є для: UK, Японії, Канади, Південної Кореї, Аргентини, Нової Зеландії, Ізраїлю та деяких інших. У 2023 році було прийнято EU-US Data Privacy Framework (наступник Privacy Shield), що знову дозволяє передачу до сертифікованих американських компаній
  • SCCs (Standard Contractual Clauses — Стандартні контрактні положення): затверджені Європейською комісією типові контрактні умови, які зобов'язують отримувача даних забезпечити адекватний захист. Це найпоширеніший механізм для передачі даних до третіх країн. Нові SCCs (2021) вимагають проведення TIA перед передачею
  • BCRs (Binding Corporate Rules — Обов'язкові корпоративні правила): внутрішні правила транснаціональної корпорації для передачі даних між її підрозділами в різних країнах. Вимагають схвалення DPA і підходять лише для великих корпорацій
  • TIA (Transfer Impact Assessment — Оцінка впливу передачі): обов'язкова оцінка ризиків передачі даних до третьої країни. Організація має оцінити законодавство країни-отримувача (чи може уряд отримати доступ до даних?) та визначити додаткові технічні заходи (наприклад, шифрування з ключами, недоступними для отримувача)

Регіони хмарних провайдерів в ЄС:

  • AWS: Ireland (eu-west-1), Frankfurt (eu-central-1), Paris, Stockholm, Milan, Spain
  • Azure: West Europe (Netherlands), North Europe (Ireland), France, Germany, Switzerland
  • GCP: europe-west1 (Belgium), europe-west2 (London), europe-west3 (Frankfurt)

Compliance Frameworks

Окрім регуляторних вимог (GDPR, HIPAA), існують добровільні compliance frameworks, які організації приймають для підтвердження своєї безпеки перед клієнтами та партнерами:

SOC 2 в контексті аудиту

SOC 2 (детально описаний у розділі 2.1) є одним з найпопулярніших compliance frameworks для хмарних компаній. У контексті аудиту важливо розуміти різницю між типами:

  • Type I — оцінює дизайн (design) контролів на конкретну дату. Відповідає на питання: "чи правильно спроєктовані контролі?". Швидший та дешевший, але менш інформативний
  • Type II — оцінює операційну ефективність (operating effectiveness) контролів за визначений період (6-12 місяців). Відповідає на питання: "чи контролі дійсно працювали протягом тривалого часу?". Значно ціннішій, оскільки доводить стабільність безпеки
  • Trust Service Criteria: Security (обов'язковий), Availability, Processing Integrity, Confidentiality, Privacy — організація обирає, за якими критеріями проходити аудит
  • SOC 3 — публічна версія SOC 2 (без деталей контролів), яку можна розміщувати на сайті. SOC 2 — конфіденційний документ, доступний лише під NDA

FedRAMP (Federal Risk and Authorization Management Program)

FedRAMP — програма стандартизованої оцінки безпеки, авторизації та безперервного моніторингу хмарних сервісів, що використовуються федеральним урядом США. Створена у 2011 році з метою уникнути ситуації, коли кожне з сотень федеральних агентств окремо оцінює одного й того ж хмарного провайдера (принцип "do once, use many times" — "оціни один раз, використовуй багатьма").

  • Базується на NIST SP 800-53 — контролі безпеки обираються з каталогу 800-53, але з додатковими вимогами, специфічними для хмари
  • Три рівні впливу (impact levels):
    • Low — для систем з мінімальним впливом (public-facing інформація); ~125 контролів
    • Moderate — для більшості федеральних систем; ~325 контролів. Це найпоширеніший рівень
    • High — для критичних систем (правоохоронні органи, охорона здоров'я, фінансові дані); ~421 контроль
  • Обов'язковий для будь-якого хмарного провайдера, що хоче надавати послуги федеральному уряду США. Авторизація FedRAMP визнається всіма агентствами
  • 3PAO (Third Party Assessment Organization) — незалежна організація, акредитована для проведення оцінки. Аналог ISO certification body
  • Процес авторизації: документація → assessment (3PAO) → authorization (JAB або agency) → continuous monitoring. Зазвичай займає 6-18 місяців та коштує $500K-$3M

Процес аудиту хмарного провайдера

Right to Audit Clause

Right to Audit — контрактне положення, що надає клієнту право перевіряти безпеку хмарного провайдера. Це один з найважливіших та водночас найскладніших пунктів переговорів. Провайдери не хочуть, щоб кожен з тисяч клієнтів проводив свій аудит (це непрактично), а клієнти хочуть впевнитися, що їхні дані в безпеці. Існує кілька компромісних підходів:

  • Прямий аудит (Direct audit): клієнт або його аудитори безпосередньо перевіряють провайдера. Рідко доступний для малих клієнтів, але великі enterprise-клієнти можуть домовитися про це. Зазвичай обмежений за scope та частотою (раз на рік)
  • Аудит третьою стороною (Third-party audit): незалежний аудитор, обраний спільно або з акредитованого списку, проводить перевірку. Більш прийнятний для провайдерів, оскільки аудитор має NDA та професійні стандарти
  • Доступ до звітів (Report-based assurance): найпоширеніший варіант для hyperscalers (AWS, Azure, GCP) — замість прямого аудиту провайдер надає клієнтам SOC 2 Type II звіти, ISO 27001 сертифікати, CSA STAR реєстрацію, penetration test summaries. Ці звіти доступні через портали (наприклад, AWS Artifact)
  • Contractual obligations: GDPR Art. 28(3)(h) прямо вимагає, щоб Data Processor (провайдер) надавав Controller'у (клієнту) можливість аудиту. Це підсилює позицію клієнта на переговорах

Інструменти Compliance Automation

Ручна перевірка compliance у хмарних середовищах з тисячами ресурсів — це непрактично та ненадійно. Тому сучасні організації використовують автоматизацію для безперервного моніторингу відповідності. Основні підходи:

CSPM (Cloud Security Posture Management)

CSPM — клас інструментів для автоматичного моніторингу та оцінки безпекової конфігурації хмарної інфраструктури. CSPM сканує ваше хмарне середовище, порівнює конфігурації з best practices та compliance frameworks, і виявляє відхилення (misconfigurations).

  • Автоматична перевірка конфігурацій: сканування сотень параметрів — чи зашифровані бази даних, чи закриті публічні порти, чи увімкнений logging
  • Mapping до compliance frameworks: автоматичне відображення стану ресурсів на вимоги ISO 27001, NIST, PCI DSS, CIS Benchmarks
  • Continuous monitoring: перевірки відбуваються не раз на рік, а постійно — кожну хвилину чи годину
  • Remediation (виправлення): деякі CSPM можуть автоматично виправляти виявлені проблеми (auto-remediation) або створювати тикети для команди
  • Приклади: AWS Security Hub, Microsoft Defender for Cloud (раніше Azure Defender), GCP Security Command Center, Prisma Cloud (Palo Alto), Wiz, Orca Security

Policy-as-Code (Політика як код)

Policy-as-Code — підхід, при якому безпекові та compliance-політики визначаються у вигляді коду, а не документів. Це дозволяє версіонувати політики (git), автоматично перевіряти їх у CI/CD pipeline (Continuous Integration / Continuous Delivery — конвеєр безперервної інтеграції та доставки), та гарантувати, що інфраструктура, яка не відповідає політикам, не буде розгорнута.

  • Shift-left security: виявлення проблем на етапі написання коду, а не після розгортання в production
  • Автоматична перевірка в CI/CD pipeline — pull request не може бути merged, якщо код порушує політику
  • Інструменти:
    • OPA (Open Policy Agent) — універсальний policy engine, використовує мову Rego для визначення політик. Інтегрується з Kubernetes, Terraform, API gateways
    • Checkov — open-source сканер для IaC (Infrastructure as Code — інфраструктура як код), підтримує Terraform, CloudFormation, Kubernetes, Dockerfile
    • Terraform Sentinel — policy-as-code framework від HashiCorp для Terraform Enterprise/Cloud
    • AWS CloudFormation Guard — перевірка CloudFormation templates на відповідність правилам

Compliance Dashboards (Панелі відповідності)

Кожен великий хмарний провайдер надає вбудовані інструменти для візуалізації стану compliance:

  • AWS Audit Manager — автоматизує збір доказів (evidence) для аудитів, підтримує frameworks: SOC 2, PCI DSS, GDPR, HIPAA, CIS Benchmarks
  • Azure Compliance Manager — assessment-based tool, що розраховує compliance score та надає action items для покращення
  • GCP Security Command Center — централізований dashboard з findings по безпеці, compliance status, та рекомендаціями

2.5 Практичні інструменти Compliance

Теорія стандартів та фреймворків — це лише половина справи. На практиці організації потребують конкретних інструментів для автоматизації compliance-перевірок. У цьому розділі ми розглянемо нативні інструменти трьох великих хмарних провайдерів (AWS Config, Azure Policy) та open-source рішення (Checkov, OPA), які дозволяють перевіряти відповідність стандартам автоматично — як у реальному часі, так і на етапі розгортання інфраструктури.

AWS Config Rules

AWS Config — сервіс AWS для відстеження конфігурацій ресурсів та автоматичної перевірки їх відповідності заданим правилам (rules). Config записує кожну зміну конфігурації кожного ресурсу (EC2 instances, S3 buckets, Security Groups тощо) та оцінює їх проти визначених compliance rules. Якщо ресурс не відповідає правилу — він позначається як NON_COMPLIANT:

Приклад: Перевірка шифрування S3

# AWS Config Rule для перевірки шифрування S3
{
  "ConfigRuleName": "s3-bucket-server-side-encryption-enabled",
  "Description": "Checks if S3 bucket has server-side encryption enabled",
  "Source": {
    "Owner": "AWS",
    "SourceIdentifier": "S3_BUCKET_SERVER_SIDE_ENCRYPTION_ENABLED"
  },
  "Scope": {
    "ComplianceResourceTypes": ["AWS::S3::Bucket"]
  }
}

Приклад: Custom Config Rule (Lambda)

# Lambda function для custom compliance check
import boto3
import json

def lambda_handler(event, context):
    config = boto3.client('config')

    # Отримуємо конфігурацію ресурсу
    configuration_item = json.loads(event['invokingEvent'])['configurationItem']

    # Перевіряємо compliance
    if configuration_item['resourceType'] == 'AWS::EC2::SecurityGroup':
        # Перевірка на відкритий SSH (port 22) з 0.0.0.0/0
        for rule in configuration_item['configuration'].get('ipPermissions', []):
            if rule.get('fromPort') == 22:
                for ip_range in rule.get('ipRanges', []):
                    if ip_range.get('cidrIp') == '0.0.0.0/0':
                        return {
                            'compliance_type': 'NON_COMPLIANT',
                            'annotation': 'SSH open to world'
                        }

    return {'compliance_type': 'COMPLIANT'}

Azure Policy

Azure Policy — сервіс Microsoft Azure для governance (управління) та compliance. На відміну від AWS Config, який переважно моніторить та звітує, Azure Policy може активно запобігати створенню ресурсів, що порушують політику (effect: "deny"), або автоматично виправляти конфігурації (effect: "deployIfNotExists"). Політики можна групувати у initiatives (ініціативи) — набори пов'язаних політик для конкретного compliance framework:

Приклад: Deny public IP on VMs

{
  "mode": "All",
  "policyRule": {
    "if": {
      "allOf": [
        {
          "field": "type",
          "equals": "Microsoft.Network/networkInterfaces"
        },
        {
          "field": "Microsoft.Network/networkInterfaces/ipConfigurations[*].publicIpAddress.id",
          "exists": "true"
        }
      ]
    },
    "then": {
      "effect": "deny"
    }
  },
  "parameters": {}
}

Приклад: Require tags on resources

{
  "mode": "Indexed",
  "policyRule": {
    "if": {
      "field": "[concat('tags[', parameters('tagName'), ']')]",
      "exists": "false"
    },
    "then": {
      "effect": "deny"
    }
  },
  "parameters": {
    "tagName": {
      "type": "String",
      "metadata": {
        "displayName": "Tag Name",
        "description": "Name of the required tag"
      }
    }
  }
}

Custom Checkov Policy (Python)

from checkov.terraform.checks.resource.base_resource_check import BaseResourceCheck
from checkov.common.models.enums import CheckResult, CheckCategories

class S3BucketEncryption(BaseResourceCheck):
    def __init__(self):
        name = "Ensure S3 bucket has encryption enabled"
        id = "CKV_CUSTOM_1"
        supported_resources = ['aws_s3_bucket']
        categories = [CheckCategories.ENCRYPTION]
        super().__init__(name=name, id=id, categories=categories,
                        supported_resources=supported_resources)

    def scan_resource_conf(self, conf):
        if 'server_side_encryption_configuration' in conf:
            return CheckResult.PASSED
        return CheckResult.FAILED

check = S3BucketEncryption()

Open Policy Agent (OPA)

OPA (Open Policy Agent) — це open-source, cloud-native policy engine, розроблений компанією Styra та прийнятий CNCF (Cloud Native Computing Foundation) як graduated project. OPA дозволяє визначати політики у декларативній мові Rego та застосовувати їх до будь-яких систем: Kubernetes (через Gatekeeper), API gateways, Terraform, мікросервіси. Ключова ідея OPA — decoupled policy: відокремити логіку прийняття рішень (allow/deny) від бізнес-логіки додатку.

Приклад Rego Policy для Kubernetes

# deny-privileged-containers.rego
package kubernetes.admission

deny[msg] {
    input.request.kind.kind == "Pod"
    container := input.request.object.spec.containers[_]
    container.securityContext.privileged == true
    msg := sprintf("Privileged containers not allowed: %v", [container.name])
}

deny[msg] {
    input.request.kind.kind == "Pod"
    container := input.request.object.spec.containers[_]
    container.securityContext.runAsUser == 0
    msg := sprintf("Running as root not allowed: %v", [container.name])
}

Висновки

Стандартизація та управління ризиками — це фундамент для побудови безпечної хмарної інфраструктури. Без розуміння стандартів неможливо оцінити провайдера, без управління ризиками — неможливо приймати обґрунтовані рішення.

Ключові тези

  • Стандарти — спільна мова: ISO 27017/27018, NIST, CSA CCM дозволяють оцінювати та порівнювати провайдерів
  • Ризики хмари специфічні: втрата контролю, multi-tenancy, vendor lock-in потребують окремої уваги
  • SLA ≠ гарантія: читайте дрібний шрифт, розумійте компенсації та обмеження
  • Compliance — безперервний процес: не одноразова сертифікація, а постійний моніторинг
  • Автоматизація критична: CSPM, Policy-as-Code для масштабного compliance
  • Реальні кейси вчать: Capital One, Marriott, Equifax — lessons learned

Практичні рекомендації

  1. Перед вибором провайдера — перегляньте його сертифікації (SOC 2, ISO 27001)
  2. Використовуйте CAIQ для оцінки провайдерів
  3. Проведіть risk assessment перед міграцією в хмару
  4. Уважно читайте SLA — особливо exclusions та compensation
  5. Впровадьте CSPM для continuous compliance monitoring
  6. Використовуйте Policy-as-Code (Checkov, OPA) в CI/CD pipeline
  7. Документуйте data flows для GDPR compliance
  8. Регулярно переглядайте CSA Top Threats

Практичне завдання

Ваша організація планує мігрувати CRM-систему з on-premise в хмару. CRM містить персональні дані 50,000 клієнтів (імена, email, телефони, історія покупок). Організація працює з клієнтами з ЄС та України.

Завдання:

  1. Risk Assessment: Ідентифікуйте 5 ключових ризиків міграції CRM у хмару. Для кожного ризику визначте: ймовірність (Low/Medium/High), вплив (Low/Medium/High), стратегію обробки (avoid/mitigate/transfer/accept) та конкретні контролі.
  2. Compliance checklist: Складіть перелік compliance-вимог для цієї міграції. Які стандарти та регуляції застосовуються? (Підказка: GDPR, можливо ISO 27017)
  3. Порівняння провайдерів: Використовуючи CAIQ (CSA Consensus Assessments Initiative Questionnaire), порівняйте двох провайдерів для розміщення CRM. Визначте 10 найважливіших питань.
  4. SLA аналіз: Знайдіть реальний SLA одного з великих провайдерів (AWS, Azure, GCP) для керованої бази даних. Визначте: availability target, компенсації, exclusions, що НЕ покривається SLA.

Глосарій ключових термінів

Стандарти та фреймворки

  • ISMS — Information Security Management System (система управління інформаційною безпекою) — комплексний набір політик, процедур та контролів для систематичного управління інформаційною безпекою організації. Основа стандарту ISO 27001
  • NIST — National Institute of Standards and Technology (Національний інститут стандартів і технологій) — федеральна агенція США, один з найавторитетніших джерел стандартів кібербезпеки
  • NIST CSF — Cybersecurity Framework від NIST — високорівневий фреймворк з п'ятьма функціями: Identify, Protect, Detect, Respond, Recover. Версія 2.0 (2024) додала шосту функцію Govern
  • NIST SP 800-53 — Special Publication 800-53 — найповніший каталог security controls (понад 1000 контролів у 20 сімействах), обов'язковий для федеральних агенцій США
  • NIST RMF — Risk Management Framework (NIST SP 800-37) — 7-кроковий процес управління ризиками: Prepare, Categorize, Select, Implement, Assess, Authorize, Monitor
  • CSA — Cloud Security Alliance — міжнародна галузева організація, що спеціалізується на безпеці хмарних обчислень (понад 80,000 членів)
  • CCM — Cloud Controls Matrix — флагманський фреймворк CSA з 197 контролями у 17 доменах, специфічними для хмарної безпеки
  • CAIQ — Consensus Assessments Initiative Questionnaire — стандартизована анкета CSA для оцінки безпеки хмарних провайдерів
  • STAR — Security, Trust, Assurance, and Risk — програма сертифікації CSA для хмарних провайдерів (3 рівні: Self-Assessment, Third-Party Audit, Continuous Monitoring)
  • FedRAMP — Federal Risk and Authorization Management Program — програма авторизації хмарних сервісів для федерального уряду США (три рівні: Low, Moderate, High)
  • CIS Benchmarks — Center for Internet Security Benchmarks — конкретні рекомендації з налаштування операційних систем, хмарних сервісів та додатків
  • PDCA — Plan-Do-Check-Act (цикл Демінга) — методологія безперервного вдосконалення, на якій побудований ISO 27001
  • SoA — Statement of Applicability — ключовий документ ISMS, що пояснює, які контролі з Annex A обрані та чому

Галузеві стандарти та регуляції

  • PCI DSS — Payment Card Industry Data Security Standard — стандарт безпеки даних платіжних карток (12 вимог, обов'язковий для організацій, що обробляють картки)
  • PCI SSC — Payment Card Industry Security Standards Council — організація, що розробляє та підтримує PCI DSS
  • PAN — Primary Account Number — номер платіжної картки, основний елемент cardholder data
  • QSA — Qualified Security Assessor — сертифікований аудитор PCI DSS
  • HIPAA — Health Insurance Portability and Accountability Act — федеральний закон США про захист медичної інформації
  • PHI — Protected Health Information — захищувана медична інформація (будь-які дані про здоров'я, що ідентифікують особу)
  • BAA — Business Associate Agreement — обов'язковий контракт між covered entity та хмарним провайдером за HIPAA
  • SOC 2 — Service Organization Control 2 — аудиторський стандарт AICPA для сервісних організацій (Type I — на дату, Type II — за період)
  • AICPA — American Institute of Certified Public Accountants — організація, що розробила SOC 2
  • TSC — Trust Service Criteria — п'ять критеріїв SOC 2: Security, Availability, Processing Integrity, Confidentiality, Privacy
  • HITRUST CSF — Health Information Trust Alliance Common Security Framework — комплексний фреймворк для охорони здоров'я, що об'єднує вимоги HIPAA, NIST, ISO та PCI DSS з формальною сертифікацією
  • SMB — Small and Medium Business — малий та середній бізнес

GDPR та приватність

  • GDPR — General Data Protection Regulation — регламент ЄС про захист персональних даних (чинний з 2018, екстратериторіальна дія)
  • PII — Personally Identifiable Information — будь-яка інформація, що дозволяє ідентифікувати фізичну особу
  • Data Controller — організація, що визначає цілі та засоби обробки персональних даних
  • Data Processor — організація, що обробляє персональні дані за дорученням Controller'а (у хмарі — зазвичай провайдер)
  • Data Subject — фізична особа, чиї персональні дані обробляються
  • DPO — Data Protection Officer — відповідальна особа за захист даних в організації
  • DPA (регулятор) — Data Protection Authority — наглядовий орган країни-члена ЄС
  • DPA (контракт) — Data Processing Agreement — угода про обробку персональних даних (вимога GDPR Art. 28)
  • SCCs — Standard Contractual Clauses — стандартні контрактні положення ЄС для легітимної міжнародної передачі даних
  • BCRs — Binding Corporate Rules — обов'язкові корпоративні правила для міжнародної передачі даних усередині корпорації
  • TIA — Transfer Impact Assessment — обов'язкова оцінка ризиків передачі даних до третьої країни
  • EEA — European Economic Area — Європейський економічний простір (ЄС + Ісландія, Ліхтенштейн, Норвегія)
  • CCPA — California Consumer Privacy Act — закон Каліфорнії про конфіденційність споживачів (часто порівнюють з "GDPR для Каліфорнії")
  • Schrems II — рішення Суду ЄС (2020), що визнало Privacy Shield недійсним та посилило вимоги до міжнародної передачі даних
  • Crypto-shredding — метод гарантованого видалення даних шляхом знищення ключів шифрування (замість видалення самих даних)

Управління ризиками

  • FAIR — Factor Analysis of Information Risk — кількісна модель оцінки інформаційних ризиків у грошовому вираженні
  • SLE — Single Loss Expectancy — очікуваний збиток від одного інциденту
  • ARO — Annualized Rate of Occurrence — очікувана частота інцидентів на рік
  • ALE — Annualized Loss Expectancy — річний очікуваний збиток (ALE = SLE × ARO)
  • TEF — Threat Event Frequency — частота загрозливих подій (компонент FAIR)
  • LEF — Loss Event Frequency — частота подій, що призводять до збитків (компонент FAIR)
  • KRI — Key Risk Indicators — ключові індикатори ризику для моніторингу
  • APT — Advanced Persistent Threat — складна цілеспрямована загроза, тривала та добре фінансована атака

SLA та метрики

  • SLA — Service Level Agreement — угода про рівень обслуговування між провайдером і клієнтом
  • SLO — Service Level Objective — конкретний цільовий показник у рамках SLA (наприклад, 99.99% availability)
  • SLI — Service Level Indicator — фактично виміряне значення метрики
  • RTO — Recovery Time Objective — максимально допустимий час відновлення після інциденту
  • RPO — Recovery Point Objective — максимально допустимий обсяг втрати даних (у часі)
  • MTTR — Mean Time To Repair/Recovery — середній час відновлення після збою
  • MTBF — Mean Time Between Failures — середній час безвідмовної роботи
  • RPS/TPS — Requests/Transactions Per Second — кількість запитів/транзакцій за секунду

Інструменти та технології

  • CSPM — Cloud Security Posture Management — клас інструментів для автоматичного моніторингу безпекової конфігурації хмари
  • CNAPP — Cloud-Native Application Protection Platform — комплексна платформа захисту хмарних додатків
  • Policy-as-Code — підхід до визначення та автоматичної перевірки безпекових політик через код
  • OPA — Open Policy Agent — open-source policy engine від CNCF, використовує мову Rego
  • IaC — Infrastructure as Code — підхід до управління інфраструктурою через код (Terraform, CloudFormation, Pulumi)
  • CI/CD — Continuous Integration / Continuous Delivery — конвеєр безперервної інтеграції та доставки коду
  • SIEM — Security Information and Event Management — система збору, аналізу та кореляції безпекових подій
  • SAST/DAST — Static/Dynamic Application Security Testing — статичне та динамічне тестування безпеки додатків
  • SCA — Software Composition Analysis — аналіз залежностей (бібліотек) на вразливості
  • CVE — Common Vulnerabilities and Exposures — стандартизований ідентифікатор відомих вразливостей
  • SSRF — Server-Side Request Forgery — атака, при якій зловмисник змушує сервер робити запити до внутрішніх ресурсів
  • CMK — Customer-Managed Keys — ключі шифрування, якими керує клієнт (не провайдер)
  • 3PAO — Third Party Assessment Organization — незалежна організація для проведення оцінки FedRAMP
  • WAF — Web Application Firewall — міжмережевий екран для веб-додатків, що фільтрує HTTP-трафік
  • IMDS — Instance Metadata Service — сервіс метаданих EC2-інстансів AWS (v2 захищений від SSRF)
  • HSM — Hardware Security Module — апаратний модуль для безпечного зберігання криптографічних ключів
  • CSP — Content Security Policy — HTTP-заголовок, що обмежує джерела завантаження ресурсів на веб-сторінку
  • SRI — Subresource Integrity — механізм перевірки цілісності зовнішніх ресурсів (скриптів, стилів) через hash
  • AZ — Availability Zone — зона доступності, фізично відокремлений дата-центр у межах хмарного регіону
  • VPC — Virtual Private Cloud — ізольована віртуальна мережа у хмарі
  • NSG — Network Security Group — група мережевої безпеки в Azure (аналог Security Groups в AWS)
  • SDLC — Software Development Lifecycle — життєвий цикл розробки програмного забезпечення
  • MDM — Mobile Device Management — управління мобільними пристроями
  • BYOD — Bring Your Own Device — політика використання особистих пристроїв для роботи

Бізнес та юридичні терміни

  • M&A — Mergers & Acquisitions — злиття та поглинання компаній
  • Due diligence — належна перевірка — комплексна оцінка організації перед угодою (M&A, вибір провайдера)
  • OCC — Office of the Comptroller of the Currency — регулятор банківської системи США
  • HHS — Department of Health and Human Services — Міністерство охорони здоров'я та соціальних служб США
  • NDA — Non-Disclosure Agreement — угода про нерозголошення конфіденційної інформації
  • RCA — Root Cause Analysis — аналіз кореневих причин інциденту
  • ATO — Authorization to Operate — дозвіл на експлуатацію інформаційної системи (результат NIST RMF)
  • FIPS — Federal Information Processing Standards — федеральні стандарти обробки інформації США
  • SSN — Social Security Number — номер соціального страхування (найцінніший ідентифікатор у США)

Автентифікація та доступ

  • IAM — Identity and Access Management — управління ідентифікацією та доступом
  • MFA — Multi-Factor Authentication — багатофакторна автентифікація (щось що знаєш + щось що маєш + щось що є)
  • SSO — Single Sign-On — єдиний вхід, можливість автентифікуватися один раз для доступу до багатьох систем
  • SAML 2.0 — Security Assertion Markup Language — XML-протокол для обміну автентифікаційними даними між IdP та SP
  • OIDC — OpenID Connect — протокол автентифікації на основі OAuth 2.0, більш сучасний за SAML
  • IdP — Identity Provider — провайдер ідентичності (Active Directory, Okta, Azure AD)
  • RBAC — Role-Based Access Control — управління доступом на основі ролей
  • FIDO2/WebAuthn — стандарт безпарольної автентифікації з використанням hardware tokens або біометрії

Загрози та захист

  • DDoS — Distributed Denial of Service — розподілена атака на відмову в обслуговуванні (перевантаження цілі масою запитів)
  • IDS — Intrusion Detection System — система виявлення вторгнень (пасивно моніторить трафік)
  • IPS — Intrusion Prevention System — система запобігання вторгненням (активно блокує шкідливий трафік)
  • MITRE ATT&CK — структурована база знань тактик, технік та процедур (TTP) кіберзловмисників, включаючи Cloud Matrix
  • ENISA — European Union Agency for Cybersecurity — Агентство кібербезпеки ЄС
  • Magecart — група кіберзлочинців та тип атаки (digital skimming), спрямований на крадіжку платіжних даних через зловмисний JavaScript
  • DR — Disaster Recovery — відновлення після катастроф (процес повернення ІТ-систем до роботи після серйозного збою)
  • CISO — Chief Information Security Officer — головний офіцер з інформаційної безпеки організації
