Лекція 10. Захист даних та криптографія у хмарі

2 год

Вступ

Дані — це найцінніший актив більшості організацій. У хмарному середовищі захист даних набуває особливого значення, оскільки дані зберігаються на інфраструктурі третьої сторони та передаються через публічні мережі. Криптографія є фундаментальним інструментом для забезпечення конфіденційності та цілісності даних у хмарі.

У цій лекції ми розглянемо повний спектр питань захисту даних у хмарі — від класифікації та lifecycle management до advanced криптографічних технологій, таких як HSM, envelope encryption та confidential computing.

Чому захист даних у хмарі особливий?

  • Multi-tenancy: Дані різних клієнтів на спільній інфраструктурі
  • Data residency: Дані можуть зберігатися в різних юрисдикціях
  • Provider access: Технічна можливість доступу провайдера до даних
  • Network exposure: Дані передаються через публічні мережі
  • Compliance: GDPR, HIPAA, PCI DSS мають специфічні вимоги

Цілі лекції

Після цієї лекції ви зможете:

  • Класифікувати дані та визначати відповідні заходи захисту
  • Пояснити принципи encryption at-rest та in-transit
  • Описати стратегії key management у хмарі
  • Розуміти advanced криптографічні технології (HSM, envelope encryption, BYOK)
  • Впроваджувати комплексну стратегію захисту даних

10.1 Основи захисту даних у хмарі

Класифікація даних

Не всі дані однаково цінні. Класифікація допомагає визначити відповідний рівень захисту:

Типові рівні класифікації:

  • Public: Інформація для вільного поширення (маркетингові матеріали)
  • Internal: Внутрішня інформація, не для публікації
  • Confidential: Конфіденційна бізнес-інформація
  • Restricted/Secret: Найвищий рівень захисту (PII, фінансові дані)

Категорії sensitive data:

  • PII (Personally Identifiable Information): Ім'я, email, телефон, адреса
  • PHI (Protected Health Information): Медичні записи (HIPAA)
  • PCI (Payment Card Information): Номери карток, CVV
  • Credentials: Паролі, API keys, tokens
  • Intellectual Property: Вихідний код, патенти, trade secrets

Data Lifecycle

Захист даних на кожному етапі життєвого циклу:

Data Lifecycle: захист на кожному етапі
Рис. 10.1. Життєвий цикл даних: кожен етап потребує відповідних заходів захисту

1. Creation/Collection

  • Класифікація при створенні
  • Мінімізація збору (collect only what's needed)
  • Consent management для PII

2. Storage

  • Encryption at-rest
  • Access controls
  • Data residency compliance

3. Use/Processing

  • Access logging
  • Data masking для non-production
  • Encryption in-use (confidential computing)

4. Sharing/Transfer

  • Encryption in-transit
  • Secure APIs
  • Data sharing agreements

5. Archival

  • Long-term encryption
  • Key management для archived data
  • Compliance retention

6. Destruction

  • Secure deletion
  • Crypto-shredding (знищення ключів)
  • Certificate of destruction

Shared Responsibility для даних

У контексті Shared Responsibility Model:

Відповідальність Provider Customer
Physical security ✓
Infrastructure encryption ✓
Data classification ✓
Data encryption (customer data) ✓
Access management ✓
Key management Shared Shared

10.2 Криптографія: основи

Симетричне шифрування

Один ключ для шифрування та дешифрування:

Алгоритми:

  • AES (Advanced Encryption Standard): Стандарт для symmetric encryption, AES-256 рекомендований
  • ChaCha20: Швидший за AES на деяких платформах

Переваги: Швидкість, ефективність для великих обсягів даних

Недоліки: Проблема обміну ключами (key distribution)

Асиметричне шифрування

Пара ключів: public (для шифрування) та private (для дешифрування):

Алгоритми:

  • RSA: Класичний, широко підтримуваний (RSA-2048, RSA-4096)
  • ECDSA/ECDH: Elliptic Curve, менші ключі при тій самій безпеці

Переваги: Вирішує проблему key distribution

Недоліки: Повільніший за symmetric, не підходить для великих обсягів

Hybrid Encryption

Комбінація symmetric та asymmetric:

  1. Генерується random symmetric key (Data Encryption Key, DEK)
  2. Дані шифруються DEK (symmetric, швидко)
  3. DEK шифрується public key (asymmetric)
  4. Encrypted DEK зберігається разом з encrypted data

Це основа envelope encryption, який використовується в cloud KMS.

Хеш-функції

Односторонні функції для integrity verification:

  • SHA-256: Стандарт для integrity checks
  • SHA-3: Новіший стандарт
  • HMAC: Keyed hashing для authentication

Застосування: Password hashing, digital signatures, data integrity.

Digital Signatures

Підтвердження автентичності та integrity:

  1. Hash від даних
  2. Hash шифрується private key → signature
  3. Recipient дешифрує signature public key, порівнює hash

10.3 Encryption At-Rest

Що таке encryption at-rest?

Шифрування даних, що зберігаються на носіях (диски, SSD, object storage).

Захищає від:

  • Physical theft of storage media
  • Unauthorized access to storage
  • Data remnants після deletion

НЕ захищає від:

  • Compromised credentials
  • Application-level vulnerabilities
  • Insider threats з authorized access

Рівні encryption at-rest

1. Storage-level encryption

  • Encryption на рівні storage service
  • Transparent для applications
  • Приклад: AWS S3 default encryption, Azure Storage Service Encryption

2. Volume-level encryption

  • Encryption цілого disk volume
  • Приклад: AWS EBS encryption, Azure Disk Encryption

3. Database-level encryption

  • TDE (Transparent Data Encryption)
  • Column-level encryption
  • Приклад: RDS encryption, Azure SQL TDE

4. Application-level encryption

  • Encryption у application code
  • Найвищий контроль, але складніша реалізація
  • Client-side encryption

Cloud Provider Default Encryption

AWS:

  • S3: Default encryption (SSE-S3, SSE-KMS, SSE-C)
  • EBS: Optional encryption (default можна увімкнути)
  • RDS: Optional encryption at creation time

Azure:

  • Storage: SSE enabled by default (256-bit AES)
  • Managed Disks: Encryption by default
  • Azure SQL: TDE enabled by default

GCP:

  • All data encrypted by default (Google-managed keys)
  • Customer-managed encryption keys (CMEK) optional

10.4 Encryption In-Transit

Що таке encryption in-transit?

Шифрування даних під час передачі через мережу.

Захищає від:

  • Man-in-the-middle attacks
  • Network eavesdropping
  • Data interception

TLS (Transport Layer Security)

Основний протокол для encryption in-transit:

Версії:

  • TLS 1.2: Мінімально рекомендована версія
  • TLS 1.3: Найновіша, швидша, безпечніша
  • SSL, TLS 1.0, 1.1: Deprecated, не використовувати

Cipher Suites:

  • Комбінація алгоритмів для key exchange, encryption, authentication
  • Рекомендовано: ECDHE для key exchange, AES-GCM для encryption
  • Уникати: RC4, DES, MD5

Certificate Management

TLS потребує сертифікати для authentication:

Cloud certificate services:

  • AWS Certificate Manager (ACM): Free public certificates, auto-renewal
  • Azure Key Vault Certificates: Certificate lifecycle management
  • GCP Certificate Manager: Managed SSL certificates

Best practices:

  • Automate certificate renewal
  • Monitor certificate expiration
  • Use managed certificates where possible

Internal Traffic Encryption

Шифрування east-west traffic всередині cloud:

  • Service Mesh (Istio, Linkerd): mTLS між services
  • VPC encryption: AWS VPC traffic encryption
  • Database connections: TLS для database connections

10.5 Key Management

Чому key management критичний?

Encryption без proper key management — це false sense of security:

  • Компрометація ключа = компрометація всіх даних
  • Втрата ключа = втрата доступу до даних
  • Key rotation необхідний для long-term security

Cloud Key Management Services (KMS)

AWS KMS:

  • Managed service для створення та управління ключами
  • Integration з 100+ AWS services
  • Automatic key rotation (annual)
  • CloudTrail logging всіх key operations
  • Types: AWS-managed, Customer-managed, Custom key stores

Azure Key Vault:

  • Keys, secrets, certificates management
  • HSM-backed keys available
  • Soft-delete та purge protection
  • Access policies та RBAC

Google Cloud KMS:

  • Hierarchical key management
  • Automatic та manual rotation
  • HSM та software protection levels
  • External Key Manager (EKM) support

Envelope Encryption

Стандартний pattern для cloud encryption:

Як працює:

  1. Data Encryption Key (DEK): Random key для шифрування даних
  2. Key Encryption Key (KEK): Master key в KMS для шифрування DEK
  3. Дані шифруються DEK
  4. DEK шифрується KEK → Encrypted DEK
  5. Encrypted DEK зберігається разом з encrypted data

Переваги:

  • Master key ніколи не покидає KMS
  • Efficient: великі дані шифруються швидким symmetric key
  • Scalable: кожен об'єкт може мати унікальний DEK
  • Key rotation: достатньо rotate KEK

Key Rotation

Регулярна заміна ключів для зменшення ризику:

Automatic rotation:

  • AWS KMS: Automatic annual rotation для customer-managed keys
  • Новий key material, старий зберігається для decryption

Manual rotation:

  • Створення нового key
  • Re-encryption даних новим key
  • Deprecation старого key

Best practices:

  • Rotate keys annually minimum
  • Immediate rotation при suspected compromise
  • Automate rotation where possible

10.6 Advanced Криптографічні технології

Hardware Security Modules (HSM)

Dedicated hardware для криптографічних операцій:

Характеристики:

  • Tamper-resistant hardware
  • Keys ніколи не покидають HSM
  • FIPS 140-2 Level 3 certification
  • High-assurance key protection

Cloud HSM services:

  • AWS CloudHSM: Dedicated HSM instances
  • Azure Dedicated HSM: Single-tenant HSM
  • GCP Cloud HSM: HSM-backed keys in Cloud KMS

Use cases:

  • Regulatory compliance (PCI DSS, HIPAA)
  • PKI та certificate authorities
  • Database encryption keys
  • Code signing

BYOK (Bring Your Own Key)

Використання власних ключів у cloud KMS:

Як працює:

  1. Генерація ключа на власному HSM
  2. Wrapping ключа для transfer
  3. Import в cloud KMS
  4. Використання для encryption

Переваги:

  • Повний контроль над key generation
  • Compliance requirements
  • Key escrow outside cloud

Обмеження:

  • Складніший management
  • Не всі features KMS доступні
  • Customer responsibility для key backup

HYOK (Hold Your Own Key)

Ключі залишаються повністю під контролем клієнта:

  • External Key Manager
  • Cloud service запитує key від external KMS
  • Приклад: GCP External Key Manager, Azure Key Vault Managed HSM

Confidential Computing

Захист даних під час processing (encryption in-use):

Технології:

  • Intel SGX: Secure Enclaves
  • AMD SEV: Secure Encrypted Virtualization
  • ARM TrustZone: Trusted Execution Environment

Cloud offerings:

  • Azure Confidential Computing: DCsv2/DCsv3 VMs
  • GCP Confidential VMs: AMD SEV
  • AWS Nitro Enclaves: Isolated compute

Use cases:

  • Multi-party computation
  • Secure data collaboration
  • Protecting data from cloud provider

10.7 Data Protection Practices

Data Loss Prevention (DLP)

Запобігання витоку sensitive data:

Capabilities:

  • Content inspection: Виявлення PII, PCI, PHI у даних
  • Context analysis: Хто, куди, як передає дані
  • Policy enforcement: Block, alert, encrypt

Cloud DLP services:

  • Google Cloud DLP: Content inspection та de-identification
  • Amazon Macie: ML-based data discovery в S3
  • Microsoft Purview: Unified data governance

Data Masking та Tokenization

Data Masking:

  • Заміна sensitive data на fictional але realistic data
  • Для non-production environments
  • Static masking (copy) vs dynamic masking (on-the-fly)

Tokenization:

  • Заміна sensitive data на token
  • Original data зберігається в secure vault
  • Token не має mathematical relationship до original
  • Часто для PCI compliance

Access Controls

Identity-based access:

  • IAM policies для data access
  • Least privilege principle
  • Regular access reviews

Resource-based policies:

  • S3 bucket policies
  • Database access controls
  • Network-level restrictions

Attribute-based access (ABAC):

  • Access based on tags, classifications
  • Dynamic policies
  • Fine-grained control

Backup та Recovery

Backup encryption:

  • Backups повинні бути encrypted
  • Key management для backup keys
  • Cross-region backup considerations

Recovery testing:

  • Regular restore tests
  • Key availability для decryption
  • RTO/RPO validation

10.8 Compliance та Data Protection

GDPR Requirements

  • Encryption: "Appropriate technical measures"
  • Pseudonymization: Processing без identification
  • Right to erasure: Crypto-shredding як option
  • Data portability: Export encrypted data
  • Breach notification: 72-hour requirement

PCI DSS Requirements

  • Requirement 3: Protect stored cardholder data (encryption)
  • Requirement 4: Encrypt transmission (TLS)
  • Key management: Specific requirements для crypto keys
  • HSM: Often required для key storage

HIPAA Requirements

  • Encryption addressable: Not mandatory but strongly recommended
  • Safe harbor: Breach notification exception якщо data encrypted
  • Access controls: Technical safeguards required

Data Residency

  • Деякі jurisdictions вимагають data to stay in-country
  • Cloud provider region selection
  • Encryption keys residency considerations

Висновки

Захист даних у хмарі потребує комплексного підходу, що включає classification, encryption на всіх рівнях, та robust key management. Ключові висновки:

  • Classification first: Не можна захистити те, що не класифіковано
  • Encryption everywhere: At-rest, in-transit, in-use
  • Key management критичний: Encryption без proper key management — ілюзія безпеки
  • Envelope encryption: Стандартний pattern для cloud
  • Cloud KMS: Використовуйте managed services для key management
  • HSM для high assurance: Коли compliance або risk вимагають
  • Defense in depth: Encryption — один шар, не єдиний захист

Практичні рекомендації

  1. Класифікуйте дані перед migration в cloud
  2. Enable default encryption для всіх storage services
  3. Use TLS 1.2+ для всіх connections
  4. Implement envelope encryption через cloud KMS
  5. Enable automatic key rotation
  6. Monitor key usage через audit logs
  7. Test backup restoration включаючи key availability
