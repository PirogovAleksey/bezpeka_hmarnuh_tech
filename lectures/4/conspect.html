Лекція 4. Безпека віртуалізації та контейнеризації

2 год

Вступ

У цій лекції ми розглянемо технології, які лежать в основі хмарних обчислень, — віртуалізацію та контейнеризацію. Без них хмара в її сучасному вигляді просто не існувала б. Водночас разом із потужними можливостями ці технології створюють і специфічні безпекові виклики, які потребують глибокого розуміння.

Віртуалізація дає змогу запускати кілька операційних систем на одному фізичному сервері, ефективно розділяючи апаратні ресурси між різними користувачами та застосунками. Контейнеризація йде далі — вона дозволяє ізолювати окремі застосунки разом із їхніми залежностями, забезпечуючи портативність і швидкість розгортання. Обидві технології критично залежать від правильної конфігурації: навіть найнадійніша технологія стає вразливою, якщо її налаштовано неправильно.

У лекції ми розглянемо загрози для гіпервізорів та віртуальних машин, детально вивчимо безпеку Docker та Kubernetes, а також познайомимося з категорією інструментів CWPP (Cloud Workload Protection Platform — платформа захисту хмарних робочих навантажень), які допомагають захищати інфраструктуру в хмарі.

Чому це важливо?

Статистика свідчить, що більшість інцидентів безпеки в хмарі пов'язані не з вразливостями самих технологій, а з неправильною конфігурацією. Неправильно налаштований Docker-контейнер, занадто широкі права доступу в Kubernetes, незахищений гіпервізор — усе це створює вектори атак (шляхи, якими зловмисник може проникнути в систему), які активно використовуються зловмисниками. За даними Gartner, до 2025 року понад 99% порушень безпеки в хмарі були спричинені саме помилками конфігурації з боку клієнтів, а не недоліками хмарних провайдерів.

Цілі лекції

Після цієї лекції ви зможете:

  • Пояснити архітектуру та типи гіпервізорів, їхні безпекові особливості
  • Визначати основні загрози для віртуалізованих середовищ (втеча з VM, атаки побічних каналів)
  • Застосовувати найкращі практики безпеки Docker: захист образів, Dockerfile, середовища виконання
  • Налаштовувати безпеку Kubernetes: контроль доступу на основі ролей (RBAC), мережеві політики, стандарти безпеки подів
  • Розуміти роль та можливості CWPP у захисті хмарних робочих навантажень
  • Застосовувати принципи безпеки ланцюга постачання програмного забезпечення (supply chain security) для контейнерних середовищ

Структура лекції

Ми рухатимемося від фундаменту до практики: спочатку віртуалізація (базовий рівень ізоляції), потім Docker (контейнерна ізоляція), далі Kubernetes (оркестрація контейнерів), CWPP (інструменти захисту) і, нарешті, — безпека ланцюга постачання (supply chain security). Кожен рівень додає нові можливості, але водночас і нові вектори атак.

4.1 Безпека віртуалізації

Що таке віртуалізація?

Віртуалізація — це технологія, яка дає змогу створювати віртуальні (програмні) версії фізичних ресурсів: серверів, сховищ даних, мереж. У контексті хмарних обчислень найчастіше йдеться про серверну віртуалізацію — можливість запускати кілька віртуальних машин (Virtual Machine, VM) на одному фізичному сервері. Кожна віртуальна машина поводиться як окремий комп'ютер зі своєю операційною системою, хоча насправді всі вони працюють на спільному апаратному забезпеченні.

Ключовий компонент віртуалізації — гіпервізор (англ. hypervisor), також відомий як монітор віртуальних машин (Virtual Machine Monitor, VMM). Гіпервізор — це спеціалізоване програмне забезпечення, яке керує віртуальними машинами та розподіляє фізичні ресурси (процесор, оперативну пам'ять, дисковий простір, мережеві інтерфейси) між ними. Саме гіпервізор забезпечує ізоляцію між VM: кожна машина «думає», що вона працює на виділеному сервері.

Типи гіпервізорів

Тип 1: Bare-metal гіпервізори (апаратні)

Встановлюються безпосередньо на апаратне забезпечення (англ. bare metal — «голе залізо»), без операційної системи-посередника. Це означає, що між гіпервізором та фізичним процесором немає жодного додаткового програмного шару. Гіпервізор першого типу сам виконує роль мінімальної операційної системи — він керує апаратурою та розподіляє ресурси між віртуальними машинами напряму.

Як це працює: при включенні сервера завантажується не Windows чи Linux, а сам гіпервізор. Він безпосередньо отримує контроль над процесором, оперативною пам'яттю та пристроями введення-виведення, а потім створює ізольовані середовища для кожної VM. Це означає, що апаратні ресурси використовуються максимально ефективно — немає «зайвого» шару ОС, який споживає пам'ять та процесорний час.

  • VMware ESXi — найпоширеніший корпоративний гіпервізор, який займає лише ~150 МБ дискового простору. Підтримує до 1024 VM на одному хості. Має власну мінімальну ОС на базі Linux-ядра, оптимізовану виключно для запуску віртуальних машин. Керується через веб-інтерфейс vSphere Client або централізовано через vCenter Server
  • Microsoft Hyper-V — вбудований у Windows Server як серверна роль, а також доступний у Windows 10/11 Pro. Тісно інтегрований з екосистемою Microsoft: Active Directory, System Center, Azure. Хоча формально Hyper-V встановлюється «поверх» Windows, насправді при активації він «підсаджує» Windows як першу гостьову VM, а сам стає гіпервізором Type 1 — це називається «архітектура мікроядра з батьківським розділом»
  • Xen — гіпервізор з відкритим вихідним кодом, розроблений у Кембриджському університеті. Використовує унікальну архітектуру з привілейованою гостьовою VM під назвою Dom0 (Domain 0), яка має прямий доступ до апаратури та керує іншими VM (DomU — непривілейовані домени). Саме Xen використовувала Amazon Web Services (AWS) для свого хмарного сервісу EC2 з моменту запуску у 2006 році (пізніше AWS перейшла на власну систему Nitro на базі KVM)
  • KVM (Kernel-based Virtual Machine) — не окрема програма, а модуль, вбудований безпосередньо в ядро Linux починаючи з версії 2.6.20 (2007). Це означає, що будь-який сучасний сервер з Linux може стати гіпервізором — потрібно лише активувати модуль KVM. Перетворює ядро Linux на гіпервізор Type 1, використовуючи апаратну віртуалізацію процесора (Intel VT-x або AMD-V). Використовується Google Cloud Platform (GCP), DigitalOcean, Red Hat та багатьма іншими провайдерами. Для управління зазвичай використовується інструмент libvirt та графічний інтерфейс virt-manager

Переваги з точки зору безпеки:

  • Менша поверхня атаки: оскільки відсутня повноцінна операційна система хоста з тисячами пакетів, бібліотек та сервісів, зловмисникові значно складніше знайти вразливість для початкового проникнення
  • Вища продуктивність: прямий доступ до апаратури без додаткових шарів абстракції зменшує затримки та збільшує пропускну здатність
  • Апаратна підтримка ізоляції: сучасні процесори мають спеціальні інструкції для віртуалізації (Intel VT-x, AMD-V), які дозволяють гіпервізору Type 1 забезпечити апаратний рівень ізоляції між VM — кожна VM працює у власному «кільці» процесора

Тип 2: Hosted гіпервізори (програмні)

Працюють як звичайний застосунок поверх наявної операційної системи (Windows, macOS, Linux). Це створює додатковий програмний шар між гіпервізором та апаратурою. Запити від гостьової VM проходять довший шлях: VM → гіпервізор → хост-ОС → апаратура (замість VM → гіпервізор → апаратура у Type 1).

Як це працює: ви встановлюєте гіпервізор як звичайну програму на свій комп'ютер — так само, як браузер чи текстовий редактор. Хост-ОС продовжує працювати як зазвичай, а гіпервізор запускає віртуальні машини всередині неї, використовуючи ресурси, які виділяє хост-ОС. Це простіше у налаштуванні, але менш ефективно та безпечно.

  • VMware Workstation (Windows/Linux) / Fusion (macOS) — комерційні рішення для професійної розробки та тестування. Підтримують snapshots (знімки стану VM), клонування, мережі між VM. Workstation Pro дозволяє запускати зашифровані VM та обмежувати доступ паролем
  • Oracle VirtualBox — безкоштовний гіпервізор з відкритим кодом (ліцензія GPLv2). Працює на Windows, macOS, Linux і навіть Solaris. Підтримує широкий спектр гостьових ОС. Популярний серед студентів та розробників завдяки безкоштовності та простоті. Має Extension Pack для підтримки USB 3.0, шифрування дисків та віддаленого доступу через RDP
  • Parallels Desktop — популярне рішення для macOS, особливо для запуску Windows на комп'ютерах Apple. Відзначається тісною інтеграцією з macOS: Coherence Mode дозволяє запускати Windows-застосунки так, ніби вони є рідними програмами macOS. Підтримує чіпи Apple Silicon (M1/M2/M3) через емуляцію ARM

Безпекові особливості Type 2:

  • Більша поверхня атаки: зловмисник може атакувати не лише гіпервізор, а й хост-ОС з усіма її сервісами, драйверами та встановленими програмами. Якщо хост-ОС скомпрометована — усі VM на ній теж під загрозою
  • Залежність від безпеки хост-ОС: будь-яка вразливість у хост-ОС (наприклад, неоновлена Windows з відомою CVE) потенційно відкриває доступ до всіх VM
  • Спільне використання ресурсів: гіпервізор конкурує за ресурси з іншими програмами хост-ОС, що може створювати непередбачувану поведінку

Використання: переважно для розробки, тестування та навчання — коли потрібно швидко створити ізольоване середовище на робочому комп'ютері. Не рекомендується для промислової експлуатації (production-середовищ). У хмарних дата-центрах завжди використовують гіпервізори Type 1.

Загрози для віртуалізованих середовищ

Віртуалізація створює додатковий програмний шар між застосунками та апаратурою, і цей шар сам по собі може стати ціллю атак. Розглянемо чотири основні категорії загроз — від найбільш критичних до найпоширеніших.

VM Escape (втеча з віртуальної машини)

Найнебезпечніша загроза для віртуалізованих середовищ — це можливість зловмисника «вирватися» з віртуальної машини та отримати доступ до гіпервізора або інших VM на тому самому фізичному сервері. Якщо атака успішна, зловмисник фактично отримує контроль над усіма віртуальними машинами, що працюють на цьому хості, оскільки гіпервізор є «коренем довіри» для всіх VM.

Як відбувається VM escape: гіпервізор надає кожній VM набір віртуальних пристроїв — відеокарту, мережеву карту, USB-контролер, дисковод тощо. Ці пристрої емулюються програмно, і код емуляції може містити вразливості. Зловмисник зсередини VM надсилає спеціально сформовані дані віртуальному пристрою, спричиняючи помилку в коді емулятора. Оскільки емулятор працює на рівні гіпервізора (за межами VM), успішна експлуатація такої помилки дозволяє виконати довільний код уже на рівні хост-системи.

Історичні приклади:

  • CVE-2017-4902 (VMware): вразливість у драйвері віртуальної відеокарти (SVGA) давала змогу виконати код на рівні гіпервізора з гостьової VM. Зловмисник міг надіслати спеціально сформовані графічні команди, які спричиняли переповнення буфера (buffer overflow) у коді емулятора відеокарти
  • VENOM (CVE-2015-3456): назва розшифровується як Virtualized Environment Neglected Operations Manipulation. Вразливість знаходилася у контролері віртуального дисковода (floppy drive) емулятора QEMU, яка впливала на гіпервізори Xen та KVM. Цікаво, що вразливість стосувалася компонента, який майже ніхто не використовував, — віртуального флопі-дисковода, але він був увімкнений за замовчуванням у більшості конфігурацій. Це яскравий приклад того, чому слід вимикати непотрібні віртуальні пристрої
  • Cloudburst: одна з перших публічних демонстрацій VM escape на VMware, представлена дослідником Іммуніті (Immunity Inc.) на конференції Black Hat у 2009 році. Атака експлуатувала вразливість у віртуальному відеоадаптері VMware для виконання коду на хост-системі

Актуальні приклади (2024–2025):

  • CVE-2025-22224, CVE-2025-22225, CVE-2025-22226 (VMware ESXi, березень 2025): ланцюжок із трьох вразливостей нульового дня (zero-day), які разом дозволяють здійснити повну втечу з VM. CVE-2025-22224 (CVSS 9.3) — вразливість типу TOCTOU (Time-of-Check Time-of-Use) у компоненті VMCI (Virtual Machine Communication Interface), яка спричиняє переповнення купи (heap overflow) та дає змогу виконати код як процес VMX на хості. CVE-2025-22225 (CVSS 8.2) — довільний запис у ядро ESXi, що дозволяє вирватися з пісочниці (sandbox escape). CVE-2025-22226 (CVSS 7.1) — витік пам'яті процесу VMX через компонент HGFS (Host-Guest File System). Ці вразливості вже активно експлуатувалися зловмисниками в реальних атаках (in the wild) — CISA додала їх до каталогу відомих експлуатованих вразливостей (KEV), а дослідники Huntress виявили сліди використання ще з лютого 2024 року. Вразливість торкнулася VMware ESXi, Workstation, Fusion та Cloud Foundation. Особливо тривожним є те, що ці вразливості вже використовуються у кампаніях з розповсюдження програм-вимагачів (ransomware)
  • CVE-2024-22252, CVE-2024-22253 (VMware, березень 2024): критичні вразливості у компонентах емуляції USB-контролерів (XHCI та UHCI) у VMware ESXi, Workstation та Fusion. Зловмисник з правами адміністратора всередині VM може спричинити use-after-free помилку та виконати код на хості. Для ESXi код виконується в пісочниці VMX, а для Workstation/Fusion — безпосередньо на хост-машині
  • CVE-2026-22719 (VMware Aria Operations, березень 2026): вразливість ін'єкції команд (command injection) із оцінкою CVSS 8.1, що дозволяє неавтентифікованому зловмиснику виконати довільні команди на сервері VMware Aria Operations під час процесу міграції. Причина — недостатня валідація вхідних даних у робочому процесі міграції: зловмисник може вставити спеціальні символи та розділювачі команд, щоб вийти за межі передбаченого контексту. CISA додала цю вразливість до каталогу KEV (Known Exploited Vulnerabilities) у березні 2026 року, підтвердивши активну експлуатацію в реальних атаках. Вразливість торкнулася VMware Cloud Foundation та vSphere Foundation 9.x

Чому VM escape вже не така рідкісна: раніше вважалося, що VM escape потребує надзвичайно глибоких знань і трапляється лише в лабораторних умовах. Однак інциденти 2024–2026 років довели, що ці атаки активно використовуються «в дикій природі» (in the wild) — зокрема, у кампаніях з розповсюдження програм-вимагачів, а також для атак на інфраструктуру управління віртуалізацією. Код гіпервізорів проходить ретельний аудит безпеки, а виробники (VMware/Broadcom, Microsoft, Red Hat) виплачують значні винагороди за знайдені вразливості. Проте складність сучасних гіпервізорів та великий обсяг коду емуляції створюють постійний потік нових вразливостей. Саме тому оперативне оновлення гіпервізорів є абсолютно критичним.

Захист:

  • Регулярне оновлення гіпервізора та гостьових доповнень (VM tools): оновлення самого гіпервізора закриває вразливості в коді емуляції пристроїв (як у випадках VENOM та Cloudburst). Окрім цього, всередині кожної VM зазвичай встановлено спеціальний пакет — гостьові доповнення (VMware Tools, QEMU Guest Agent, Hyper-V Integration Services). Ці доповнення покращують продуктивність VM та дозволяють гіпервізору керувати нею (наприклад, коректно вимикати, синхронізувати час, передавати буфер обміну). Але оскільки гостьові доповнення є «мостом» між VM та гіпервізором, вразливість у них може стати шляхом для втечі з VM. Тому їх також потрібно регулярно оновлювати
  • Мінімізація віртуального обладнання: вимкнення непотрібних віртуальних пристроїв (дисковод, USB-контролер, послідовний та паралельний порти, аудіопристрій). Кожен емульований пристрій — це додатковий код, який може містити вразливості. Як показав VENOM, навіть застарілий флопі-контролер може стати вектором атаки. Принцип простий: якщо VM не потрібен USB — вимкніть віртуальний USB-контролер
  • Обережне використання вкладеної віртуалізації (nested virtualization): запуск гіпервізора всередині VM (наприклад, KVM всередині VMware) створює додатковий рівень складності. Зовнішній гіпервізор повинен правильно емулювати апаратні інструкції віртуалізації (VT-x/AMD-V) для внутрішнього гіпервізора — це значно збільшує поверхню атаки та кількість потенційних помилок. Використовуйте nested virtualization лише для тестування та розробки, ніколи — для продуктивних навантажень
  • Сегментація за рівнем чутливості: не розміщуйте VM з різними рівнями довіри на одному фізичному сервері. Наприклад, VM із конфіденційними фінансовими даними не повинні працювати на тому самому хості, що й VM для розробки з доступом до інтернету

Атаки побічних каналів (Side-Channel Attacks)

Це особливий клас атак, які використовують не логічні вразливості у програмному коді, а фізичні характеристики апаратури. У віртуалізованих середовищах вони особливо небезпечні, оскільки кілька VM спільно використовують один і той самий процесор, його кеш-пам'ять та інші ресурси.

Принцип атаки: уявіть бібліотеку, де кілька людей користуються одними й тими самими полицями. Спостерігаючи, які книги зникають і з'являються на полицях, можна зрозуміти, що читає інша людина — навіть не маючи прямого доступу до її книг. Аналогічно, зловмисник зі своєї VM може «підслуховувати» дані іншої VM, аналізуючи стан спільної кеш-пам'яті процесора. Якщо доступ до певної ділянки кешу відбувається швидше за очікуване — значить, інша VM нещодавно завантажила туди свої дані, і за цими часовими різницями можна відновити конфіденційну інформацію (наприклад, криптографічні ключі).

  • Spectre/Meltdown (2018): дві вразливості, які потрясли всю ІТ-індустрію, оскільки стосувалися не програмного забезпечення, а самих процесорів. Meltdown дозволяв звичайному процесу прочитати пам'ять ядра ОС, що раніше вважалося неможливим. Spectre давав змогу обманути механізм передбачення переходів (branch prediction) процесора, змушуючи його «випадково» виконати інструкції, які дають доступ до пам'яті іншого процесу або VM. Обидві атаки експлуатували механізм спекулятивного виконання (speculative execution) — оптимізацію, коли процесор «наперед» виконує інструкції, які ще не були перевірені, щоб не простоювати. Ці вразливості торкнулися практично кожного процесора Intel за останні 20 років та частини процесорів AMD і ARM
  • L1TF (L1 Terminal Fault): вразливість, яка давала змогу читати дані з кеш-пам'яті першого рівня (L1 cache) інших VM, що працюють на тому самому процесорному ядрі. L1 cache — це найшвидший і найближчий до процесора рівень кешу (зазвичай 32–64 КБ на ядро), де зберігаються дані, з якими процесор працює прямо зараз. Якщо два потоки (threads) одного ядра належать різним VM — зловмисник з одного потоку може прочитати дані іншого
  • MDS (Microarchitectural Data Sampling): сімейство атак (RIDL, Fallout, ZombieLoad) на внутрішні буфери процесора — store buffers, fill buffers, load ports. Ці буфери є тимчасовими сховищами даних усередині процесора, які не очищуються належним чином при перемиканні між процесами та VM. Зловмисник може зчитати «залишки» даних попереднього процесу з цих буферів

Чому це важливо для хмари: у публічних хмарних середовищах (AWS, Azure, GCP) ваша VM може працювати на тому самому фізичному процесорі, що й VM іншого клієнта. Атаки побічних каналів потенційно дозволяють одному клієнту «підслуховувати» дані іншого — саме тому хмарні провайдери дуже серйозно ставляться до мітигацій цих вразливостей і пропонують виділені хости (dedicated instances) для чутливих робочих навантажень.

Захист:

  • Своєчасне оновлення мікрокоду процесора: виробники процесорів (Intel, AMD) випускають оновлення мікрокоду, які виправляють апаратні вразливості на рівні мікрокоманд процесора. Ці оновлення зазвичай доставляються через оновлення BIOS/UEFI або через пакети ОС. Без мікрокодних оновлень програмні патчі ОС лише частково ефективні
  • Патчі операційної системи: ядро ОС також потребує оновлень для активації мітигацій. Наприклад, KPTI (Kernel Page Table Isolation) для захисту від Meltdown створює окремі таблиці сторінок пам'яті для ядра і для користувацьких процесів, щоб процес не міг «побачити» адреси ядра навіть через спекулятивне виконання. Ця мітигація може знижувати продуктивність на 5–30% залежно від типу навантаження
  • Виділені сервери (dedicated hosts): для критичних робочих навантажень (обробка платіжних даних, медична інформація, криптографічні операції) рекомендується орендувати виділені фізичні сервери без спільного розміщення з іншими орендарями (multi-tenancy). Усі великі хмарні провайдери пропонують цю опцію: AWS Dedicated Hosts, Azure Dedicated Hosts, GCP Sole-tenant Nodes
  • Контроль Hyper-Threading / SMT: технологія одночасної багатопотоковості (SMT — Simultaneous Multi-Threading, яку Intel називає Hyper-Threading) дозволяє двом потокам виконуватися на одному фізичному ядрі, спільно використовуючи його ресурси. Саме це спільне використання робить можливими атаки L1TF та MDS. Для максимального захисту SMT можна вимкнути, але це зменшить пропускну здатність процесора приблизно на 20–30%. Компроміс: не розміщувати VM різних клієнтів або різних рівнів довіри на потоках одного ядра

Hyperjacking (перехоплення гіпервізора)

Атака, під час якої зловмисник непомітно встановлює шкідливий гіпервізор «під» легітимну систему. Фактично зловмисник створює прихований рівень управління, який контролює все, що відбувається на сервері. Це один із найскладніших типів атак, оскільки шкідливий гіпервізор може бути практично невидимим для всіх діагностичних інструментів — адже ці інструменти самі працюють «зверху» і не мають доступу до рівня, який знаходиться «під» ними.

Принцип атаки: зловмисник використовує апаратні інструкції віртуалізації процесора (Intel VT-x або AMD-V) для того, щоб «підсадити» свій мініатюрний гіпервізор під чинну операційну систему. Після цього ОС продовжує працювати як зазвичай, але вже як гостьова VM під контролем шкідливого гіпервізора. Аналогія: уявіть, що хтось непомітно вмонтував двостороннє дзеркало в підлогу вашої кімнати — ви ходите по ній як зазвичай і нічого не помічаєте, але спостерігач знизу бачить кожен ваш рух.

  • Непомітність: оригінальна операційна система стає «гостем», навіть не підозрюючи про це — вона продовжує працювати як зазвичай, усі програми, мережеві з'єднання, системні виклики функціонують нормально. Єдина відмінність — мінімальне уповільнення, яке практично неможливо відрізнити від звичайних коливань продуктивності
  • Повний контроль: зловмисник отримує можливість перехоплювати будь-які дані (включно з паролями та ключами шифрування), модифікувати вміст оперативної пам'яті, маніпулювати процесами, підміняти мережевий трафік — і все це непомітно для ОС та її засобів захисту (антивірусів, EDR-агентів)
  • Blue Pill — концептуальний руткіт (rootkit), продемонстрований польською дослідницею Джоанною Рутковською на конференції Black Hat у 2006 році. Назва «Синя пігулка» — відсилка до фільму «Матриця», де синя пігулка залишає людину в ілюзорному світі. Blue Pill використовував апаратну віртуалізацію AMD-V для прихованого перехоплення контролю над системою. Хоча це був лише дослідницький прототип, він продемонстрував принципову можливість такої атаки та стимулював розробку захисних механізмів
  • SubVirt — ще один дослідницький проєкт (Microsoft Research та Мічиганський університет, 2006), що демонстрував аналогічну концепцію: шкідливий гіпервізор, який підсаджувався під Windows або Linux і залишався невидимим для всіх антивірусних сканерів

Захист:

  • Secure Boot та UEFI: Secure Boot — це механізм, вбудований у прошивку UEFI (наступник BIOS), який перевіряє цифровий підпис кожного компонента завантажувального ланцюжка: прошивки → завантажувача → ядра ОС → драйверів. Якщо будь-який компонент модифіковано або підмінено (зокрема, вставлено шкідливий гіпервізор), Secure Boot зупинить завантаження та повідомить про порушення цілісності. Це найефективніший захист від hyperjacking, оскільки шкідливий гіпервізор просто не зможе завантажитися
  • TPM (Trusted Platform Module): апаратний модуль довіреної платформи — окремий криптографічний чіп, впаяний у материнську плату. TPM зберігає хеш-значення (цифрові «відбитки») кожного етапу завантаження системи у спеціальних регістрах PCR (Platform Configuration Registers). Якщо будь-який компонент завантажувального ланцюжка змінено — хеш-значення не збіжиться, і система виявить порушення. TPM також зберігає криптографічні ключі, прив'язані до конкретного стану системи: якщо систему було скомпрометовано, ключі стають недоступними
  • Механізми атестації (attestation): процес, при якому сервер надає криптографічний доказ (на основі даних TPM) того, що на ньому запущено саме те програмне забезпечення, яке очікується. Це дає змогу віддалено перевірити цілісність сервера — наприклад, центральна система безпеки може автоматично перевіряти всі сервери в дата-центрі та блокувати ті, що не пройшли атестацію. У хмарних середовищах це реалізується через Intel TXT (Trusted Execution Technology) або AMD SEV (Secure Encrypted Virtualization)
  • Моніторинг продуктивності: хоча шкідливий гіпервізор важко виявити програмно, він неминуче вносить мінімальні затримки (overhead) при обробці системних викликів та переривань. Спеціалізовані інструменти моніторингу можуть виявити ці аномалії — наприклад, якщо затримка обробки переривань раптом збільшилася на кілька мікросекунд без видимих причин

Вичерпання ресурсів (Resource Exhaustion / «Шумний сусід»)

Ситуація, коли одна VM споживає непропорційно багато ресурсів, негативно впливаючи на продуктивність інших VM на тому самому фізичному сервері. Цю проблему називають «ефектом шумного сусіда» (noisy neighbor), за аналогією з гучним сусідом у багатоквартирному будинку. Хоча це не є класичною «атакою» у традиційному розумінні (частіше виникає ненавмисно), вичерпання ресурсів може використовуватися зловмисниками цілеспрямовано для атак відмови в обслуговуванні (DoS — Denial of Service) проти інших орендарів хмарної платформи.

Як це впливає на безпеку: вичерпання ресурсів може не лише знижувати продуктивність, а й порушувати роботу критичних систем захисту. Наприклад, якщо VM з системою моніторингу безпеки (SIEM, IDS) не отримує достатньо процесорного часу, вона може пропускати підозрілі події. Або якщо VM з базою даних зазнає нестачі пам'яті, вона може почати використовувати незашифрований swap-файл на диску, створюючи ризик витоку конфіденційних даних.

  • Нестача процесорного часу (CPU starvation): одна VM монополізує процесор, виконуючи ресурсомісткі обчислення (наприклад, майнінг криптовалют або обробку великих даних). Інші VM на тому самому хості отримують лише «залишки» процесорного часу, що призводить до різкого уповільнення або навіть зависання сервісів
  • Тиск на пам'ять (memory pressure): надмірне споживання оперативної пам'яті однією VM може призвести до того, що гіпервізор почне застосовувати техніки повторного використання пам'яті — memory ballooning (примусове вивільнення пам'яті з VM через спеціальний драйвер) або swapping (скидання сторінок пам'яті на диск). Обидва механізми різко знижують продуктивність усіх VM на хості
  • Конкуренція за введення-виведення (I/O contention): інтенсивна робота однієї VM з дисковою підсистемою (наприклад, масове копіювання даних або інтенсивний запис логів) уповільнює дискові операції всіх інших VM, оскільки вони спільно використовують фізичні диски або контролер сховища. Особливо критично для VM з базами даних, де затримка вводу-виводу безпосередньо впливає на час відповіді
  • Вичерпання мережевої пропускної здатності: одна VM може зайняти весь мережевий канал фізичного сервера, наприклад, під час DDoS-атаки (як ціль або як джерело) або при передачі великих обсягів даних. Інші VM на тому самому хості втрачають мережевий зв'язок або зазнають значних затримок

Захист:

  • Квоти та ліміти ресурсів (resource quotas/limits): жорсткі обмеження на споживання CPU, пам'яті та I/O для кожної VM. Наприклад, у VMware можна задати «reservation» (гарантований мінімум ресурсів), «limit» (максимум) та «shares» (пріоритет при конкуренції). У KVM аналогічні обмеження налаштовуються через cgroups. Важливо задавати не лише ліміти (верхню межу), а й резервації (гарантований мінімум) — щоб критичні VM завжди отримували необхідні ресурси
  • Політики якості обслуговування (QoS — Quality of Service): механізми пріоритизації ресурсів для критичних робочих навантажень. QoS дозволяє визначити, що VM з продуктивною базою даних має пріоритет над VM для розробки при конкуренції за ресурси. Для мережі QoS реалізується через traffic shaping (обмеження швидкості) та traffic prioritization (пріоритизація за типом трафіку)
  • Планування ємності (capacity planning): правильний розрахунок кількості VM на фізичний сервер з урахуванням піків навантаження. Правило великого пальця: завантаження фізичного сервера не повинно перевищувати 70–80% навіть у піковий час, щоб залишався запас для непередбачених ситуацій. Інструменти моніторингу (Prometheus, Zabbix, vRealize Operations) допомагають відстежувати реальне навантаження та планувати розширення
  • Моніторинг та сповіщення: автоматичні оповіщення при перевищенні порогів споживання ресурсів — наприклад, якщо VM споживає понад 90% доступної пам'яті або дискові затримки (latency) перевищують 10 мс. Це дозволяє адміністраторам реагувати до того, як проблема вплине на інші VM

Найкращі практики безпеки віртуалізації

Безпека віртуалізованого середовища будується на трьох рівнях: захист самого гіпервізора (фундамент), захист віртуальних машин (робочі навантаження) та захист мережевої взаємодії між ними. Слабкість на будь-якому рівні може скомпрометувати всю інфраструктуру.

Захист гіпервізора

Гіпервізор — це «корінь довіри» для всієї віртуалізованої інфраструктури. Якщо зловмисник отримає доступ до гіпервізора, він зможе контролювати всі VM на цьому сервері — читати їхню пам'ять, перехоплювати мережевий трафік, модифікувати дані. Тому захист гіпервізора є абсолютним пріоритетом:

  • Мінімальна інсталяція: встановлювати лише необхідні компоненти — чим менше коду працює на гіпервізорі, тим менше потенційних вразливостей. Наприклад, VMware ESXi у мінімальній конфігурації займає лише ~150 МБ, тоді як повноцінний Linux-сервер — десятки гігабайтів. Кожен додатковий пакет, сервіс або драйвер — це додатковий код, який може містити помилки
  • Окрема мережа управління (management network): трафік адміністрування гіпервізора (SSH, веб-консоль, API) повинен проходити через окрему фізичну або логічну мережу, ізольовану від робочого (production) трафіку VM. Якщо зловмисник скомпрометує одну з VM, він не повинен мати мережевий доступ до інтерфейсу управління гіпервізором. На практиці це означає окремий мережевий інтерфейс (NIC) або VLAN для управління
  • Багатофакторна автентифікація (MFA): для доступу до консолі управління гіпервізором (vSphere, Proxmox, oVirt). Один лише пароль недостатній — навіть складний пароль може бути скомпрометований через фішинг, brute-force або витік бази даних. MFA додає другий фактор (одноразовий код, апаратний ключ), що робить компрометацію значно складнішою
  • Регулярні оновлення: встановлення патчів (виправлень) безпеки без затримок. Вразливості в гіпервізорах, такі як VENOM або CVE-2017-4902, можуть дозволити втечу з VM — тому оновлення гіпервізора є критично важливішим, ніж оновлення звичайного сервера. Рекомендується мати процес тестування та швидкого розгортання патчів — не довше ніж 72 години для критичних вразливостей
  • Журналювання та моніторинг: централізований збір журналів (логів) подій з усіх гіпервізорів до SIEM-системи (Security Information and Event Management). Слід відстежувати: спроби входу (успішні та невдалі), створення та видалення VM, зміни конфігурації, аномальне споживання ресурсів. Без централізованого моніторингу адміністратор може не помітити компрометацію тижнями

Захист віртуальних машин

Кожна VM — це окремий сервер з власною операційною системою, і до неї застосовуються ті самі принципи безпеки, що й до фізичного сервера. Однак віртуалізація додає специфічні аспекти, які потрібно враховувати:

  • Зміцнення (hardening): застосування еталонних конфігурацій безпеки CIS Benchmarks для гостьових ОС. CIS Benchmarks — це детальні покрокові інструкції від Center for Internet Security, які описують, які сервіси вимкнути, які параметри ядра налаштувати, які права доступу обмежити. Наприклад, CIS Benchmark для Ubuntu 22.04 містить понад 300 рекомендацій. Інструменти на кшталт OpenSCAP або Ansible дозволяють автоматизувати перевірку та застосування цих рекомендацій
  • Шифрування VM: шифрування дисків віртуальних машин для захисту даних у стані спокою (at rest). Це критично важливо, оскільки файли дисків VM (VMDK, QCOW2, VHD) зберігаються на спільному сховищі — адміністратор сховища або зловмисник з доступом до SAN/NAS може скопіювати файл диска VM і прочитати всі дані. VMware підтримує VM Encryption, а для KVM використовується LUKS-шифрування на рівні гостьової ОС або шифрування на рівні сховища
  • Управління знімками (snapshots): знімок VM фіксує повний стан машини — диск, пам'ять, конфігурацію — на певний момент часу. Це зручно для резервного копіювання та тестування, але знімки можуть містити конфіденційні дані: паролі в пам'яті, сесійні токени, розшифровані дані. Тому знімки слід зберігати з тим самим рівнем захисту, що й саму VM, та своєчасно видаляти застарілі знімки. Також надмірна кількість знімків знижує продуктивність, оскільки кожна операція запису створює додаткове навантаження
  • Шаблони VM (golden images): стандартизовані, попередньо захищені образи (так звані «золоті образи»), з яких створюються нові віртуальні машини. Замість того щоб кожен раз вручну налаштовувати ОС, встановлювати оновлення та застосовувати hardening, адміністратор створює один еталонний образ з усіма необхідними налаштуваннями безпеки і використовує його як шаблон. Шаблони повинні регулярно оновлюватися для включення свіжих патчів

Мережева безпека

У віртуалізованому середовищі мережева безпека має свою специфіку: весь мережевий трафік між VM на одному хості проходить через віртуальний комутатор (virtual switch) всередині гіпервізора і ніколи не потрапляє на фізичну мережу. Це означає, що традиційні фізичні міжмережеві екрани «не бачать» цей трафік і не можуть його контролювати. Тому потрібні спеціальні механізми:

  • Віртуальні комутатори (virtual switches): програмні мережеві пристрої всередині гіпервізора, через які проходить увесь трафік VM. Наприклад, VMware vSwitch або Open vSwitch (OVS) для KVM. Віртуальні комутатори дозволяють створювати ізольовані порт-групи — набори портів, між якими дозволено трафік. VM з різних порт-груп не можуть обмінюватися даними напряму, навіть якщо працюють на одному фізичному сервері
  • Тегування VLAN (Virtual LAN): логічна сегментація мережі, яка дозволяє розділити одну фізичну мережу на кілька ізольованих логічних сегментів. Наприклад, VM середовища розробки (VLAN 10), тестування (VLAN 20) та продуктивного середовища (VLAN 30) можуть працювати на одному фізичному сервері, але не матимуть мережевого доступу одна до одної без явного налаштування маршрутизації. Це реалізує принцип найменших привілеїв на мережевому рівні
  • Розподілені міжмережеві екрани та мікросегментація: технології на кшталт VMware NSX або Cisco ACI дозволяють створювати правила фільтрації трафіку не лише між мережевими сегментами, а й між окремими VM. Наприклад, можна дозволити VM з веб-сервером спілкуватися лише з VM бази даних на порту 5432 (PostgreSQL) і заборонити будь-який інший трафік. Якщо зловмисник скомпрометує веб-сервер, він не зможе вільно переміщуватися по мережі (lateral movement) до інших серверів — кожне з'єднання контролюється окремим правилом

Детальний аналіз атак побічних каналів

Атаки побічних каналів (side-channel attacks) використовують фізичні характеристики системи, а не логічні помилки в коді. У віртуалізованих середовищах вони особливо небезпечні, оскільки кілька VM спільно використовують один процесор, його кеш-пам'ять та внутрішні буфери. Атакуючий може опосередковано «зчитувати» конфіденційні дані іншої VM, аналізуючи затримки доступу до кеша або поведінку процесора.

Захист від атак побічних каналів

На системі Linux можна перевірити, які мітигації (заходи захисту) активні для вашого процесора:

# Перевірка стану мітигацій для кожної відомої вразливості процесора
cat /sys/devices/system/cpu/vulnerabilities/*

# Типовий вивід (кожен рядок — окрема вразливість та стан її мітигації):
spectre_v1: Mitigation: usercopy/swapgs barriers...   # Захист від Spectre v1 активний
spectre_v2: Mitigation: Retpolines, IBPB...           # Захист від Spectre v2 активний
meltdown: Mitigation: PTI                             # PTI = Page Table Isolation — захист від Meltdown
l1tf: Mitigation: PTE Inversion; SMT vulnerable       # Захист є, але Hyper-Threading залишається вразливим
mds: Mitigation: Clear CPU buffers; SMT vulnerable    # Буфери очищуються, але SMT все ще ризик

Зверніть увагу на позначки SMT vulnerable — вони означають, що хоча основні мітигації активні, технологія Hyper-Threading (SMT — Simultaneous Multi-Threading) залишається потенційним вектором атаки. Для критичних навантажень (наприклад, обробки фінансових даних або персональної інформації) рекомендується вимикати SMT або використовувати виділені хости.

  • Оновлення мікрокоду: регулярне оновлення мікрокоду процесора для виправлення апаратних вразливостей
  • Патчі ядра: KPTI (Kernel Page Table Isolation — ізоляція таблиць сторінок ядра) — захист від Meltdown
  • Retpolines: спеціальна компіляторна техніка, яка нейтралізує Spectre v2, змінюючи спосіб обробки непрямих переходів
  • Вимкнення SMT/Hyper-Threading: вимикання одночасної багатопотоковості для критичних робочих навантажень усуває ризики витоку даних між потоками
  • Виділені хости (dedicated hosts): використання фізичних серверів без спільного розміщення з іншими орендарями

4.2 Безпека Docker

Що таке контейнеризація?

Контейнеризація — це легка форма віртуалізації на рівні операційної системи. На відміну від VM, контейнери не емулюють повну операційну систему — вони використовують ядро хост-системи, ізолюючи лише простір користувача (userspace). Завдяки цьому контейнери запускаються за лічені секунди (на відміну від хвилин для VM) і споживають значно менше ресурсів.

Docker — найпопулярніша платформа контейнеризації, яка стала стандартом де-факто в індустрії. Docker використовує вбудовані механізми ядра Linux для забезпечення ізоляції:

  • Простори імен (namespaces): забезпечують ізоляцію процесів, мережі, файлової системи та користувачів. Кожен контейнер «бачить» лише свої процеси, свою мережу та свою файлову систему, навіть якщо фізично все працює на одній машині
  • Контрольні групи (cgroups — Control Groups): обмежують ресурси, доступні контейнеру (процесорний час, оперативну пам'ять, швидкість введення-виведення). Це запобігає ситуації, коли один контейнер споживає всі ресурси сервера
  • Об'єднані файлові системи (Union File Systems): багатошарова файлова система, яка дозволяє ефективно зберігати образи контейнерів — спільні шари використовуються повторно, що економить дисковий простір

Ключовий компроміс: контейнери обирають ефективність та швидкість на шкоду рівню ізоляції. Тому в ситуаціях, де ізоляція критична (наприклад, запуск коду ненадійних користувачів), слід використовувати VM або спеціалізовані рішення, що поєднують переваги обох підходів — наприклад, gVisor (мікроядро від Google, що перехоплює системні виклики контейнера) або Kata Containers (кожен контейнер працює всередині легкої VM).

Загрози безпеці Docker

1. Вразливі базові образи

Кожен контейнер створюється на основі базового образу (base image), який може містити вразливості ще до того, як ви додасте свій код:

  • Застарілі пакети з відомими CVE — якщо базовий образ давно не оновлювався, у ньому можуть бути десятки або навіть сотні вразливостей з публічними експлойтами
  • Шкідливе ПЗ (malware) у публічних образах — на Docker Hub може хто завгодно публікувати образи, тому серед них трапляються навмисно шкідливі
  • Тайпосквотинг (typosquatting) — створення образів із назвами, дуже схожими на популярні (наприклад, ngingx замість nginx), в розрахунку на помилку користувача при введенні

Приклад: у 2018 році на Docker Hub було виявлено 17 шкідливих образів, які таємно запускали програми для майнінгу криптовалют. Ці образи було завантажено мільйони разів, перш ніж їх виявили та видалили.

2. Втеча з контейнера (Container Escape)

Аналог втечі з VM, але для контейнерів. Оскільки контейнери використовують спільне ядро з хост-системою, вразливість у ядрі або в середовищі виконання контейнерів може дозволити зловмиснику «вирватися» на хост:

  • CVE-2019-5736 (runc): критична вразливість, що давала змогу перезаписати виконуваний файл runc (компонент, який безпосередньо запускає контейнери) зсередини контейнера, отримуючи таким чином контроль над хостом
  • CVE-2020-15257: вразливість у containerd (середовищі виконання контейнерів), що давала змогу отримати доступ до файлової системи хост-машини
  • Dirty COW (CVE-2016-5195): вразливість в ядрі Linux, що дозволяла підвищення привілеїв (privilege escalation) — зловмисник з контейнера міг отримати права суперкористувача (root) на хості
  • CVE-2024-21626 «Leaky Vessels» (runc, 2024): вразливість у runc версій ≤1.1.11, пов'язана з витоком внутрішнього файлового дескриптора. Зловмисник міг маніпулювати робочою директорією новоствореного процесу контейнера, щоб отримати доступ до файлової системи хоста. Експлуатація можлива через запуск шкідливого образу або збірку з шкідливого Dockerfile. Оцінка CVSS — 8.6 (високий). Виправлено у runc 1.1.12
  • CVE-2025-31133, CVE-2025-52565, CVE-2025-52881 (runc, 2025): три критичні вразливості у runc, що експлуатують слабкості в операціях монтування та захисту файлів під час створення контейнера. Зловмисник може подолати ізоляцію між контейнером та хостом через маніпулювання монтуваннями та символічними посиланнями, отримуючи root-доступ до хост-системи. Виправлено у runc версіях 1.2.8, 1.3.3 та 1.4.0-rc.3
  • CVE-2025-9074 (Docker Desktop, 2025): критична вразливість (CVSS 9.3) у Docker Desktop для Windows та macOS (до версії 4.44.3). Docker Engine API був доступний без автентифікації зсередини будь-якого контейнера через фіксовану адресу підмережі (192.168.65.7:2375). Це дозволяло шкідливому або скомпрометованому контейнеру створювати нові контейнери з монтуванням хостової файлової системи, ескалювати привілеї та отримувати повний контроль над хост-машиною. На Windows зловмисник міг змонтувати весь диск від імені адміністратора та підмінити системні DLL. Виправлено у Docker Desktop 4.44.3

3. Неправильна конфігурація (Misconfiguration)

Найпоширеніша причина інцидентів безпеки — це не складні хакерські атаки, а банальні помилки конфігурації:

  • Привілейовані контейнери (privileged containers): контейнер, запущений у привілейованому режимі, фактично отримує повні права root на хост-системі, повністю нівелюючи ізоляцію
  • Відкритий Docker-сокет: доступ до файлу /var/run/docker.sock зсередини контейнера дає змогу керувати Docker-демоном хоста — створювати, видаляти та запускати будь-які контейнери, фактично отримуючи повний контроль над сервером
  • Монтування критичних директорій: підключення системних каталогів хоста (наприклад, /etc, /) до контейнера відкриває прямий доступ до конфігурації та файлів хост-машини
  • Запуск від імені root: якщо процеси всередині контейнера працюють від імені суперкористувача (root), у разі втечі з контейнера зловмисник одразу отримує найвищі привілеї

4. Витік секретів (Secrets Exposure)

Конфіденційні дані (паролі, ключі, токени) можуть потрапити в образ контейнера або його конфігурацію і стати доступними зловмисникам:

  • API-ключі у змінних середовища — хоча змінні середовища здаються «прихованими», їх легко переглянути через docker inspect або з процесу всередині контейнера
  • Паролі в Dockerfile — навіть якщо ви видалите секрет у наступному шарі, він залишиться в історії шарів образу (layer history), і будь-хто з доступом до образу зможе його відновити
  • Сертифікати та приватні ключі — вбудовані безпосередньо в образи контейнерів, вони стають доступними всім, хто має доступ до реєстру образів

Найкращі практики безпеки Docker

Безпека образів

Вибір базового образу — перше і найважливіше рішення з точки зору безпеки:

# ПОГАНО: тег "latest" може вказувати на різні версії в різний час,
# що робить збірку непередбачуваною і може непомітно внести вразливості
FROM ubuntu:latest

# КРАЩЕ: конкретна версія — збірка завжди дає однаковий результат
FROM ubuntu:22.04

# НАЙКРАЩЕ: мінімальний образ Alpine (~5 МБ замість ~77 МБ для Ubuntu)
# Менше пакетів = менше потенційних вразливостей
FROM alpine:3.18

# АБО: distroless від Google — взагалі не містить shell, пакетного менеджера,
# утиліт. Навіть якщо зловмисник потрапить у контейнер — йому нічим скористатися
FROM gcr.io/distroless/java17-debian11
  • Використовуйте мінімальні базові образи: Alpine (≈5 МБ), Distroless (без shell та пакетного менеджера), scratch (порожній образ) — чим менше компонентів в образі, тим менше потенційних вразливостей
  • Скануйте образи на вразливості: інструменти Trivy, Snyk, Clair, Docker Scout автоматично перевіряють усі пакети в образі на відомі вразливості
  • Підписуйте образи: Docker Content Trust або Sigstore/Cosign гарантують, що образ не було підмінено або модифіковано після збірки
  • Використовуйте приватний реєстр образів: з контролем доступу та автоматичним скануванням при завантаженні

Безпечний Dockerfile

Розглянемо приклад безпечного Dockerfile з поясненням кожного рядка:

# === ПЕРШИЙ ЕТАП: збірка (builder) ===
# Використовуємо повний образ Go лише для компіляції
FROM golang:1.21 AS builder
WORKDIR /app
COPY . .
# CGO_ENABLED=0 — статична збірка без залежності від C-бібліотек
RUN CGO_ENABLED=0 go build -o myapp

# === ДРУГИЙ ЕТАП: фінальний образ ===
# Alpine — мінімальний образ (~5 МБ), значно менша поверхня атаки
FROM alpine:3.18
# Створюємо непривілейованого користувача та групу
# Прапорець -S означає "системний" (без домашньої директорії)
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
# Перемикаємося на непривілейованого користувача
USER appuser
# Копіюємо ЛИШЕ скомпільований бінарник з першого етапу
# Компілятор, вихідний код, залежності — все залишилося в builder
COPY --from=builder /app/myapp /myapp
ENTRYPOINT ["/myapp"]

Зверніть увагу: фінальний образ не містить компілятора Go (≈800 МБ), вихідного коду, інструментів збірки. Якщо зловмисник і проникне в контейнер, він не зможе скомпілювати або завантажити додаткові інструменти, оскільки в мінімальному образі Alpine відсутні навіть базові утиліти, а процес працює від непривілейованого користувача.

  • Багатоетапна збірка (multi-stage builds): фінальний образ не містить інструментів збірки (компіляторів, бібліотек розробки), що зменшує його розмір і поверхню атаки
  • Непривілейований користувач: інструкція USER в Dockerfile гарантує, що процеси в контейнері не працюють від імені root
  • Мінімізація кількості шарів: об'єднуйте команди RUN для зменшення розміру образу та спрощення аудиту
  • Не копіюйте секрети в образ: використовуйте механізм build secrets (--mount=type=secret) або передавайте секрети під час запуску контейнера
  • Файл .dockerignore: виключає непотрібні файли (документацію, тести, конфігурацію IDE) зі збірки образу

Безпека під час виконання (Runtime Security)

Навіть якщо образ контейнера побудовано безпечно, важливо також правильно запускати його. Кожен прапорець у команді нижче додає окремий рівень захисту:

docker run \
  --read-only \                           # Файлова система лише для читання
  --cap-drop=ALL \                        # Забрати ВСІ Linux capabilities
  --cap-add=NET_BIND_SERVICE \            # Повернути лише одну: можливість слухати порти < 1024
  --security-opt=no-new-privileges:true \ # Заборонити підвищення привілеїв
  --memory=512m \                         # Максимум 512 МБ оперативної пам'яті
  --cpus=1 \                              # Максимум 1 ядро CPU
  --user 1000:1000 \                      # Запуск від непривілейованого користувача (UID:GID)
  myimage

Linux capabilities (привілеї) — це механізм ядра Linux, який розбиває повні права root на окремі дрібні привілеї. Наприклад, NET_BIND_SERVICE дозволяє слухати порти нижче 1024, SYS_ADMIN дозволяє монтувати файлові системи, NET_RAW дозволяє створювати «сирі» мережеві пакети. Замість того щоб давати контейнеру повний root-доступ, ми забираємо усі привілеї і повертаємо лише ті, що дійсно потрібні для роботи застосунку.

  • Скидання привілеїв (capabilities): --cap-drop=ALL забирає всі додаткові привілеї, після чого додаються лише необхідні — наприклад, NET_BIND_SERVICE для прослуховування портів нижче 1024
  • Файлова система лише для читання: прапорець --read-only запобігає запису у файлову систему контейнера, що ускладнює завантаження шкідливого коду
  • Заборона отримання нових привілеїв: --security-opt=no-new-privileges запобігає підвищенню привілеїв через SUID/SGID-біти
  • Обмеження ресурсів: --memory та --cpus встановлюють жорсткі ліміти споживання пам'яті та процесорного часу
  • Профілі Seccomp: обмежують набір системних викликів, доступних контейнеру — наприклад, можна заборонити виклики, пов'язані з монтуванням файлових систем або завантаженням модулів ядра
  • AppArmor/SELinux: системи обов'язкового контролю доступу (mandatory access control), які додатково обмежують дії контейнера на рівні ядра ОС

Управління секретами

Правильне зберігання та передача конфіденційних даних (паролів, ключів API, токенів) є однією з найважливіших складових безпеки контейнерів:

  • Docker Secrets: вбудований механізм для Docker Swarm — секрети зберігаються зашифрованими та доступні контейнерам як файли в пам'яті
  • Змінні середовища (environment variables): використовуйте лише для некритичних налаштувань (назва середовища, рівень журналювання), а не для паролів
  • Зовнішні менеджери секретів: спеціалізовані рішення, такі як HashiCorp Vault або AWS Secrets Manager, забезпечують централізоване, зашифроване зберігання з контролем доступу та аудитом
  • Секрети Kubernetes: з обов'язковим увімкненням шифрування у стані спокою (encryption at rest)

4.3 Безпека Kubernetes

Архітектура Kubernetes

Kubernetes (скорочено K8s) — це платформа оркестрації (автоматизованого управління) контейнерів, яка автоматизує розгортання, масштабування та управління контейнеризованими застосунками. Kubernetes було розроблено компанією Google на основі їхнього внутрішнього досвіду управління мільярдами контейнерів. Розуміння архітектури Kubernetes є критично важливим для розуміння його безпеки, оскільки кожен компонент має свої специфічні вразливості.

Площина управління (Control Plane)

Це «мозок» кластера, що приймає рішення про розміщення та управління контейнерами:

  • API Server (сервер API): центральний компонент, через який проходять абсолютно всі запити — як від користувачів, так і від внутрішніх компонентів. Це єдина точка входу в кластер
  • etcd: розподілене сховище типу «ключ-значення», де зберігається вся конфігурація кластера, включно з секретами. Це «пам'ять» кластера
  • Scheduler (планувальник): вирішує, на якому робочому вузлі (worker node) запустити кожен под (pod), враховуючи доступні ресурси та обмеження
  • Controller Manager (менеджер контролерів): стежить за бажаним станом кластера і виправляє відхилення — наприклад, автоматично перезапускає контейнери, що впали

Робочі вузли (Worker Nodes)

Це сервери, на яких безпосередньо виконуються контейнери:

  • Kubelet: агент, що працює на кожному вузлі та відповідає за запуск і моніторинг подів (pods) — найменших одиниць розгортання в Kubernetes
  • Середовище виконання контейнерів (Container Runtime): containerd, CRI-O або інше сумісне середовище, яке безпосередньо запускає контейнери
  • Kube-proxy: забезпечує мережеву маршрутизацію та балансування навантаження між подами

Загрози безпеці Kubernetes

1. Відкритий API Server

Сервер API — головна ціль зловмисників. Якщо він доступний з інтернету без належної автентифікації, наслідки можуть бути катастрофічними:

  • Повний контроль над усім кластером — можливість створювати, видаляти та модифікувати будь-які ресурси
  • Можливість запускати довільні контейнери, зокрема з шкідливим кодом
  • Доступ до всіх секретів (паролів, ключів, токенів) та конфігурації кластера

Приклад: у 2018 році зловмисники виявили незахищену панель управління Kubernetes компанії Tesla та використали хмарні ресурси Tesla для майнінгу криптовалют.

2. Компрометація сховища etcd

etcd зберігає всю конфігурацію кластера, у тому числі секрети (Secrets). Якщо зловмисник отримає доступ до etcd, він фактично отримає доступ до всього кластера:

  • За замовчуванням секрети зберігаються лише в кодуванні base64, яке не є шифруванням — будь-хто може легко декодувати їх назад
  • Доступ до etcd рівнозначний доступу до всіх паролів, токенів та ключів API у кластері

3. Проблеми безпеки подів

Поди (pods) — це найменші одиниці розгортання в Kubernetes, і неправильна їх конфігурація створює серйозні ризики:

  • Привілейовані поди (privileged pods) — мають повний доступ до ресурсів хост-системи, фактично обходячи ізоляцію контейнерів
  • Спільне використання просторів імен хоста — параметри hostPID та hostNetwork дають поду доступ до процесів та мережі хост-машини
  • Монтування чутливих томів — підключення критичних каталогів хоста до поду відкриває прямий доступ до системних файлів
  • Запуск від імені root — контейнери, що працюють від імені суперкористувача, становлять більший ризик у разі компрометації

4. Неправильна конфігурація RBAC

RBAC (Role-Based Access Control — контроль доступу на основі ролей) — це основний механізм авторизації в Kubernetes. Його неправильна конфігурація є поширеною проблемою:

  • Занадто широкі права: надання ролі cluster-admin усім користувачам, замість створення обмежених ролей під конкретні завдання
  • Дозволи з підстановкою (wildcard): використання символу * у правах доступу фактично дає необмежений доступ до всіх ресурсів
  • Стандартні облікові записи служб: за замовчуванням кожному поду призначається обліковий запис служби (service account), який може мати надмірні права, якщо адміністратор не подбав про обмеження

5. Актуальні вразливості Kubernetes (2024–2025)

Безпека Kubernetes продовжує бути критичною темою — нові серйозні вразливості з'являються щороку:

  • CVE-2025-1974 «IngressNightmare» (березень 2025): критична вразливість (CVSS 9.8) у контролері Ingress NGINX для Kubernetes. Зловмисник без автентифікації може надіслати спеціально сформований запит до компонента Admission Controller, що призводить до віддаленого виконання коду (RCE) у поді контролера. Оскільки контролер Ingress зазвичай має доступ до секретів усіх просторів імен кластера, експлуатація може призвести до повного захоплення кластера. За оцінками Wiz Research, близько 43% хмарних середовищ були вразливі, включно з кластерами компаній зі списку Fortune 500. Виправлено у версіях Ingress NGINX v1.12.1 та v1.11.5
  • CVE-2024-10220 (2024): критична вразливість у застарілому механізмі томів gitRepo в Kubernetes. Зловмисник, який має право створювати поди з використанням тому gitRepo, може виконати довільні команди на робочому вузлі (node) з правами root. Хоча gitRepo є застарілим (deprecated) механізмом, він все ще доступний у багатьох кластерах за замовчуванням

Механізми безпеки Kubernetes

RBAC — контроль доступу на основі ролей

RBAC (Role-Based Access Control) визначає, хто (користувач або сервіс) може виконувати які дії (читати, створювати, видаляти) над якими ресурсами (подами, секретами, сервісами) у кластері. RBAC працює за принципом «за замовчуванням заборонено» — якщо право не надано явно, дія буде заблокована.

RBAC складається з двох частин: Role (роль) визначає набір дозволів, а RoleBinding (прив'язка ролі) призначає цю роль конкретному користувачу або групі:

# Role — визначає ПРАВА в межах конкретного namespace
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: production    # Ця роль діє ЛИШЕ в namespace "production"
  name: pod-reader         # Назва ролі
rules:
- apiGroups: [""]          # "" = core API group (pods, services, secrets)
  resources: ["pods"]      # Ресурси, до яких надається доступ
  verbs: ["get", "list", "watch"]  # Дозволені дії: читати, переглядати список, спостерігати за змінами
  # ЗВЕРНІТЬ УВАГУ: немає "create", "update", "delete" — лише читання!

---
# RoleBinding — ПРИВ'ЯЗУЄ роль до конкретного користувача
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: production    # Прив'язка теж діє лише в цьому namespace
subjects:
- kind: User
  name: developer          # Користувач "developer" отримує цю роль
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: pod-reader         # Посилання на роль, визначену вище
  apiGroup: rbac.authorization.k8s.io

У цьому прикладі користувач «developer» може лише переглядати поди в namespace «production» — він не може їх створювати, видаляти чи модифікувати, і не має доступу до подів в інших namespace. Якщо потрібні глобальні права (на весь кластер), використовуються ClusterRole та ClusterRoleBinding, але їх слід застосовувати з великою обережністю.

Найкращі практики RBAC:

  • Принцип найменших привілеїв: надавайте лише ті права, які мінімально необхідні для виконання завдання
  • Ізоляція через простори імен (namespaces): розмежовуйте доступ різних команд та середовищ (розробка, тестування, продуктивне)
  • Уникайте ClusterRole з широкими правами: використовуйте Role (обмежену простором імен), а не ClusterRole (глобальну), де це можливо
  • Регулярний аудит прав: періодично перевіряйте, хто має які права, та відкликайте непотрібні

Мережеві політики (Network Policies)

Мережеві політики контролюють, які поди можуть спілкуватися між собою по мережі. За замовчуванням у Kubernetes усі поди можуть вільно обмінюватися трафіком — це зручно для розробки, але небезпечно для продуктивного середовища. Мережеві політики дозволяють обмежити цей доступ за принципом «заборонити все, дозволити лише необхідне».

Типовий підхід — спочатку заборонити весь вхідний трафік, а потім додати виключення лише для необхідних з'єднань:

# КРОК 1: Заборонити ВЕСЬ вхідний трафік у namespace "production"
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all-ingress
  namespace: production
spec:
  podSelector: {}      # Порожній селектор = політика застосовується до ВСІХ подів
  policyTypes:
  - Ingress            # Забороняємо вхідний трафік (ingress)
  # Немає розділу "ingress:" — це означає "жоден вхідний трафік не дозволений"

---
# КРОК 2: Дозволити КОНКРЕТНЕ з'єднання: frontend → backend на порт 8080
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: backend        # Ця політика застосовується до подів з міткою app=backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend   # Дозволяємо вхідний трафік ЛИШЕ від подів з міткою app=frontend
    ports:
    - protocol: TCP
      port: 8080          # І лише на порт 8080

Після застосування цих двох політик: поди backend приймають з'єднання лише від подів frontend і лише на порт 8080. Будь-яка інша комунікація в namespace «production» заблокована. Це означає, що навіть якщо зловмисник скомпрометує один під, він не зможе вільно переміщуватися по мережі кластера (так зване «бокове переміщення» або lateral movement).

Важливо: мережеві політики потребують мережевого плагіна (CNI plugin), який їх підтримує, — наприклад, Calico, Cilium або Weave Net. Стандартний плагін Kubernetes не забезпечує їх виконання, тому без правильного плагіна політики просто ігноруватимуться.

Стандарти безпеки подів (Pod Security Standards, PSS)

Стандарти безпеки подів визначають три рівні обмежень, які можна застосувати до простору імен (namespace):

  • Privileged (привілейований): без обмежень — для системних компонентів, які потребують повного доступу до ресурсів хоста (наприклад, мережеві плагіни, агенти моніторингу)
  • Baseline (базовий): мінімальні обмеження, що запобігають найочевиднішим векторам підвищення привілеїв — підходить для більшості некритичних застосунків
  • Restricted (обмежений): суворі обмеження, що відповідають найкращим практикам безпеки — рекомендується для всіх продуктивних (production) середовищ

Стандарти безпеки подів застосовуються через мітки (labels) на рівні namespace. Це дозволяє задати різну поведінку для порушень:

apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    # enforce — БЛОКУВАТИ поди, що порушують стандарт
    pod-security.kubernetes.io/enforce: restricted
    # audit — ЗАПИСАТИ порушення в журнал аудиту (але не блокувати)
    pod-security.kubernetes.io/audit: restricted
    # warn — ПОПЕРЕДИТИ користувача при створенні (але дозволити)
    pod-security.kubernetes.io/warn: restricted

Три режими дозволяють поступово впроваджувати обмеження: спочатку увімкнути лише warn, щоб побачити, які поди порушують стандарт, потім audit для запису в журнал, і нарешті enforce для повного блокування. Такий поетапний підхід запобігає раптовому виходу з ладу застосунків, які ще не адаптовані до суворих вимог.

Управління секретами в Kubernetes

За замовчуванням секрети в Kubernetes зберігаються у сховищі etcd у кодуванні base64, яке не є шифруванням — це лише спосіб представлення даних, і будь-хто з доступом до etcd може їх декодувати. Для реального захисту необхідно увімкнути шифрування секретів у стані спокою:

# Конфігурація шифрування секретів у сховищі etcd
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
      - secrets                # Шифрувати ресурси типу "secrets"
    providers:
      - aescbc:                # Алгоритм шифрування AES-CBC
          keys:
            - name: key1
              secret: <base64-encoded-key>   # Ключ шифрування (має бути згенерований безпечно)
      - identity: {}           # Фолбек: якщо дані не зашифровані — читати як є
                               # Це потрібно для читання старих секретів, створених до увімкнення шифрування

Після застосування цієї конфігурації всі нові секрети будуть зашифровані алгоритмом AES-CBC перед записом у etcd. Для шифрування наявних секретів потрібно виконати команду kubectl get secrets --all-namespaces -o json | kubectl replace -f -, яка перезапише їх уже в зашифрованому вигляді.

Найкращі практики:

  • Шифрування у стані спокою (encryption at rest): обов'язково увімкніть шифрування секретів у сховищі etcd
  • Зовнішнє управління секретами: використовуйте спеціалізовані інструменти — HashiCorp Vault, Sealed Secrets або External Secrets Operator — для безпечного зберігання та автоматичної синхронізації секретів
  • Не зберігайте секрети в Git: навіть у зашифрованому вигляді це ризиковано, оскільки ключ шифрування може бути скомпрометований
  • Регулярна ротація секретів: періодично змінюйте паролі, токени та ключі для мінімізації наслідків потенційного витоку

Контролери допуску (Admission Controllers)

Контролери допуску перехоплюють запити до API Server ще до їх виконання і можуть перевірити (валідувати) або навіть модифікувати їх. Це потужний механізм для автоматичного застосування політик безпеки — наприклад, заборона створення привілейованих подів або обов'язкове додавання лімітів ресурсів:

  • PodSecurity: вбудований контролер для застосування стандартів безпеки подів (PSS)
  • OPA Gatekeeper: дозволяє описувати політики безпеки як код (policy-as-code) мовою Rego
  • Kyverno: рідний для Kubernetes механізм політик, який використовує звичний YAML-формат, що простіший для розуміння

Контрольний список безпеки Kubernetes

  • Сервер API доступний лише з приватної мережі, з обов'язковою автентифікацією
  • RBAC увімкнено, права доступу мінімальні відповідно до принципу найменших привілеїв
  • Мережеві політики (Network Policies) налаштовані для всіх просторів імен
  • Стандарти безпеки подів: рівень Restricted для продуктивного середовища
  • Сховище etcd зашифроване та ізольоване від робочих вузлів
  • Секрети зашифровані у стані спокою (encryption at rest)
  • Журналювання аудиту (audit logging) увімкнено для відстеження всіх дій у кластері
  • Образи контейнерів автоматично скануються на вразливості перед розгортанням
  • Облікові записи служб (service accounts) мають лише мінімально необхідні права

Реальні інциденти безпеки Kubernetes

Криптомайнінг у Tesla (2018)

Зловмисники виявили незахищену панель управління (dashboard) Kubernetes компанії Tesla, доступну з інтернету без жодної автентифікації:

  • Вектор атаки: публічно доступна панель управління Kubernetes без пароля — будь-хто міг зайти та керувати кластером
  • Експлуатація: зловмисники запустили поди для майнінгу криптовалют, маскуючи свою активність шляхом обмеження споживання ресурсів
  • Наслідки: використання хмарних ресурсів AWS компанії Tesla для майнінгу за рахунок Tesla
  • Урок: ніколи не відкривайте панель управління Kubernetes без автентифікації; обмежуйте доступ до API лише з внутрішньої мережі

Shopify Bug Bounty (2020)

Дослідник безпеки в межах програми винагород за вразливості (bug bounty) знайшов спосіб отримати доступ до метаданих Google Cloud через Kubernetes:

  • Вектор: SSRF-атака (Server-Side Request Forgery — підробка серверного запиту) — зловмисник змушує сервер (у цьому випадку — под) зробити запит до внутрішнього сервісу метаданих хмарного провайдера (IP-адреса 169.254.169.254), який доступний лише зсередини хмари
  • Отримані дані: облікові дані сервісного акаунта (service account credentials)
  • Вплив: потенційний доступ до ресурсів GCP Shopify
  • Захист: використання Workload Identity (прив'язка подів до конкретних сервісних акаунтів) та приховування сервісу метаданих (metadata concealment)

Приклад безпечного маніфесту поду

Нижче наведено маніфест поду, який реалізує всі основні рекомендації безпеки. Розглянемо кожен елемент:

apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:              # Контекст безпеки на рівні ВСЬОГО поду
    runAsNonRoot: true          # Заборона запуску від root — K8s відхилить под, якщо контейнер спробує запуститися від root
    runAsUser: 10000            # Конкретний UID (ідентифікатор користувача) — не 0 (root)
    runAsGroup: 10000           # Конкретний GID (ідентифікатор групи)
    fsGroup: 10000              # Група для доступу до змонтованих томів
    seccompProfile:
      type: RuntimeDefault      # Seccomp-профіль за замовчуванням — блокує небезпечні системні виклики
  containers:
  - name: app
    # Образ з приватного реєстру, конкретна версія + хеш SHA256
    # Хеш гарантує, що навіть якщо тег v1.2.3 буде переписано, ви отримаєте саме той образ
    image: myregistry/myapp:v1.2.3@sha256:abc123...
    securityContext:             # Контекст безпеки на рівні КОНТЕЙНЕРА
      allowPrivilegeEscalation: false   # Заборона підвищення привілеїв (SUID/SGID)
      readOnlyRootFilesystem: true      # Файлова система лише для читання
      capabilities:
        drop:
          - ALL                  # Скидання ВСІХ Linux capabilities (додаткових привілеїв)
    resources:
      limits:                    # Максимум ресурсів — контейнер буде зупинено при перевищенні
        memory: "128Mi"
        cpu: "500m"              # 500 мілі-CPU = 0.5 ядра
      requests:                  # Мінімум гарантованих ресурсів
        memory: "64Mi"
        cpu: "250m"
    livenessProbe:               # Перевірка працездатності — K8s перезапустить контейнер, якщо він не відповідає
      httpGet:
        path: /health
        port: 8080
      initialDelaySeconds: 3     # Зачекати 3 сек. перед першою перевіркою
      periodSeconds: 10          # Перевіряти кожні 10 секунд
  automountServiceAccountToken: false  # НЕ монтувати токен сервісного акаунта — якщо поду не потрібен доступ до API K8s, токен не слід надавати

Параметр automountServiceAccountToken: false заслуговує на окрему увагу: за замовчуванням кожен під отримує токен, який дозволяє йому звертатися до API Kubernetes. Якщо зловмисник скомпрометує контейнер, він зможе використати цей токен для розвідки кластера або ескалації привілеїв. Вимикаючи автоматичне монтування, ми прибираємо цей вектор атаки.

4.4 CWPP та захист під час виконання

Що таке CWPP?

CWPP (Cloud Workload Protection Platform — платформа захисту хмарних робочих навантажень) — це категорія рішень безпеки, визначена аналітичною компанією Gartner, яка зосереджена на захисті серверних робочих навантажень (workloads) у хмарних, гібридних та локальних (on-premises) середовищах. Під «робочим навантаженням» розуміють будь-який обчислювальний процес — віртуальну машину, контейнер, серверлес-функцію або навіть фізичний сервер.

CWPP еволюціонувала з традиційних рішень для захисту кінцевих точок (endpoint protection), адаптувавшись до специфіки хмарних та контейнерних середовищ, де робочі навантаження є ефемерними (тимчасовими), динамічними та розподіленими.

Ключові можливості CWPP

1. Управління вразливостями (Vulnerability Management)

  • Автоматичне сканування образів контейнерів на відомі вразливості (CVE)
  • Сканування віртуальних машин та хостів на наявність вразливого програмного забезпечення
  • Пріоритизація вразливостей з урахуванням можливості їх експлуатації та контексту (наскільки ресурс доступний ззовні, чи є публічний експлойт)
  • Інтеграція з конвеєрами CI/CD для раннього виявлення проблем (підхід «зміщення вліво» — shift left)

2. Захист під час виконання (Runtime Protection)

  • Виявлення аномальної поведінки процесів — наприклад, запуск shell у контейнері, де це не передбачено
  • Захист від шкідливого ПЗ та програм для прихованого майнінгу криптовалют (cryptominers)
  • Моніторинг цілісності файлів (file integrity monitoring) — виявлення несанкціонованих змін у критичних файлах
  • Виявлення мережевих аномалій — нетипові з'єднання, спроби зв'язку з підозрілими IP-адресами

3. Відповідність та конфігурація (Compliance & Configuration)

  • Перевірка конфігурації на відповідність еталонним стандартам (CIS Benchmarks)
  • Звітність про відповідність вимогам регуляторів (PCI DSS для платіжних систем, HIPAA для медичних даних, SOC 2 для хмарних сервісів)
  • Сканування інфраструктури як коду (Infrastructure as Code, IaC) — перевірка Terraform, CloudFormation та інших шаблонів ще до розгортання

4. Мережева безпека

  • Візуалізація мережевих потоків — наочне відображення, які сервіси з якими обмінюються даними
  • Мікросегментація — розділення мережі на дрібні сегменти для обмеження бокового переміщення зловмисника (lateral movement)
  • Політики міжмережевого екрана — автоматичне створення та застосування правил фільтрації трафіку

Захист під час виконання (Runtime Security)

Захист під час виконання зосереджений на виявленні та блокуванні загроз у момент, коли системи вже працюють. Це критично важливий рівень захисту, оскільки жодне попереднє сканування не може гарантувати 100% безпеку — нові вразливості з'являються щодня, а зловмисники постійно вдосконалюють свої методи.

Підходи до захисту під час виконання

1. Моніторинг системних викликів (System Call Monitoring)

Системні виклики (syscalls) — це інтерфейс між застосунком та ядром ОС. Моніторинг цих викликів дозволяє виявити підозрілу активність, яка може свідчити про компрометацію:

  • Несподіване виконання процесів — наприклад, запуск зворотної оболонки (reverse shell) зсередини контейнера веб-сервера
  • Аномальний доступ до файлів — спроби прочитати /etc/shadow або записати файли в системні каталоги
  • Нетипові мережеві з'єднання — вихідні підключення до невідомих IP-адрес або нестандартних портів

Інструменти:

  • Falco: інструмент з відкритим кодом для виявлення загроз під час виконання від Sysdig, проєкт CNCF
  • Tetragon: моніторинг безпеки на основі eBPF від Cilium
  • Tracee: захист під час виконання на основі eBPF від Aqua Security

Приклад правила Falco:

- rule: Terminal shell in container
  desc: Detect a shell opened in a container   # Опис правила
  condition: >                                  # Умова спрацювання:
    spawned_process and container and           #   новий процес у контейнері
    shell_procs and proc.tty != 0              #   це shell (bash, sh) з підключеним терміналом
  output: >                                     # Що записати в журнал при спрацюванні:
    Shell opened in container
    (user=%user.name container=%container.name shell=%proc.name)
  priority: WARNING                             # Рівень критичності
  tags: [container, shell]                      # Теги для класифікації

Це правило спрацює, коли хтось відкриє інтерактивну командну оболонку (bash, sh) всередині контейнера — наприклад, через kubectl exec -it pod-name -- /bin/bash. У продуктивному середовищі це майже завжди аномальна подія: у нормальних умовах ніхто не повинен вручну заходити в контейнер. Якщо таке стається о 3:00 ночі без зареєстрованого інциденту — це потенційний сигнал компрометації, який потребує негайного розслідування.

2. Безпека на основі eBPF

eBPF (extended Berkeley Packet Filter) — це сучасна технологія ядра Linux, яка дає змогу запускати ізольовані (sandboxed) програми безпосередньо в ядрі без необхідності його модифікувати або завантажувати модулі ядра. Це революційний підхід до моніторингу безпеки:

  • Низькі накладні витрати: програми eBPF виконуються з мінімальним впливом на продуктивність (зазвичай менше 1%)
  • Глибока видимість: доступ до подій на рівні ядра — мережевих пакетів, системних викликів, файлових операцій
  • Можливість активного блокування: не лише моніторинг, а й примусове застосування правил безпеки (enforcement) — наприклад, заборона певних системних викликів у реальному часі

3. Підпис та верифікація образів (Image Signing & Verification)

Механізм, який гарантує, що в кластері запускаються лише перевірені та довірені образи контейнерів. Це запобігає підміні образів або запуску несанкціонованого коду:

  • Sigstore/Cosign: відкриті інструменти для криптографічного підпису та верифікації образів контейнерів
  • Контролер допуску Kubernetes: автоматичне блокування розгортання образів, які не мають дійсного підпису
  • Binary Authorization (GCP): вбудований механізм Google Cloud для перевірки образів перед запуском — інтегрований з Artifact Registry та Cloud Build

Комерційні рішення

  • Palo Alto Prisma Cloud: повнофункціональна платформа CNAPP
  • CrowdStrike Falcon: поєднання EDR (виявлення та реагування на кінцевих точках) та CWPP

  • Sysdig: моніторинг та безпека в єдиному рішенні
  • Wiz: безагентна платформа CNAPP (не потребує встановлення агентів на сервери)

Рішення з відкритим кодом

  • Trivy: сканування образів контейнерів та IaC на вразливості
  • Checkov: сканування інфраструктури як коду (Terraform, CloudFormation, Kubernetes)

CWPP у хмарних провайдерах

Кожен великий хмарний провайдер пропонує власні CWPP-інструменти. Перевага вбудованих рішень — глибока інтеграція з іншими сервісами провайдера та простота налаштування. Недолік — прив'язка до конкретного постачальника (vendor lock-in) та обмежена підтримка мультихмарних сценаріїв.

AWS

AWS пропонує набір окремих сервісів, які разом формують CWPP-стек:

  • Amazon Inspector: автоматичне сканування EC2-інстансів, Lambda-функцій та образів контейнерів на вразливості. Інтегрується з ECR (Elastic Container Registry) — сканує образи автоматично при завантаженні. Пріоритизує вразливості на основі оцінки CVSS та доступності ресурсу з мережі (network reachability)
  • Amazon GuardDuty: виявлення загроз на базі машинного навчання. Для EKS аналізує журнали аудиту Kubernetes та виявляє: підозрілі API-виклики, спроби підвищення привілеїв, прихований майнінг криптовалют, витік даних через DNS
  • AWS Security Hub: централізована панель керування безпекою, що збирає виявлені проблеми з Inspector, GuardDuty, Macie та IAM Access Analyzer. Підтримує автоматичне реагування (remediation) через EventBridge
  • Amazon ECR Image Scanning: базове та розширене (на основі Inspector) сканування образів контейнерів

Azure

Microsoft пропонує найбільш інтегроване CWPP-рішення серед трьох провайдерів:

  • Microsoft Defender for Cloud: поєднує CSPM (управління безпековою позицією) та CWPP в одному рішенні. Покриває віртуальні машини, контейнери, безсерверні функції, бази даних. Також підтримує мультихмарні середовища (AWS, GCP) через Azure Arc
  • Defender for Containers: спеціалізований модуль для AKS (Azure Kubernetes Service) та EKS: сканування образів на вразливості, виявлення загроз під час виконання, управління безпековою позицією Kubernetes, рекомендації щодо контролю допуску
  • Defender for Servers: рішення EDR (виявлення та реагування на кінцевих точках) для віртуальних машин на базі Microsoft Defender for Endpoint. Включає оцінку вразливостей, моніторинг цілісності файлів та адаптивний контроль застосунків

GCP

GCP робить акцент на безпеці за замовчуванням (security-by-default) та інтеграції з Kubernetes (Google створив Kubernetes):

  • Security Command Center (SCC): централізоване управління безпекою. Преміум-рівень включає виявлення загроз у контейнерах (Container Threat Detection), сканер веб-безпеки (Web Security Scanner) та виявлення подій безпеки (Event Threat Detection)
  • Container Threat Detection: захист під час виконання, що виявляє підозрілий запуск програм, зворотні оболонки (reverse shells) та витік даних із подів GKE
  • Binary Authorization: контроль допуску, що дозволяє запуск лише криптографічно підписаних образів. Інтегрується з Artifact Registry та Cloud Build для наскрізної безпеки ланцюга постачання
  • Artifact Analysis: сканування образів контейнерів на вразливості в Artifact Registry. Автоматичне сканування при завантаженні образу

4.5 Безпека ланцюга постачання програмного забезпечення для контейнерів

Безпека ланцюга постачання (supply chain security) — одна з найактуальніших тем у хмарній безпеці. Сучасний контейнерний застосунок складається з десятків шарів: базовий образ операційної системи, системні бібліотеки, середовище виконання мови програмування, фреймворки, бібліотеки сторонніх розробників і, нарешті, ваш власний код. Компрометація будь-якого з цих шарів означає компрометацію всього застосунку. За аналогією з фізичним ланцюгом постачання: якщо один з постачальників комплектуючих підмішає шкідливий елемент, кінцевий продукт буде скомпрометовано.

Чому безпека ланцюга постачання є критичною?

Низка резонансних інцидентів змусила індустрію кардинально переглянути підхід до безпеки ланцюга постачання:

  • SolarWinds (2020): зловмисники впровадили шкідливий код у процес збірки оновлень для платформи моніторингу SolarWinds Orion. Скомпрометоване оновлення було встановлено понад 18 000 організаціями, зокрема Міністерством фінансів та Міністерством внутрішньої безпеки США. Цей інцидент вважається одним із найсерйозніших кіберінцидентів в історії.
  • Log4Shell (2021): критична вразливість CVE-2021-44228 у бібліотеці Log4j — компоненті журналювання для Java, вбудованому в мільйони застосунків та контейнерів по всьому світу. Організації тижнями не могли знайти всі місця, де використовувався Log4j, що наочно продемонструвало потребу в SBOM.
  • Codecov (2021): зловмисники модифікували скрипт завантаження для системи аналізу покриття коду Codecov, отримавши доступ до секретів конвеєрів CI/CD тисяч компаній.
  • Шкідливі образи на Docker Hub: регулярно виявляються образи з вбудованими програмами для майнінгу криптовалют або прихованими входами (backdoors), які маскуються під легітимні популярні образи.

SBOM — перелік компонентів програмного забезпечення

SBOM (Software Bill of Materials — специфікація матеріалів програмного забезпечення) — це повний структурований перелік усіх компонентів, з яких складається ваш програмний продукт: бібліотеки, фреймворки, утиліти, їхні версії та ліцензії. Це аналог списку інгредієнтів на харчовому пакуванні, але для програмного забезпечення.

Навіщо потрібен SBOM:

  • Швидке реагування на вразливості: коли з'являється нова CVE (як Log4Shell), SBOM дає змогу миттєво знайти всі уражені компоненти в усіх ваших застосунках, замість тижнів ручного пошуку
  • Відповідність ліцензійним вимогам: точно знати, які ліцензії відкритого коду використовуються та які зобов'язання з них випливають
  • Регуляторні вимоги: указ Президента США № 14028 (2021) вимагає надання SBOM від усіх постачальників програмного забезпечення для федерального уряду

Формати SBOM:

  • SPDX (Software Package Data Exchange): стандарт Linux Foundation, ISO/IEC 5962:2021
  • CycloneDX: стандарт OWASP, фокус на безпеці

Інструменти генерації SBOM:

# Генерація SBOM для Docker-образу за допомогою Syft
# Syft аналізує всі шари образу та створює перелік у форматі SPDX (JSON)
$ syft myimage:latest -o spdx-json > sbom.json

# Альтернатива: генерація SBOM за допомогою Trivy
$ trivy image --format spdx-json myimage:latest > sbom.json

# Перевірка згенерованого SBOM на відомі вразливості за допомогою Grype
# Grype порівнює кожен компонент зі SBOM з базами даних вразливостей (NVD, GitHub Advisory)
$ grype sbom:sbom.json

На практиці ці команди вбудовуються в конвеєр CI/CD: при кожній збірці образу автоматично генерується SBOM, перевіряється на вразливості, і якщо знайдено критичні проблеми — збірка зупиняється. SBOM також зберігається як артефакт збірки для подальшого аудиту та швидкого реагування на нові вразливості.

SLSA — рівні безпеки ланцюга постачання для програмних артефактів

SLSA (Supply-chain Levels for Software Artifacts, вимовляється «сальса») — це фреймворк (набір рекомендацій), розроблений Google, для забезпечення цілісності ланцюга постачання програмного забезпечення. Він визначає чотири рівні зрілості, де кожен наступний рівень забезпечує більш надійний захист:

Рівень Вимоги Захищає від
SLSA 1 Задокументований процес збірки, автоматична генерація провенансу Незадокументовані зміни — ви хоча б знаєте, як і де збирається ваш код
SLSA 2 Система контролю версій, зовнішній сервіс збірки, автентифікований провенанс Модифікація артефактів після збірки — підпис підтверджує цілісність
SLSA 3 Захищена платформа збірки, провенанс, який неможливо підробити Компрометація процесу збірки — навіть якщо зловмисник отримає доступ до CI/CD, він не зможе підробити провенанс
SLSA 4 Перевірка двома особами, герметичні збірки (без доступу до мережі), відтворюваність Внутрішні загрози та складні атаки — навіть інсайдер не зможе непомітно модифікувати код

Для більшості організацій рекомендується прагнути до рівня SLSA 3 для продуктивних систем. Рівень SLSA 4 наразі досягнутий лише кількома великими компаніями (Google, Microsoft), оскільки вимагає значних інвестицій в інфраструктуру збірки.

Провенанс (provenance — походження) — ключове поняття SLSA. Це криптографічно підписаний документ, який відповідає на три фундаментальні питання: що саме було зібрано? з якого вихідного коду? на якій платформі збірки? Провенанс дозволяє довести, що конкретний артефакт (образ контейнера, бінарний файл) був створений саме з конкретного вихідного коду без сторонніх модифікацій.

Sigstore — підпис та верифікація програмних артефактів

Sigstore — це проєкт з відкритим кодом під егідою Linux Foundation, призначений для підписування, верифікації та захисту ланцюга постачання програмного забезпечення. Sigstore робить криптографічний підпис артефактів таким же простим, як підпис коміту в Git. Він складається з трьох основних компонентів:

  • Cosign: інструмент для підписування та верифікації образів контейнерів та інших OCI-артефактів (стандарт Open Container Initiative)
  • Fulcio: центр сертифікації для підпису без ключів (keyless signing) — використовує ідентифікатор OIDC (наприклад, ваш обліковий запис Google або GitHub) замість довготривалих криптографічних ключів, які потрібно зберігати та захищати
  • Rekor: публічний, незмінний реєстр (transparency log) усіх підписів — будь-хто може перевірити, коли і ким було підписано конкретний артефакт

Приклад підпису та верифікації образу:

# Підпис образу (keyless — автентифікація через Google/GitHub/Microsoft акаунт)
# Cosign відкриє браузер для підтвердження особи, після чого підпише образ
$ cosign sign myregistry.io/myimage:v1.0

# Верифікація підпису — перевіряє, що образ не було змінено після підпису
# і що його підписала саме та особа/сервіс, якій ми довіряємо
$ cosign verify myregistry.io/myimage:v1.0

    match:
      any:
      - resources:
          kinds:
          - Pod
    verifyImages:
    - imageReferences:
      - "myregistry.io/*"
      attestors:
      - entries:
        - keyless:
            issuer: "https://accounts.google.com"
            subject: "builder@mycompany.iam.gserviceaccount.com"

Ця політика Kyverno автоматично перевіряє кожен под перед запуском: якщо образ контейнера з реєстру myregistry.io не має дійсного криптографічного підпису від сервісного акаунта builder@mycompany — розгортання буде заблоковано. Це гарантує, що в кластері працюють лише ті образи, які пройшли офіційний процес збірки, а не створені вручну або модифіковані зловмисником.

Повний конвеєр безпеки ланцюга постачання

Комплексний підхід до безпеки ланцюга постачання передбачає захист на кожному етапі життєвого циклу програмного забезпечення — від написання коду до роботи в продуктивному середовищі:

  1. Вихідний код (Source): підписані коміти (git commit signing), правила захисту гілок (branch protection rules), обов'язковий перегляд коду (code review) — гарантують, що жодна зміна не потрапить до коду без перевірки
  2. Залежності (Dependencies): аналіз складу програмного забезпечення (SCA — Software Composition Analysis) за допомогою Snyk, Dependabot або Renovate. Lock-файли для фіксації точних версій залежностей — запобігають несанкціонованим оновленням
  3. Збірка (Build): захищені конвеєри CI/CD з ізольованими середовищами виконання (runners), без секретів у журналах. Генерація провенансу за стандартом SLSA
  4. Пакування (Package): сканування образів на вразливості (Trivy, Grype), генерація SBOM, криптографічний підпис образу (Cosign)
  5. Розгортання (Deploy): контроль допуску (Kyverno, OPA Gatekeeper) — автоматична перевірка підпису образу та відповідності політикам безпеки перед запуском
  6. Виконання (Runtime): безперервний моніторинг (Falco), виявлення відхилень (drift detection — коли працюючий контейнер відрізняється від еталону), повторне сканування образів на нові вразливості

Висновки

Віртуалізація та контейнеризація є основою сучасних хмарних обчислень. Водночас ці технології створюють специфічні безпекові виклики, які потребують системного підходу та глибокого розуміння.

Ключові тези

  • Гіпервізор — критичний компонент безпеки: його компрометація впливає на всі віртуальні машини на сервері. Втеча з VM — рідкісна, але катастрофічна за наслідками атака.
  • Контейнери — не віртуальні машини: спільне ядро означає слабшу ізоляцію. Привілейований контейнер фактично надає повний контроль над хост-системою.
  • Неправильна конфігурація — головний ворог: більшість інцидентів безпеки спричинені помилками налаштувань, а не вразливостями нульового дня (zero-day).
  • Глибокий захист (defense in depth): жоден окремий механізм не забезпечує повної безпеки — потрібно кілька рівнів: сканування образів, контроль допуску, моніторинг під час виконання, мережеві політики.
  • Зміщення вліво (shift left): виявляйте проблеми безпеки на етапі розробки, а не в продуктивному середовищі. Використовуйте SBOM та SLSA для контролю ланцюга постачання.
  • Ланцюг постачання — новий критичний вектор: ваш застосунок захищений рівно настільки, наскільки захищений найслабший компонент у його ланцюзі залежностей.

Практичні рекомендації

  1. Використовуйте мінімальні базові образи (Alpine, Distroless) для зменшення поверхні атаки
  2. Автоматично скануйте образи на вразливості в конвеєрі CI/CD перед кожним розгортанням
  3. Запускайте контейнери від непривілейованого користувача (non-root)
  4. Застосовуйте мережеві політики (Network Policies) у Kubernetes для обмеження трафіку між подами
  5. Увімкніть стандарти безпеки подів (Pod Security Standards) рівня Restricted для продуктивних середовищ
  6. Шифруйте секрети у стані спокою (encryption at rest) у сховищі etcd
  7. Впровадьте моніторинг під час виконання (наприклад, Falco) для виявлення підозрілої активності в реальному часі

  5. Як працює контроль доступу на основі ролей (RBAC) у Kubernetes? Чим відрізняються Role та ClusterRole?

Практичне завдання

Ваша команда розробляє мікросервісний застосунок на Kubernetes, що складається з чотирьох сервісів: frontend (React), API-шлюз (Node.js), backend (Python), база даних (PostgreSQL).

Завдання:

  1. Зміцнення Dockerfile: Напишіть безпечний Dockerfile для Python backend-сервісу. Використайте багатоетапну збірку (multi-stage build), непривілейованого користувача, мінімальний базовий образ, перевірку працездатності (health check). Поясніть кожне рішення з точки зору безпеки.
  2. Безпека Kubernetes: Створіть набір маніфестів Kubernetes:
    • Мережева політика (NetworkPolicy) — frontend може звертатися лише до API-шлюзу; API-шлюз — лише до backend; backend — лише до бази даних
    • Контекст безпеки поду (Pod Security Context) для backend — рівень Restricted
    • RBAC — роль «developer» може читати поди та журнали, але не може виконувати команди всередині контейнерів (exec)
  3. Конвеєр безпеки CI/CD: Спроєктуйте конвеєр для GitHub Actions, що включає: статичний аналіз коду (Semgrep), сканування образів (Trivy), генерацію SBOM (Syft), підпис образів (Cosign), розгортання з контролем допуску (Kyverno).
  4. Реагування на інцидент: Falco виявив сповіщення «shell opened in container» (відкриття командної оболонки в контейнері) для поду backend о 3:00 ночі. Опишіть покрокову процедуру реагування на цей інцидент.

Глосарій ключових термінів

  • Гіпервізор (Hypervisor) — програмне забезпечення для управління віртуальними машинами. Тип 1 (bare-metal, апаратний): KVM, ESXi, Xen — встановлюється безпосередньо на апаратуру. Тип 2 (hosted, програмний): VirtualBox, VMware Workstation — працює поверх звичайної ОС
  • Втеча з VM (VM Escape) — атака, під час якої зловмисник виходить із віртуальної машини на рівень гіпервізора або хост-системи, отримуючи контроль над усіма VM на сервері
  • Простір імен (Namespace) — механізм ізоляції ядра Linux, що забезпечує ізольований вигляд системних ресурсів (процесів, мережі, файлової системи, користувачів) для кожного контейнера
  • Контрольна група (Cgroup, Control Group) — механізм ядра Linux для обмеження та обліку ресурсів (процесор, оперативна пам'ять, введення-виведення) для групи процесів
  • Втеча з контейнера (Container Escape) — вихід зловмисника з контейнера на хост-систему через вразливість у середовищі виконання (runc, containerd) або неправильну конфігурацію
  • Distroless — мінімальні образи контейнерів від Google, що містять лише застосунок та його залежності виконання (без командної оболонки та пакетного менеджера)
  • RBAC (Role-Based Access Control) — контроль доступу на основі ролей — основний механізм авторизації в Kubernetes, що визначає, хто може виконувати які дії над якими ресурсами
  • Стандарти безпеки подів (Pod Security Standards) — три рівні обмежень Kubernetes: Privileged (без обмежень), Baseline (мінімальні обмеження), Restricted (максимальні обмеження)
  • Мережева політика (NetworkPolicy) — ресурс Kubernetes для контролю мережевого трафіку між подами, що забезпечує мікросегментацію мережі
  • CWPP (Cloud Workload Protection Platform) — платформа захисту хмарних робочих навантажень — категорія рішень для комплексного захисту серверних навантажень у хмарі
  • eBPF (extended Berkeley Packet Filter) — технологія для запуску ізольованих програм у ядрі Linux без його модифікації, що забезпечує глибокий моніторинг з мінімальними накладними витратами
  • Falco — інструмент з відкритим кодом для виявлення загроз під час виконання в контейнерах та Kubernetes (проєкт CNCF)
  • SBOM (Software Bill of Materials) — специфікація матеріалів програмного забезпечення — повний структурований перелік компонентів програмного продукту з версіями та ліцензіями
  • SLSA (Supply-chain Levels for Software Artifacts) — рівні безпеки ланцюга постачання для програмних артефактів — фреймворк від Google для забезпечення цілісності ланцюга постачання (4 рівні)
  • Sigstore — інфраструктура з відкритим кодом для криптографічного підпису та верифікації програмних артефактів (складається з компонентів: Cosign, Fulcio, Rekor)
  • OCI (Open Container Initiative) — Ініціатива відкритих контейнерів — стандарт, що визначає формат образів контейнерів та середовищ їх виконання
  • Контролер допуску (Admission Controller) — компонент Kubernetes, що перехоплює запити до сервера API та може дозволити, модифікувати або заблокувати їх на основі заданих правил
  • Сервісна сітка (Service Mesh) — інфраструктурний рівень для міжсервісної комунікації із вбудованим взаємним шифруванням (mTLS), спостережуваністю (observability) та управлінням трафіком (Istio, Linkerd)
