Тема 4.3: Безпека Kubernetes

Архітектура Kubernetes

Загрози Kubernetes

  • Відкритий API Server: повний контроль над кластером
  • Скомпрометований etcd: доступ до всіх секретів
  • Неправильна конфігурація подів: привілейований режим, hostPID
  • Помилки RBAC: cluster-admin для всіх

RBAC: Role-Based Access Control

RBAC — Найкращі практики

  • Уникайте підстановки (*) у правах доступу
  • Не давайте cluster-admin без потреби
  • Окремі облікові записи служб для кожного застосунку
  • Регулярний аудит прав

Мережеві політики (Network Policies)

  • Контроль трафіку між подами
  • Заборона за замовчуванням: все заборонено
  • Явно дозволяйте лише потрібний трафік
  • Потрібен CNI-плагін: Calico, Cilium, Weave

Стандарти безпеки подів (PSS)

  • Privileged: без обмежень (системні компоненти)
  • Baseline: мінімальні обмеження
  • Restricted: суворі правила (production)
  • Застосовуються через мітки namespace

Управління секретами

  • За замовчуванням — base64 (це НЕ шифрування!)
  • Шифрування у стані спокою: обов'язково для etcd
  • Зовнішні рішення: Vault, Sealed Secrets
  • Ніколи не зберігайте секрети в Git!

Контролери допуску (Admission Controllers)

  • Перехоплюють запити до API Server
  • PodSecurity: застосування стандартів безпеки подів
  • OPA Gatekeeper: політики як код (мова Rego)
  • Kyverno: політики у форматі YAML

Контрольний список безпеки K8s

  • API Server — тільки приватна мережа
  • RBAC — мінімальні права доступу
  • Мережеві політики — заборона за замовчуванням
  • PSS: Restricted для production-середовища
